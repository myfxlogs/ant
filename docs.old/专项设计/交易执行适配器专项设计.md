# 交易执行适配器专项设计

本文档定义 AntTrader 借鉴 QuantDinger 执行适配器思想时，对 MT4/MT5 执行边界进行行为等价治理的设计依据。

## 1. 目标与非目标

目标：

- 将 MT4/MT5 平台差异收敛到统一 ExecutionAdapter 边界。
- 减少平台分支向 TradingService、AutoTradingService、调度、Agent 或 MCP 扩散。
- 统一下单、平仓、撤单、查询、成交回报、错误归一和审计 metadata。
- 保持既有 MT4/MT5 行为等价，不改变用户交易语义。

非目标：

- 不新增币圈、美股或其他非 MT 市场实盘接入。
- 不绕过 RiskEngine 或 ExecutionGateway。
- 不把适配器变成最终业务决策层。
- 不手改上游 MT 协议生成文件。

## 2. 现有依赖模块

- TradingService：手动交易入口。
- AutoTradingService：自动交易信号执行入口。
- StrategyScheduleRunner：策略调度执行入口。
- RiskEngine：统一风控决策。
- ExecutionGateway：执行生命周期、幂等、熔断与审计。
- MT4 / MT5 Gateway：外部交易系统连接与协议交互。

## 3. QuantDinger 可借鉴点

- factory / adapter 模式。
- 统一订单请求、响应和错误模型。
- 多后端执行系统通过统一接口暴露给业务层。

AntTrader 的借鉴目标是治理 MT4/MT5 内部边界，不是扩展新的实盘市场。

## 4. AntTrader 实施边界

执行链路必须保持：

```text
TradingService / AutoTradingService / ScheduleRunner
  -> RiskEngine
  -> ExecutionGateway
  -> ExecutionAdapter
  -> MT4 / MT5
```

适配器只负责连接外部交易系统并归一执行语义，不负责最终业务决策。

## 5. 接口草案

统一适配器建议覆盖：

```text
ExecutionAdapter
  CheckAccountTradingCapability
  GetSymbolSpec
  PlaceOrder
  ClosePosition
  CancelOrder
  GetPositions
  GetOrders
  NormalizeExecutionReport
  NormalizeError
```

工厂建议：

```text
ExecutionAdapterFactory
  GetAdapter(platform)
```

业务服务不得在 handler 或页面路径中分散 MT4/MT5 判断。

## 6. 请求与响应模型草案

统一下单请求建议字段：

| 字段 | 说明 |
|------|------|
| `account_id` | 账户 ID |
| `platform` | `mt4` / `mt5` |
| `symbol` | 交易品种 |
| `side` | 买卖方向 |
| `order_type` | 市价、限价等 |
| `volume` | 手数 |
| `price` | 价格，可选 |
| `stop_loss` | 止损，可选 |
| `take_profit` | 止盈，可选 |
| `client_request_id` | 幂等或请求追踪 ID |
| `metadata` | 调度、策略、Agent 或用户上下文 |

统一响应建议字段：

| 字段 | 说明 |
|------|------|
| `accepted` | 外部系统是否接受 |
| `external_order_id` | MT 订单号 |
| `external_position_id` | MT 持仓号 |
| `execution_status` | 归一执行状态 |
| `raw_status` | 原始状态 |
| `error_code` | 归一错误码 |
| `raw_error` | 原始错误信息 |
| `metadata` | 执行 metadata |

原始错误必须保留，便于排障。

## 7. 错误归一

错误归一至少覆盖：

- 账户不可交易。
- 投资者密码或只读账户。
- symbol 不存在或不可交易。
- volume 非法。
- 市场关闭。
- 保证金不足。
- 价格偏离或重报价。
- 网络或 Gateway 超时。
- 外部系统未知错误。

归一错误用于业务展示和审计；原始错误用于排障，不直接暴露敏感信息。

## 8. 审计与风控关系

- 交易前检查由 RiskEngine 输出 `RiskDecision`。
- ExecutionGateway 记录执行生命周期、幂等、熔断、风险拒绝和引擎错误。
- ExecutionAdapter 返回归一执行结果和原始 metadata。
- Agent 或 MCP 触发的交易还必须写 Agent audit。

## 9. 行为等价迁移步骤

1. 梳理现有 MT4/MT5 下单、平仓、查询路径。
2. 定义统一请求、响应和错误码映射。
3. 为 MT4 实现 adapter，并用现有路径对照回归。
4. 为 MT5 实现 adapter，并用现有路径对照回归。
5. TradingService 逐步改为依赖统一 adapter。
6. AutoTradingService 与 ScheduleRunner 迁移。
7. 移除业务层散落的平台分支。

## 10. 前端边界

前端只展示后端返回的执行状态、错误码、订单日志和审计信息。

前端不得：

- 自行判断 MT4/MT5 执行差异。
- 自行计算交易能力或风控结果。
- 绕过后端直接调用 Gateway。

## 11. 验收标准与禁止事项

验收标准：

- 迁移前后 MT4/MT5 下单行为等价。
- 业务服务只依赖统一执行模型。
- 归一错误码可用于展示，原始错误可用于排障。
- 所有交易入口仍经过 RiskEngine 与 ExecutionGateway。

禁止事项：

- 禁止以适配器治理为名新增非 MT 实盘市场。
- 禁止绕过 RiskEngine 下单。
- 禁止绕过 ExecutionGateway 记录执行生命周期。
- 禁止手改 MT4/MT5 生成协议文件。
