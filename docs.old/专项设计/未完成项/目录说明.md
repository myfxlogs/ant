# 「未完成项」子目录说明

## 0. 先读这句（避免误解）

- **本目录里现在往往只有一两篇文档**，是因为我们**收窄了「放进这个文件夹」的口径**，**不是因为**原来那些专项所描述的能力都已经在代码里**全部实现**了。  
- **迁出 ≠ 废弃 ≠ 已完工**：策略实验、策略资产库、策略市场升级、用户体验与工程化方案等，只是改放在 **`docs/专项设计/`** 根目录下，文件名未改；文中 **目标 / 分阶段 / 未做部分** 仍以各专项正文为准。  
- **本目录名字容易误导**：若把「所有未 100% 的专项」都塞进来，会让人觉得整仓只有这些没做完；实际上未完成工作分散在多篇 **`docs/专项设计/*.md`**、**`docs/domains/*.md`** 与 **`docs/技术债清单与治理状态.md`** 里。

## 1. 本目录只放什么

本目录**仅收录**同时满足下列条件的专项设计：

1. 正文内存在**大块已明确延后、尚未实现的系统工程**（与主干已交付能力并存）；  
2. 若移到普通 `docs/专项设计/` 下，**易被误读为「全文已落地」**，从而误导排期与接口边界。

当前符合上述口径的文档：

- **[市场状态模型训练与集成专项设计](./市场状态模型训练与集成专项设计.md)**（训练 Job、ML、沙箱长驻自定义指标等见文内「延后」）

## 2. 不应再放入本目录的情形

以下请直接放在 **`docs/专项设计/`** 根目录（或 `docs/domains/` 等已约定位置），**不要**因「实现未 100%」就放入本目录：

- 正常迭代中的产品/技术专项（策略实验、策略资产、用户体验路线图等）；  
- 以「目标与非目标 + 分阶段实施」为主、且无「与当前主干严重错位」的专项。
