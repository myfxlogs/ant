# 新成员一站式导读

> **定位**：用**一份文档**在约 **45～90 分钟**内建立对项目**边界、技术栈、规范红线、文档地图与完成度量级**的全局认识；**不等于**已读完所有专项或已能独立合入复杂功能。  
> **深度补充**：需要按日拆任务、自检题与代码纵切时，再打开 **[新成员三日上手指南](./新成员三日上手指南.md)**（选修）。

## 2. 已有文档里的「目标」完成了多少？

**没有单一的「全项目完成百分比」。** 原因：各专项描述的是**不同层次**的目标——有的是**持续约束**（纲领），有的是**已落地简化版 + 明确延后**（市场状态），有的是**演进中能力**（辩论与 Job）。下面用**定性档位**概括（截至登记簿 `updated_at`，随实现变化；以 Git 与专项正文为准）。

| 文档/主题 | 目标性质 | 完成度（定性） | 说明 |
|-----------|----------|----------------|------|
| **接口与数据流架构约定** | 长期约束 | **持续生效** | 不是「做完」而是「后续实现须遵守」；违反即应视为架构债 |
| **市场状态模型训练与集成** | 混合：简化版 + 延后 | **部分完成** | 规则检测、内置特征、与回测/实验/调度等**简化闭环**已落地；训练 Job、ML、沙箱长驻自定义指标等**明确延后** |
| **市场指标体验与自定义指标资产** | 产品/技术路线 | **以设计为主** | 体验重构与资产化多为**待开发**路线，与上项「延后」章节衔接 |
| **辩论与代码生成异步任务 + 二期验收** | 可实施设计 | **主体已对齐** | 长耗时 Job、SSE 等按文档迭代；网关/超时等仍可能随运维变化，见 Cloudflare 对照清单 |
| **技术债清单与治理状态** | 滚动台账 | **持续更新** | 反映扫描与排期，非「清零即结项」 |
| **项目文档与进度管理体系** | 元文档 | **R0 已落地** | 索引、YAML、PR 模板、导读、三日选修；产品内进度看板仍为 **0%**（未立项） |
| **remediation-2026-05 等历史治理** | 阶段性计划 | **部分完成** | 以是否已合并代码为准；索引中多标为「需复核/仅历史」 |

**关于 `docs/专项设计/未完成项/`**：该子目录**仅收**「与主干并存的大块延后系统工程」类专项（当前主要为市场状态训练与集成）；策略实验、策略资产、UX 路线图等已迁至 **`docs/专项设计/`** 根目录，详见 **[目录说明](./专项设计/未完成项/目录说明.md)**。

**粗量级（仅帮助心理预期，非会计口径）**：若把「各专项中**已标记为当前阶段应交付**且已在主干体现」的条目粗加，整体可理解为 **大量主路径已跑通**，但 **市场状态全量训练、自定义指标 Runtime、产品内进度 RPC** 等大块仍**未做或延后**；**类型收敛、驱动统一**等技术债文档目标则**长期未完成**。

## 3. 进度登记里有没有体现？

**有，但是「模块里程碑估计」，不是上表的逐条换算。**

- **[模块进度登记簿](./进度/模块进度登记簿.yaml)** 里每个 `module_id` 的 **`percent`** 表示团队对该模块**当前里程碑**的共识（0～100），**不是**「所有设计文档目标条目的加权完成率」。
- **为何不能自动等于「文档目标完成度」**：设计文档里同时存在**已交付**、**刻意不做（延后）**、**规范型（永不 100% 结项）**三类叙述，无法合成一个无歧义数字。
- **如何对齐**：上表 §2 的定性结论，应在各模块的 **`summary` / `doc_refs`** 里用一句话呼应；重大范围变化时**同 PR 更新 YAML**。

若你希望登记簿里**显式出现「文档目标快照」**，可在 YAML 中为对应 `module_id` 增加 `doc_refs` 指向本文 §2 表，或在专项设计文末增加「与登记簿 `module_id` 映射」一行（任选其一，避免三处各写一套数字）。

## 5. 三条红线（背下来再写代码）

1. **新业务能力**：Connect / gRPC / SSE；**不把 REST 当新主 API**（窄幅修复除外）。  
2. **实时性**：**不以轮询为主路径**；异步进度用事件/SSE/流式 RPC。  
3. **不新引入 WebSocket**（见接口纲领）。
