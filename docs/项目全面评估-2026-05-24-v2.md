## 一、数据基础（Market Data Foundation）

**评分：A- | 完成度 ~92%**（上次 A- / 90%）

### 修复带来的提升
- Backfiller 电线完成：不再是空循环。Source → Target → 6h cron → PG NOTIFY 全部就位
- MT4/MT5 不再因一次断连而永久失联：指数退避重连保活
- Gap/clock skew 从死代码变成活监控：可被 Prometheus 抓取
- Dedup 窗口从 100 扩大到 1000，去重 hash 不再包含 ArrivedUnixMs → 回放 tick 与实时 tick hash 一致
- 陈旧数据告警：5 分钟无 tick = stale，15 分钟 = dead
- md_bars TTL = 2 年，md_ticks TTL = 90 天（已存在）

### 架构总览
```
MT4/MT5 gRPC Stream
  → recvLoop (指数退避重连)
  → Manager.HandleTick
    → [1] Normalizer (PG 缓存 + LISTEN 失效)
    → [2] Quality (硬丢弃 + 中位数MAD离群 + gap/skew metric)
    → [3] Dedup (ring buffer 1000, hash=ts+bid+ask+vol)
    → [4] BarAggregator (close_ts 精确去重 finality)
    → [5] Publisher (NATS JetStream + Nats-Msg-Id dedup)
    → [6] CHWriter (双写: Buffer Engine + Spill 兜底)
  → Backfiller (6h cron + PG NOTIFY 即时触发)
  → healthMonitor (30s: 陈旧检测 + 内存压力 → Buffer 降级)
```

### 仍待改进
1. **gateway 创建是桩**（`runner.go:128-130`）：`loadActiveAccounts` 返回了活跃账户列表，但只打日志不构建 `mt4.Gateway` / `mt5.Gateway`、不 Connect、不 Subscribe。这是生产最后一个未完成的大块
2. **`MTAPIBarSource` 无实现**：MT4/MT5 gateway 缺少 `GetPriceHistory` 方法，导致 backfiller 的 `Source` 是 nil adapter。backfiller cron 能运行但取不到任何 bar
3. **CircuitBreaker 未测试**：代码完整但无测试覆盖
4. **Publisher 测试全是 nil JetStream**：真实 NATS 交互未测

## 三、策略引擎（Intent → Strategy Code）

**评分：B- | 完成度 ~58%**（上次 B / 55%）

### 已具备
- **Factor DSL**：**这是整个项目最扎实的代码**。完整 lexer → recursive-descent parser → compiler → evaluator，支持算术/比较/逻辑/三元、字段引用（`$close`）、17 种窗口函数（SMA/EMA/RSI/ATR/MACD/Bollinger/STD/VAR/rank/zscore）、标量函数（abs/log/exp/sqrt/pow）、跨序列相关/协方差
- `quantengine`：每策略独立 goroutine + panic recovery + PG 快照持久化
- `risksvc`：6 条风险规则（MaxPosition/DailyLoss/Drawdown/Session/Margin/CanonicalAuth），可插拔 pipeline，完整测试
- PG 迁移完整（111 版本）

### 关键缺失
1. **ONNX 推理是桩**：`tryLoadONNX` / `predictONNX` 始终返回错误。CGO 构建标签存在但 `predictONNX` 委托给 `predictDSL`。真正模型推理 = 不存在
2. **`quantengine` 全员 V1-LEGACY**：标注"将被 M7.1-7.4 cards 替代，不要扩展"
3. **`FactorCHWriter` 是桩**：flush 循环打 "flushed to CH (stub)"，无实际 ClickHouse 写入
4. **`FactorMetrics` 是空壳**：计数器存在但未接入任何可观测系统
5. **`factorsvc` 订阅用 `interface{}`**（`subscriber.go:57 TODO: switch from in-process dispatch to NATS subscription`）
6. **无回测引擎**：DSL 可以评估因子但不能回测策略
7. **参数优化/超参搜索**：完全不存在

## 四、订单执行（Order Execution）

**评分：C+ | 完成度 ~32%**（上次 C+ / 30%）

### 已具备
- `oms`：完整订单状态机（10 个中间状态 + 5 个终态）+ PG repository（Insert/UpdateState/FindByID/FindByTicket）
- `mthub`：MT 订单执行 hub，PG 支撑
- `risksvc`：6 条风控规则，全部测试通过

### 关键缺失
1. **`BrokerAdapter` 无实现**：接口定义了 Submit/Cancel/Modify/Query，但没有任何 broker（MT4/MT5/Binance）实现它
2. **无仓位管理**：没有 position tracking、netting/hedging 模式选择
3. **无滑点/部分成交模型**：订单状态机假设全量成交
4. **无订单路由**：没有 smart order routing 或 broker 选择逻辑
5. **`TradingService` 是空文件**：`connect/` 目录下 `trading_service.go` 只有 `package connect`
6. **无 FIX 协议支持**：只有 mtapi gRPC

## 当前最关键的 5 个问题

1. **gateway 创建是桩**（数据基础 P0）：活跃账户列表有，MT4/MT5 适配器有，但 runner.go 不把它们连起来。数据管道 92% 完成但**实际没有 tick 流入**

2. **`BrokerAdapter` 零实现**（订单执行）：OMS 状态机完整、风控完整，但没有任何 broker 实现 Submit/Cancel/Modify/Query 接口。订单永远发不出去

3. **ONNX 推理是假实现**（策略引擎）：DSL 引擎非常扎实，但模型推理路径在 `tryLoadONNX` 就死了。策略只能靠 DSL 表达式，不能用 ML 模型

4. **`main.go` 不接线 auth interceptor + config**（DevOps）：JWT/API key 中间件写好了但不装、YAML config 定义了但不加载、gRPC listener 不启动。安全边界和环境管理形同虚设

5. **`MarketServer` / `TradingService` / `StrategyService` 是空壳**（前端依赖）：proto 定义了 25 服务 156 RPC，ConnectRPC handler 里有 3 个是空文件或返回零值。前端页面再多也拿不到数据
