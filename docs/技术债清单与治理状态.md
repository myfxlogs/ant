## 3. 低优先级 / 架构级债

### 3.1 后端 `lib/pq` 与 `pgx` 并存

- **`github.com/lib/pq`** 仅见于 **回测相关** repository/service（如 `backtest_run_repository.go`），主路径已 **pgx/sqlx**。
- **治理**：若需统一栈，可评估将 **数组类型 / COPY** 等 pq 专用逻辑迁到 **pgx**，再移除 `lib/pq`；属**行为等价**迁移，需单测与回归。

- **`docs/remediation-2026-05/`** 下多篇迁移与清单文档描述 **REST → ConnectRPC** 的进程；当前仍以 **`docs/接口与数据流架构约定.md`** 为接口形态纲领。
- **REST 业务面**是否清零以该批文档 +代码为准，本清单不重复枚举路径。

## 4. 外部参考与能力对标（QuantDinger）

`Emulator/QuantDinger` 为同领域开源项目（Python/Flask + Vue 预构建），其以下能力可作为 AntTrader 演进参考：

| QuantDinger 能力 | AntTrader 现状 | 借鉴价值 |
|------------------|----------------|----------|
| **多交易所执行适配抽象**（factory / adapter / 统一订单接口） | 仅 MT4/MT5 Gateway，执行差异仍需持续收敛 | 先用于治理 MT4/MT5 内部执行边界；新增币圈/美股实盘接入暂缓 |
| **策略运行时 DSL**（`on_init`/`on_bar` + `ctx.buy/sell/close_position`） | Python 策略沙箱通过文件通信 | 可研究更轻量的「策略脚本内嵌运行时」，降低启动开销 |
| **AI 实验系统**（参数进化、网格/随机搜索、市场状态识别、评分） | 辩论 V2 生成策略 + 回测 | 参数自动优化与多轮进化可作为策略调优的增强方向 |
| **积分/计费/会员** | 无 | 若未来 SaaS 化或团队版，可参考积分消耗 + USDT 支付 + VIP 体系 |
| **MCP Server / Agent Gateway** | 无 | Agent 可调接口、token scope、审计日志可作为安全边界参考 |
| **策略共享/社区**（发布、审核、克隆、本地副本同步） | 无等价模块，仅有个人策略模板、发布与调度 | 可先建设内部策略资产库，支持团队复用、版本同步和 Agent 检索 |
| **K线缓存与 TTL** | 行情通过 MT Gateway 推送 | 若增加独立行情源，可参考内存缓存 + 按周期 TTL |
| **数据多源聚合**（AkShare/YFinance/Finnhub/Tiingo 等） | 依赖 MT 行情 | 扩展数据源时可参考 `DataSourceFactory` 模式 |

> 注：QuantDinger 前端为预构建 `dist`，源码不在本仓库；其 REST API 形态与 AntTrader 的 ConnectRPC 体系不同，借鉴时须按 `docs/接口与数据流架构约定.md` 重新设计接口。
> Agent Gateway、MCP、通用 Job、Market Regime、策略实验、策略资产库、K 线缓存与交易执行适配器边界详见 `docs/domains/agent-gateway-and-strategy-experiment.md`。

### 4.2 实施约束

- 借鉴 QuantDinger 的 Python 实现时，须评估是否迁移至 Go（后端）或保留 Python（策略服务）。
- 涉及新增 `.proto` 或 RPC 时，必须先更新 `docs/接口与数据流架构约定.md` 或专项设计文档，再编写代码。
- 新增数据源或交易所适配须考虑：AntTrader 当前以 **MT4/MT5 订单流** 为核心，扩展市场类型需同步调整风控与订单记录模型。
- 新增非 MT 市场实盘接入、积分/计费/会员体系暂不纳入近期排期；不得因参考 QuantDinger 而扩大交易核心、资金模型和商业化系统的改造面。
- 交易执行适配器抽象是近期可借鉴方向，但目标是先治理 **MT4/MT5** 内部边界，不是立即接入币圈或美股实盘。
- 策略共享应先定位为**内部策略资产库**，不得直接实现公开市场、付费购买或跨租户分发。
