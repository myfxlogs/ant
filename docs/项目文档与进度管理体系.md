# 项目文档与进度管理体系

> **目标**：解决「缺少统一进度视图、新成员上手慢、文档陈旧易混」三类问题；在符合 **`docs/接口与数据流架构约定.md`** 的前提下，分阶段建设**文档库 + 进度登记 + 交接规范 +（可选）产品内进度能力**。

---

## 1. 原则与范围

### 1.1 文档单一事实来源

| 层级 | 用途 | 存放 |
|------|------|------|
| **纲领** | 接口形态、数据流、禁止项 | `docs/接口与数据流架构约定.md` |
| **全景** | 架构、模块、技术栈总览 | `docs/项目技术全景分析报告.md` |
| **索引与新鲜度** | 读什么、是否过时、谁维护 | **`docs/文档索引与维护状态.md`**（必读入口） |
| **专项设计** | 单主题可实施设计 | `docs/专项设计/**/*.md` |
| **域文档** | 子系统深度说明 | `docs/domains/*.md` |
| **历史/治理** | 迁移、排期、基线 | `docs/remediation-2026-05/` 等 |
| **进度登记** | 模块完成度、时间线、责任人 | **`docs/进度/模块进度登记簿.yaml`** + `docs/进度/说明.md` |

**冲突处理**：以 **纲领** 为准；专项与代码不一致时，**先改文档再改代码**（见接口约定 §1）。

### 1.2 进度信息的两种形态

1. **仓库内登记（P0，立即采用）**：YAML + Markdown，随 PR 更新，Git 历史即**时间线**与**责任人**（commit/PR author）。支持按模块查询文件；按人员查询用 `git log -- docs/进度/`。
2. **产品内可视化（P1+，可选）**：在需要跨团队、非 Git 读者或大屏看板时，再实现 **ConnectRPC** 查询 + **SSE** 推送进度变更；**禁止**以轮询为主路径、**禁止**新增业务 REST 作为进度主 API。实施前须更新接口纲领并单独写一页「进度 RPC 契约」。

### 1.3 新成员目标（3 个工作日）

可验证产出：读完 **索引 + 全景 + 上手指南 + 本文件 §6 学习路径**，能口述：

- 前后端与策略服务边界、Connect/gRPC/SSE 用法边界；
- 当前自己负责模块在 **`模块进度登记簿.yaml`** 中的状态与最近变更；
- 未完成任务在进度登记与相关专项中的位置。

## 2. 结构化项目文档库（对应需求 2）

### 2.1 目录约定

- 所有新增架构/流程/接口说明：**`docs/` 下中文文件名**（与仓库规则一致）。
- **英文文件名**仅限历史遗留或对接外部工具链（如 `domains/agent-gateway-and-strategy-experiment.md`）；新增域文档优先中文名，并在 **`文档索引与维护状态.md`** 中登记。

### 2.2 必备文档类型

| 类型 | 内容 | 维护动作 |
|------|------|----------|
| 架构与数据流 | 接口纲领、网关、长连接、异步 Job | 改 API/流式行为前必更新 |
| 核心模块 | 全景报告、domains、专项设计 | 大功能合入后 1 周内同步摘要 |
| 接口 | `.proto` + Connect 风格；禁止把 REST 当新主面 | `make proto` 与生成物一并提交 |
| 开发规范 | Cursor rules、Connect 风格指南、proto 拆分 | 规范变更开独立 PR |
| 历史变更 | remediation、技术债修订记录、重要专项修订表 | 合并时更新修订记录 |

### 2.3 文档防陈旧机制

1. **维护状态字段**：在 `文档索引与维护状态.md` 中为每篇标记 `当前 / 需随版本复核 / 已知滞后 / 仅历史`。
2. **大 PR 门禁（建议）**：PR 模板勾选「已更新相关 `docs/`」；触及对外行为时必选「已更新接口与数据流架构约定或专项」。
3. **季度复核**：负责人对 `需随版本复核` 清单走读；结果记入索引表「最后复核日期」。

## 3. 进度追踪功能（对应需求 1）

### 3.1 P0：仓库内可视化与约定

- **数据文件**：`docs/进度/模块进度登记簿.yaml`  
  - 字段建议：`module_id`、`name`、`owner`、`status`（未开始/进行中/阻塞/已交付）、`percent`（0–100）、`summary`、`blockers`、`doc_refs`、`updated_at`、`updated_by`（可省略，以 Git 为准）。
- **人类说明**：`docs/进度/说明.md`（如何改 YAML、如何做里程碑、与专项设计的关系）。
- **百分比规则**：`percent` 为团队对「该模块当前里程碑」的共识估计；**重大变更**须在 `summary` 或专项文档中写一句依据，避免数字与文字长期矛盾。**不是**「全仓库所有设计文档目标的加权完成率」；全局定性见 **`docs/新成员一站式导读.md`** §2～§3。

### 3.2 查询维度

| 维度 | P0 做法 | P1+ 产品化 |
|------|---------|------------|
| **按模块** | 读 YAML 按 `module_id` / `name` 过滤 | Connect `ListProgressByModule` |
| **按时间线** | `git log -p -- docs/进度/` | 服务端存 `progress_events` + SSE |
| **按人员** | YAML `owner` + Git blame / log | 人员维度索引表 + 查询 RPC |

### 3.3 「实时更新」语义

- **P0**：合并到默认分支即「最新」；本地开发分支自行维护副本，合并时解决冲突。
- **P1+**：关键里程碑由服务写入事件表，前端 **SSE** 订阅；仍保留 YAML 或 DB 导出为备份与审计。

## 4. 新成员快速上手指南（对应需求 3）

- **默认（短路径）**：**`docs/新成员一站式导读.md`** — 约 45～90 分钟建立全局图，并说明文档目标**定性**完成度与 YAML 的关系。  
- **选修（深路径）**：**`docs/新成员三日上手指南.md`** — 按日拆任务、代码纵切与自检题。

本体系要求：**每位模块 owner** 在索引表中登记自己负责文档链接；**入职 buddy** 首日给新人：**一站式导读**（必）+ **文档索引**（易混说明）+ 按需打开全景/纲领全文；三日指南仅在需要深度或考核时使用。

---

## 5. 任务交接标准化（对应需求 4）

所有涉及 **上下文切换** 的工作（请假、转岗、模块换手）须填写 **`docs/任务交接清单模板.md`** 副本（可存内部 Wiki 或作为 Issue/PR 描述粘贴）。

最低限度传递信息：

- 当前分支与未合并 PR；
- 负责模块在 YAML 中的条目与下一里程碑；
- 依赖服务、密钥与环境（不含秘密值，只写**存放位置与申请流程**）；
- 已知坑与相关 Issue/专项章节。

---

## 6. 新成员学习路径（摘要）

**首选**：读完 **`新成员一站式导读.md`** 并按文内顺序打开纲领/全景小标题即可。

**选修（原「三日」摘要）**：

| 日 | 主题 | 必读 / 必做 |
|----|------|-------------|
| **D1** | 地图与规范 | `文档索引与维护状态.md`、`接口与数据流架构约定.md`、`项目技术全景分析报告.md` §1–§4；本地 `docker compose` 或 README 级启动（以仓库说明为准） |
| **D2** | 代码纵切 | 任选一个业务域：从 `proto` → `backend` Connect 实现 → `frontend` 页面；读对应 `docs/domains` 或专项 |
| **D3** | 债务与排期 | `技术债清单与治理状态.md`、`docs/进度/模块进度登记簿.yaml`；在索引中补一条自己负责文档的「最后阅读日期」或修正过时说明 |

详细拆解与自检题见 **`新成员三日上手指南.md`**。
