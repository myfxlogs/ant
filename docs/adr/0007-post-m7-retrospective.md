# ADR-0007 · M7-M9 执行回顾：B 方案叙事与实际结果的偏离

- **状态**：Accepted
- **日期**：2026-05-24
- **关联**：ADR-0001、ROADMAP、BACKLOG

## 1. 背景

ADR-0001 选择路线 B（地基重做 + 业务渐进重构），理由是"保留 AI/marketplace/53 页前端等已完成产品功能，业务代码零损失"。M7-M9 三个阶段已执行完成。

## 2. 实际结果与 ADR-0001 承诺的差异

| ADR-0001 承诺（B 方案） | 实际结果 |
|---|---|
| "保留 53 页前端" | 53 页全删。前端重建为 2 页（marketplace + admin）|
| "业务功能零损失" | AI 助手、策略市场、admin、K 线分析等页面全部从零重建 |
| "业务渐进重构" | service 层 90+ 文件被 M7 地基变动牵连，一次性删除 |
| "风险可控" | 前后端同时断裂，CI 阻塞 2 天 |
| "最优 ROI" | 实际走的是 A 方案（全量重写），但消耗了 B 方案的叙事信用 |

## 3. 根本原因

不是执行错误。是 ADR-0001 对"M7 地基重做"的连锁影响评估不足：

- M7 把 `proto/` 整体重布局（`antrader` → `ant.v1`），所有 import 全部断裂
- M7 删除了 `mt4client`/`mt5client`/`kline_service`/`broker/`/`connection/` 等底层包
- 依赖旧 import 的 service 层、connect 层、前端 gen 层**必然**全部编译失败
- "业务渐进重构"的前提（旧代码能逐步迁移）从 proto 重布局那一刻起就不成立

## 4. 结论

路线 B 在纸面上的逻辑自洽，但在代码层面等价于路线 A。Opus 4.7 将"全量重写"拆成两个里程碑（M7 地基 + M8 重构）来规避"第二系统效应"这个否决理由，但执行结果是完整的前后端重写。

## 5. 教训

1. **proto 重布局 = 全量重写**。proto 是整个代码库的骨架，动了它就不可能"渐进"
2. **"保留业务"的前提是业务代码不 import 被替换的包**。M7 替换了 6+ 个底层包，所有上层必然断裂
3. **AI 写的 ADR 需要人类审视其"策略性表述"**——漂亮的框架和精确的措辞可能掩盖执行层面的不可行性

## 6. 对后续的影响

- ROADMAP 的卡片完成状态已修正（`BACKLOG.md` 2026-05-24 版）
- 不再按原 ROADMAP 的卡片粒度推进，改为按"最快可验收"优先级排序
- 下一个里程碑的目标是：前后端打通 → 行情数据可看 → 最小可登录 → 业务功能逐步恢复
