# Ant 项目全面评估

> 评估日期：2026-05-24
> 评估范围：全仓库（backend / frontend / deploy / proto / docs）
> 评估框架：产品价值链（表达 → 生成 → 执行 → 变现）+ 数据基础 + 前端 + DevOps

---

## 产品定位

> "让普通人能用自然语言把交易想法变成自动执行的策略，并把好策略变现"

价值链 4 个环节：**表达 → 生成 → 执行 → 变现**。以下按此框架逐段评估。

---

## 一、表达（自然语言 → 交易意图）

**评分：B+ | 完成度 ~70%**

### 已具备

- AI Provider 层抽象了 12 家厂商（OpenAI / Anthropic / DeepSeek / 智谱 / 月之暗面 / 通义千问 / Groq / OpenRouter 等），多模型路由就绪
- `backend/internal/ai/prompt/strategy_prompt.go` 中文 system prompt 成熟：内置 MQL 风格指标（iMA / iRSI / iMACD / iBands / iATR）、风控函数（risk_size / atr_size / kelly_size）、强制沙箱约束
- `backend/internal/ai/symbol_extractor.go` 中英文符号映射覆盖 ~50 个品种（外汇 / 商品 / 指数 / 加密货币）
- `backend/internal/ai/code_compliance.go` 预沙箱扫描：禁 import os/subprocess/socket/eval/exec
- `backend/internal/ai/symbol_validator.go` 符号可交易性校验，关联 `broker_symbols` 表

### 关键缺失

- **AI 生成的 Python 代码只存为字符串，从未被执行** — `StrategyTemplate.Code` 写入 PostgreSQL 后流程终止。这是整个价值链最关键的断裂点
- 没有 Python sandbox runtime（gRPC subprocess 或 container），生成的策略代码无法回测、无法 paper trade、更无法自动执行
- 前端策略构建器页面为空壳（`backend/internal/connect/strategy_service.go` 和 `trading_service.go` 仅含 `package connect`）

---

## 二、生成（交易想法 → 可执行策略）

**评分：B | 完成度 ~55%**

### 已具备

- **DSL 引擎**（`backend/internal/factor/dsl/`）是真正的亮点：
  - 完整 lexer → parser → AST → operator 编译器
  - 支持 SMA / EMA / RSI / MACD / Bollinger / ATR / Z-Score 等 20+ 技术指标
  - 可直接编译为可执行信号
- `backend/internal/quantengine/runner.go` goroutine-per-strategy 的 bar-driven 运行时，信号生成 → OMS 订单提交链路完整
- `backend/internal/risksvc/` 风控引擎：实时仓位 / 保证金 / 敞口检查
- 策略调度模型完整：cron / interval / event 三种触发方式
- `backend/internal/factorsvc/engine.go` 因子注册 + 滚动窗口缓冲求值

### 关键缺失

- **AI 路径和 DSL 路径是两条平行线，从未交汇** — DSL 能自动执行但不涉及自然语言；AI 能生成代码但从不执行
- ONNX 模型推理标注为 STUB：需要 CGO + libonnxruntime.so，未集成
- `backend/internal/ai/training_engine.go` 是 build-tag 门控的骨架代码：RL（PPO/SAC/TD3/DQN）、ES（CMA-ES）、GA、Bayesian 训练均从未真实运行
- 无 Python subprocess 桥接层

## 三、执行（策略信号 → 真实订单）

**评分：C+ | 完成度 ~30%**

### 已具备

- `backend/internal/mthub/` session hub 管理、订单事件 pub/sub、仓位追踪骨架完整
- `backend/internal/quantengine/signal_oms_bridge.go` 信号 → 订单转换 + 风控检查链路实现
- MT account 凭证存储 + envelope encryption（`backend/internal/secrets/` AES-256-GCM + HKDF + 版本化 KEK）
- `backend/internal/connect/mthub_service.go` 153 行真实实现：PlaceOrder / CloseOrder / GetPositions / SearchBrokers
- `backend/internal/connect/auth_handler.go` JWT + API Key 鉴权 ready
- `deploy/` docker-compose 定义了独立的 strategy-service 容器（384M, NET_ADMIN）

### 关键缺失

- **MT4/MT5 broker connector 不存在** — `oms.BrokerAdapter` 和 `mthub.OrderExecutor` 只是 Go interface，无真实实现。订单可生成但无法送达任何真实交易账户
- strategy-service 容器在 docker-compose 中定义但无功能代码
- 策略执行和 API 网关共享同一 Go 二进制 — 单点故障风险
- 订单状态机不完整（无部分成交 / 滑点处理）

## 四、变现（好策略 → 收入）

**评分：C | 完成度 ~20%**

- 无收益分成结算逻辑
- 无订阅到期/续费管理

## 五、数据基础（行情存储 + 分析支撑）

**评分：A- | 完成度 ~90%**

### 已具备

M10 硬化后的完整行情数据管线：

```
MT4/MT5 adapter → normalize → quality → dedup → aggregate
                                   ↓           ↓
                                 DLQ        NATS publish (JetStream)
                                             CH write (ClickHouse)
```

- **adapter 层**：MT4/MT5 gRPC streaming (`OnQuote` server-stream) 真实对接
- **normalize**：PG 支持的 LRU 缓存 + 符号规范化规则
- **quality**：bid>ask 硬丢弃 + median/MAD 异常标注
- **dedup**：滚动窗口确定性哈希去重
- **aggregate**：6 周期 bar 聚合（1m/5m/15m/1h/4h/1d），exact-close_ts finality
- **publish**：NATS JetStream `MD_EVENTS` stream，Nats-Msg-Id 去重
- **store**：ClickHouse ReplacingMergeTree + Buffer engine + FINAL 查询
- **容灾**：DLQ 异步化 + spill/replay JSONL + NATS dual-write
- **工具**：md-doctor（5 子命令对账）、slo-report（4 SLO 查询）
- **测试**：e2e smoke（三方对账）+ 100-account loadtest（spill=0, P99<500ms）
- **校验**：M10.5 全部 13 张卡片通过 C1-C4 加固校验

### 残留设计缺陷

| 编号 | 缺陷 | 状态 |
|------|------|------|
| S-1 | CH dedup 与应用层 hash 冲突 | ✅ 已修（md_ticks_v2 ORDER BY 含 bid/ask/volume）|
| S-2 | Buffer engine 无 OOM 保护上限 | ⚠️ 有开关（`ANT_CH_BUFFER_ENABLED`），无自动降级 |
| S-3 | bar finality 用 `<= MAX` 拒回填 | ✅ 已修（IngestExternalBar exact-match）|
| M-1 | backfiller 单全局 limiter | ✅ 已修（per-account `map[string]*rate.Limiter`）|
| M-2 | spill replay NATS 重复无 dedup | ✅ 已修（Nats-Msg-Id header）|
| M-3 | runner loadFinalizedBars CH-down 不 fatal | ✅ 已修 |
| L-1 | DLQ 同步阻塞热路径 | ✅ 已修（`dlqQ chan` + 后台 goroutine）|
| L-2 | OTel sampling 1% 无环境覆盖 | ⚠️ SDK 已集成，未在 e2e 环境验证 |
| L-3 | vault rotate 不支持 EnvMasterKey | ❌ `Rotate()` 仍 return error |

## 九、关键路径（按产品价值排序）

### P0 — 策略执行闭环（打通价值链最核心的断裂点）

1. **Python sandbox runtime**：gRPC subprocess 方案，Go 端传入 `run(context)` 代码 + 行情数据 → 返回交易信号
2. **Paper trading 模式**：信号不送真实 broker，只记录模拟成交，用户安全验证策略
3. **AI 路径接入 quantengine runner**：AI 生成策略和 DSL 策略走同一执行引擎

### P1 — 订单执行通路

4. **MT5 broker connector** 真实实现 `OrderExecutor` 接口
5. **订单状态机完善**：已提交 → 部分成交 → 完全成交 → 已取消，含滑点/延迟模型
6. **strategy-service 独立部署**：策略执行与 API 网关解耦

### P2 — 变现闭环

7. 策略回测结果可审计（PnL 独立验证，防刷数据）
8. 支付集成（至少一个渠道）
9. 收益分成自动结算 + 订阅生命周期管理

### P3 — 产品体验

10. 前端策略构建器（自然语言输入 → 实时预览生成代码 → 一键回测 → 一键发布）
11. 策略市场前端（浏览 / 筛选 / 订阅 / 评价）
12. 实时行情 Dashboard + 策略运行状态可视化

## 十、架构风险

1. **Python 沙箱安全（严重）** — 用户策略代码可能包含恶意逻辑，必须在强隔离环境（gVisor / Firecracker / 独立 container）执行；当前无任何沙箱
2. **MT broker 依赖（严重）** — 订单执行完全依赖第三方 MT4/MT5 服务器的可用性和延迟；需考虑 broker 不可用时的降级策略
3. **单点故障（中等）** — backend 是单二进制，panic 全挂；strategy-service 容器已定义但未实现独立部署
4. **回测 vs 实盘差距（中等）** — 无滑点/延迟/流动性/手续费模型，回测结果可能严重误导用户
5. **数据安全（中等）** — MT 账户密码/token 虽已 envelope 加密，但 KEK 从环境变量派生，缺少 HSM/KMS 支持
6. **合规风险（待评估）** — 策略发布/订阅若涉及实际资金管理，在不同司法管辖区可能触发监管要求

---

## 十一、总结

项目当前完成了 ~45% 的产品化路径。

**最强的资产**是数据基础 — 行情管线经过 M7-M10 三轮迭代，已达到 A- 水平，有真实 e2e 测试和负载测试背书。

**最大的单一瓶颈**是缺少 Python sandbox 让 AI 生成的策略真正跑起来 — 这是连接"自然语言表达"和"自动执行"的桥，桥没修好，两端都是孤岛。

**下一个最值得投入的方向**：在一个 milestone 内完成 Python sandbox runtime + paper trading + AI 路径接入 quantengine runner，打通"表达 → 生成 → 执行"的最小闭环。这一步一旦完成，产品从"能看"变成"能用"，后续变现才有地基。
