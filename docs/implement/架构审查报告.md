## 1. 严重缺陷（必须修复）

### ℹ️ 1.1 密码存储为项目设计要求

**位置**: `backend/internal/mdgateway/wiring.go:69-92`
**说明**: 密码以明文存储，注释写明 `"Passwords are stored as plaintext per user requirement"`。这是项目设计决策，非缺陷。

**注意项**：`internal/secrets/` 的 AES-GCM 体系虽然存在，但在密码存储路径上未被使用。如果未来安全要求升级（例如合规审计要求加密存储），需要在该处接入 secrets 包。

---

### 🟡 1.2 SRE HTTP 端点鉴权路径与 ConnectRPC 不一致

**位置**: `backend/internal/connect/admin/sre_handler.go` + `backend/internal/interceptor/auth.go:100-115`
**问题**: 8 个 SRE 管理端点使用原生 HTTP + 显式 `requireAdmin()` 调用鉴权，与 ConnectRPC 的拦截器链（AuthInterceptor → AdminInterceptor）走不同路径。

**实际情况**：鉴权是有效的——`requireAdmin()` 调用 `ai.UserIDFromHTTP(r)`（内部使用相同 JWT secret + `ValidateToken`）+ `h.checker.IsAdmin()`（与 AdminInterceptor 相同逻辑）。认证和权限检查都做了，只是**模式不同**。

**风险**: 
- 每个端点必须手动调用 `requireAdmin()`，新端点容易遗漏
- 上下文未注入 `UserIDKey`，handler 体内无法使用 `interceptor.GetUserID(ctx)`
- 非 P0 安全漏洞，但存在维护风险

**修复方案**: 添加统一的 HTTP 中间件包装器：

```go
func (i *AuthInterceptor) WrapHTTP(h func(http.ResponseWriter, *http.Request, *AuthInterceptor)) http.HandlerFunc {
    return func(w http.ResponseWriter, r *http.Request) {
        uid, err := i.UserIDFromHTTP(r)  // 复用已有方法
        if err != nil { http.Error(w, "unauthorized", 401); return }
        ctx := context.WithValue(r.Context(), UserIDKey, uid.String())
        h(w, r.WithContext(ctx), i)
    }
}
```

### 🟡 1.3 死代码：`service.AuthService` 未被使用

**位置**: `backend/internal/service/auth_service.go`（完整实现，零调用方）
**问题**: `service.AuthService` 定义了完整的 Register/Login/RefreshToken/ChangePassword/ValidateToken，但在 `main.go` 和 `connect/user/auth_handler.go` 中完全没有被引用。认证逻辑全部在 `connect/user/auth_handler.go` 中直接使用 `repository.UserRepository` 实现。

**影响**: 死代码增加维护负担。未来改动认证逻辑时，需要改两份代码或者审阅者不确定哪份是活的。
- grep 结果：`service.AuthService` 仅在自身文件和生成代码中被引用，零次被 `internal/connect/` 或 `cmd/` 调用。

**修复方案**: 删除 `service/auth_service.go`，或将其逻辑合并到 handler 层作为服务依赖注入。

```go
// 推荐方案：直接删除
$ rm backend/internal/service/auth_service.go
```

### 🔴 1.4 混合分层：Handler 直接注入 Repository

**位置**: `backend/internal/connect/admin/admin_account_handler.go:30`、`internal/connect/admin/admin_user_handler.go:25` 等
**问题**: 部分 handler 直接注入 `*repository.AdminRepository`，另一部分注入 `*service.PlatformService`。没有统一的分层规则。

**影响**: 业务逻辑散落在 handler 层（admin handler 中有 freeeze/trade/enable 等业务操作），违反 AGENT.md §7 "工程纪律" 第一条："handler 只编排；业务在 service；数据访问在 repo"。

**修复方案**: 强制分层——所有 handler 只能依赖 service 层，service 层依赖 repository：

```
handler → service → repository
        ↘ (禁止直通) ↗
```

### 🔴 1.5 Setter 注入反模式

**位置**: 
- `account_handler.go:37-39` — `SetSearcher` / `SetPublisher`
- `strategy_handler.go:34` — `SetClient`

**问题**: 对象图在构造时不完全，nil 检查推迟到运行时。调用 `SetXxx` 之前调用方法会 panic。

**修复方案**: 改为构造函数注入：

```go
// 坏
server := NewAccountServer(svc, log)
server.SetSearcher(searcher)    // 运行时才设置

server := NewAccountServer(svc, searcher, publisher, log)
```

## 2. 中度缺陷（建议修复）

### 🟡 2.1 sqlc + 手写 SQL 双轨制

**位置**: `backend/internal/repository/db.go`（sqlc 生成的 `Queries` 结构）vs 其余 18+ 个手写 repository 文件
**问题**: `broker_symbols.sql.go` 和 `factor_definitions.sql.go` 使用 sqlc 生成的查询绑定，但大多数 repository 用 `pgxpool.Pool.QueryRow()` + 手写 Scan。

**影响**: 两种风格的查询代码并存——手写 SQL 更容易出错（列名拼写、类型不匹配），且代码生成的好处（类型安全、自动更新）被稀释。

**修复方案**: 逐步迁移到手写 SQL（保持一致性）或逐步迁移到 sqlc（推荐）。短期至少统一所有 repository 使用同一模式。

---

### 🟡 2.2 全局可变 `Clk` 变量（8 个包）

**位置**: `costsvc/clock.go`, `mdgateway/*/clock.go` × 4, `mthub/clock.go`, `risksvc/clock.go`, `usermgr/limiter.go`
**问题**: 每个包声明 `var Clk clock.Clock = clock.NewRealClock()`，测试时覆盖为 `SimulatedClock`。但 Go 测试默认 `t.Parallel()` 不安全——竞态时两个测试同时读写同一个全局变量。

**影响**: CI 中出现间歇性测试失败（已存在 `TestCostBasis_WithSnapshots` + `TestCostBasis` 等并行测试使用同一 `costsvc.Clk`）。

**修复方案**: 
1. 改为依赖注入：所有需要时钟的 struct 接受 `clock.Clock` 参数
2. 短期：在并行测试中使用 `t.Setenv` 隔离或为每个测试用 `sync.Mutex`

---

### 🟡 2.3 jwt 包使用全局单例

**位置**: `backend/internal/pkg/jwt/jwt.go` — `var jwtConfig *JWTConfig` + `Init()`
**问题**: 全局 `jwtConfig` 只能 `Init` 一次，并行测试无法使用不同密钥。与 `secretbox` 包的构造器模式不一致。

**修复方案**: 改为 `NewJWT(secret)` 返回实例，去掉全局变量。

---

### 🟡 2.4 成本估算器返回值而非 error

**位置**: `backend/internal/costsvc/estimator_interface.go:10`
**问题**: `Estimate(ctx, params) CostBreakdown` 无 error 返回。`MultiModelEstimator` 找不到模型时静默返回零值。

**影响**: 调用方无法区分"零成本"和"找不到模型"，可能导致风控低估交易成本。

**修复方案**: 改为 `Estimate(ctx, params) (CostBreakdown, error)`，找不到模型返回 `ErrModelNotFound`。

---

### 🟡 2.5 FillModel 与 CostModel 的滑点关系不明确

**位置**: `backend/internal/backtest/fill_model.go` + `backend/internal/costsvc/calculator.go:63-66`
**问题**: `FillModel.SimulateFill()` 做了两件事：
1. 根据 `SlippagePips` 对 `GrossPrice` 施加随机滑点（模拟执行噪声）
2. 调用 `cm.SlippageCost(volume, grossPrice, cs)` 将结构化的 SlippageBps 成本加入 TotalCost

这两者代表不同的概念（随机 vs 结构化），但文档没有说明它们的关系。如果模型设计者同时设置了两个值，成本可能被高估。

此外，`SlippageCost()` 计算使用 `float64` 价格，而 `CostModel.Estimate()` 在 `estimator.go` 中使用 `notional * SlippageBps / 10000.0` 也是 float64，违反 AGENT.md §3.7 的"禁 float64 直接参与价格计算"规则。

**修复方案**: 
- 明确文档说明 FillModel.SlippagePips 与 CostModel.SlippageBps 的角色差异
- 或统一为单一入口：FillModel 只使用 CostModel 的成本估算，移除独立滑点计算

---

### 🟡 2.6 重复的 Sharpe 计算

**位置**: 
- `backend/internal/backtest/metrics.go:89-112` — 基于价格序列的 trailing-window
- `backend/internal/ai/walkforward.go:150` — `computeSharpe()` 基于日收益率

**问题**: 两套 Sharpe 实现，输入不同、公式可能不同。回测和 AI 评估可能得出不同的 Sharpe 值，影响策略排名。

**修复方案**: 统一为单一 `func Sharpe(returns []float64, riskFreeRate float64) float64`，放在 `internal/pkg/` 中共享。

---

### 🟡 2.7 execalgo 参数验证重复

**位置**: `twap.go:34-39`, `vwap.go:36-41`, `pov.go:39-44`, `shortfall.go:41-46`
**问题**: 4 个 algo 各自重复相同的参数验证（`TotalVolume <= 0`, `dur <= 0`）。

**修复方案**: 在 `ParentOrder` 或 `Algo` 接口上添加 `Validate()` 方法：

```go
func (p *ParentOrder) Validate() error {
    if p.TotalVolume <= 0 { return errors.New("total volume must be positive") }
    if p.Duration <= 0 { return errors.New("duration must be positive") }
    return nil
}
```

---

### 🟡 2.8 OTel Tracer Shutdown 未在所有路径执行 defer

**位置**: `backend/internal/mdgateway/runner.go:60` — `defer tracer.Shutdown(context.Background())`
**问题**: 仅在 mdgateway runner 中 defer，但 main.go 创建 tracer 的路径没有一致 defer。如果进程快速退出，span 可能丢失。

**修复方案**: 在 main.go 中创建 tracer 时即 `defer`：

```go
tracer := anttrace.New()
defer tracer.Shutdown(context.Background())
```

---

### 🟡 2.9 平台聚合器全局可变变量

**位置**: `backend/cmd/server/main.go:222` — `var platformAgg *risksvc.PlatformAggregator`
**问题**: `platformAgg` 在闭包 `OnOrderUpdate` 中被引用，但它在闭包创建后才被赋值（`platformAgg = risksvc.NewPlatformAggregator()`）。闭包引用一个尚未初始化的指针。

**影响**: 虽然当前运行时顺序保证 `OnOrderUpdate` 回调不会在赋值前触发，但重构时极易引入 nil pointer dereference。

**修复方案**: 在 mdgateway goroutine 启动前完成所有依赖赋值，或将 platformAgg 移到 `RunnerDeps` 中显式注入。

---

### 🟡 2.10 内存限流器不支持水平扩展

**位置**: `backend/internal/interceptor/ratelimit.go:26-29`
**问题**: 注释写明 "In-memory only"，清理循环每 5 分钟运行。如果未来部署多副本，限流状态不共享。

**修复方案**: 保留当前实现（单进程足够），添加 Redis-backed 实现的接口抽象，以备未来扩展。

## 3.4 Python 策略服务缺陷

### 🔴 3.4.1 `privileged: true` + `NET_ADMIN` 安全风险

**位置**: `docker-compose.yml:96-98`
**问题**: strategy-service 容器以 `privileged: true` 运行，带 `NET_ADMIN` 能力。如果策略代码逃逸沙箱（即使通过 RestrictedPython 保护），攻击者可以直接获得宿主机 root 权限。

**影响**: 整个宿主机的安全底线被突破。与 AGENT.md §3.4 安全红线冲突。

**修复方案**: 移除 `privileged: true` 和 `NET_ADMIN`。如果 iptables 隔离是必须的，使用更细粒度的 `cap_add: [NET_ADMIN]` 而非完全特权模式。

```yaml
# docker-compose.yml → 改为
cap_add: [NET_ADMIN]
# 移除 privileged: true
```

---

### 🔴 3.4.2 /api/strategy/execute 使用无进程隔离的沙箱

**位置**: `strategy-service/app/routes/strategy.py:65` 使用 `StrategyRunner`（in-thread `exec()`）
**问题**: In-thread 沙箱无法中断 C 扩展（numpy）的死循环，timer-based timeout 对 numpy 操作无效。`BacktestSandbox` 使用进程隔离（`multiprocessing.spawn`）可安全中断，但 execute 路径未使用。

**修复方案**: 将 execute 端点迁移到 `LiveWorker` 进程模型或移除该端点，只保留 `BacktestSandbox`。

---

### 🟡 3.4.3 BacktestRequest.ticks 为无类型 dict

**位置**: `strategy-service/app/schemas.py` — `ticks: List[dict]`
**问题**: `BacktestRequest` 的 `ticks` 字段是 `List[dict]`，没有 Pydantic 模型约束。解析推迟到 `_build_engine_request` 中手动 try/except。

**修复方案**: 定义 `TickData(BaseModel)` 模型：

```python
class TickData(BaseModel):
    price: float = Field(..., ge=0)
    volume: float = Field(0, ge=0)
    timestamp: int = Field(..., ge=0)
```

---

### 🟡 3.4.4 全局 Semaphore 限制回测并发

**位置**: `strategy-service/app/routes/backtest.py:15` — `Semaphore(int(os.getenv('MAX_BACKTEST_WORKERS', '2')))`
**问题**: 全局信号量限制回测并发为 2，但非异步安全 —— 使用 `run_in_executor` 在 thread pool 中执行，信号量保护的是线程而非进程。如果有多个回测同时提交，前两个运行，但所有后续请求立即被 Pending 状态返回。

## 3.5 基础设施缺陷

### 🔴 3.5.1 NATS 无认证配置

**位置**: `deploy/nats/nats.conf`
**问题**: 没有任何 `authorization {}` 块、token、或 Nkey/JWT。Docker 网络内的任何进程都可以连接并 publish/subscribe 所有 subject。

**影响**: 行情数据（tick/bar）可以被同一网络内的任何容器订阅。如果未来有任何第三方组件接入 Docker 网络，数据泄露。

---

### 🔴 3.5.2 Nginx 缺少 CSP 头

**位置**: `nginx/nginx.conf` — 无 `Content-Security-Policy` 头
**问题**: SPA 应用没有 CSP 头，允许内联脚本执行。如果 XSS 被触发，攻击者可以窃取 JWT token 并操作交易账户。

**修复方案**: 
```nginx
add_header Content-Security-Policy "default-src 'self'; script-src 'self'; style-src 'self' 'unsafe-inline'; img-src 'self' data:; connect-src 'self' https://api.example.com; font-src 'self'; frame-ancestors 'none';";
```

---

### 🟡 3.5.3 缺少 Prometheus 容器

**位置**: `docker-compose.yml` 没有 prometheus service，但 `deploy/prometheus/alerts.yml` 和 `rules.yml` 存在
**问题**: 监控配置和 Alerting rules 定义完整，但没有 Prometheus 实例来运行它们。告警规则是纯文档，实际不生效。

---

### 🟡 3.5.4 Backend 缺少 ClickHouse 和 NATS 的 depends_on

**位置**: `docker-compose.yml` — backend `depends_on` 只有 postgres + redis
**问题**: backend 启动时直接连接 CH_HOST 和 NATS_URL，如果 CH/NATS 尚未就绪，backend 会 crash 重启直到它们就绪。虽然短期内可行（docker restart），但启动序列不可预测。

---

### 🟢 3.5.5 Nginx 健康检查路径不匹配

**位置**: `frontend/nginx.conf` — `/health` 返回 200，而 backend 真实健康端点是 `/healthz`
**影响**: Nginx 认为后端健康即使真实 health check 失败。

## 4. 架构层面方法论问题

### 4.1 分层架构无编译期强制

**问题**: AGENT.md 规定了 `handler → service → repo` 的分层，但没有 lint 规则强制。目前靠 code review 人工检查。

**建议**: 在 CI 中添加 `make detect-layering` 检查：

```bash
# connect/ 下的文件不应直接 import pgxpool/clickhouse/sqlx
grep -rn '"github.com/jackc/pgx' backend/internal/connect/ && echo "FAIL"
grep -rn '"github.com/ClickHouse' backend/internal/connect/ && echo "FAIL"
```

### 4.2 策略服务客户端无容错

**位置**: `backend/internal/strategysvc/python_client.go`
**问题**: HTTP 调用 Python strategy-service 没有超时重试、熔断、trace 传播。

**建议**: 添加：
- 指数退避重试（3 次）
- 基于成功/失败的轻量熔断
- OTel trace header 传播
- `Content-Type` 响应验证

### 4.3 OTel 追踪覆盖面不足

**问题**: `internal/trace/` 提供了框架，但实际 span 创建集中在 `mdgateway/manager.go` 的 HandleTick 管线。`mthub.PlaceOrder`、`risksvc.Pipeline.Process`、`backtest.Engine.Run` 等关键路径缺少 span。

**建议**: 按 ADR-0018 的延迟预算优先级补充 span：
1. `mthub.PlaceOrder` 全过程
2. `risksvc.Pipeline.Process` 各 stage
3. `costsvc.Estimate`
4. `backtest.Engine.Run`

### 4.4 前端错误处理不一致

**位置**: `frontend/src/client/transport.ts`
**问题**: 错误处理混合了 `Modal.error`（连接失败）和 `message.error`（业务错误），但后端返回的 i18n key 翻译依赖 `translateMaybeI18nKey` 尝试性匹配——匹配不到时直接暴露英文 key 给用户。

**建议**: 
- 后端统一返回 `{code, message, user_message}` 结构
- 前端使用 `user_message` 优先，`translateMaybeI18nKey` 为后备
