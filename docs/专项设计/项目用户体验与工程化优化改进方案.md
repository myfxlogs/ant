# 项目用户体验与工程化优化改进方案

## 1. 文档状态

| 项 | 内容 |
|----|------|
| 状态 | 专项设计 / 后续落地指导（存放于 `docs/专项设计/`，非「未完成项」子目录） |
| 适用范围 | AntTrader 全栈项目 |
| 重点方向 | 用户体验、工程化、安全治理、可维护性、可观测性 |
| 权威约束 | `docs/接口与数据流架构约定.md`、`docs/技术债清单与治理状态.md` |

本文用于记录对当前项目的设计缺陷、不足与优化路线，为后续分阶段落地提供依据。

---

## 2. 总体判断

AntTrader 已具备较完整的交易、策略、AI、回测、调度、ConnectRPC、SSE、通用 Job、策略服务分层雏形。

当前主要矛盾不是单点功能不足，而是：

- 用户任务流过长、过散，核心路径不够连贯。
- 高级能力入口较多，但成熟度、权限边界、可解释性和引导不足。
- 前端体验、类型质量和状态管理落后于后端接口与文档设计。
- 工程化入口不统一，本地校验、CI、容器构建与策略服务校验仍需对齐。
- 安全配置、沙箱隔离、资源配额与审计体系需要前置治理。
- 文档较丰富，但缺少当前权威状态索引和实施优先级。

---

## 3. 优化目标

### 3.1 用户体验目标

- 围绕策略生命周期重构用户任务流。
- 降低普通用户理解 AI、回测、调度、市场状态、策略实验的门槛。
- 为关键操作提供明确前置条件、校验结果、风险提示和下一步动作。
- 将错误提示从技术异常转为可恢复的用户操作建议。
- 对未成熟能力使用统一功能开关，不再依赖代码注释或临时隐藏。

### 3.2 工程化目标

- 安全配置从开发态迁移到生产可控态。
- 本地 `verify`、CI、容器构建检查保持一致。
- 前端逐步消除热点 `any/as any`，让 ConnectRPC 生成类型真正贯穿页面。
- 轮询治理为事件流或 Connect streaming，符合数据流优先原则。
- 后端服务注册、接口形态、MCP 工具面和策略服务职责形成清晰边界。
- 用户自定义代码、AI 生成代码和自定义指标必须运行在受限 worker 或沙箱中。

## 4. P0：安全与生产配置缺陷

### 4.1 硬编码密钥与生产占位配置

当前 `docker-compose.yml` 中存在数据库密码、JWT Secret 和外部 API Key 的硬编码或占位值。

风险：

- 仓库泄露即泄露生产凭证。
- 部署环境复用默认 JWT Secret 会导致认证体系失效。
- 无法区分开发、测试、生产环境安全边界。

改进要求：

- 所有敏感配置迁移到 `.env` 或部署平台 Secret。
- `docker-compose.yml` 仅引用环境变量，不写真实密钥。
- `.env.example` 只保留字段说明和安全占位。
- 后端启动时校验生产环境 `JWT_SECRET` 强度，不满足要求拒绝启动。
- CI 增加 secret scanning 或最小化 grep 检查。

验收标准：

- 仓库内不再出现真实数据库密码、真实 API Key、生产 JWT Secret。
- 生产环境缺少安全密钥时后端无法启动。
- README 或 ops 文档说明如何配置本地和生产密钥。

### 4.2 CORS 全开放

当前后端存在 `Access-Control-Allow-Origin: *` 形态。

风险：

- 生产环境跨站访问边界过宽。
- 与鉴权、EventSource、ConnectRPC 错误暴露组合后风险上升。

改进要求：

- 引入 `ALLOWED_ORIGINS` 配置。
- 开发环境允许本地端口。
- 生产环境只允许正式域名。
- OPTIONS、ConnectRPC、SSE 统一走同一 CORS 策略。

验收标准：

- 生产环境未配置允许域名时拒绝启动或使用安全默认值。
- 浏览器端 ConnectRPC 与 SSE 在白名单域名下正常工作。

### 4.3 策略服务特权过高

当前策略服务容器存在 `privileged: true`、`NET_ADMIN` 等高权限配置。

风险：

- 与未来策略沙箱、AI 指标沙箱、用户代码隔离目标冲突。
- 用户代码执行面一旦被突破，影响面过大。

改进要求：

- 明确记录高权限配置的真实用途。
- 如非必要，移除 `privileged` 和 `NET_ADMIN`。
- 若确实需要网络能力，应拆分为独立低权限 worker 与受控网络代理。
- 用户策略、用户指标、AI 生成代码不得在高权限主进程内执行。

验收标准：

- 策略服务主容器默认非特权运行。
- 用户代码 worker 有 CPU、内存、网络、文件系统和运行时长限制。
- 策略服务主进程崩溃域与用户代码崩溃域隔离。

## 5. P1：用户体验核心缺陷

### 5.1 功能成熟度与导航不一致

当前部分策略实验、市场状态、资产库等能力存在临时隐藏或注释式开关。

问题：

- 菜单隐藏与路由访问、RPC 调用、权限控制不一致。
- 用户可能通过深链接进入未成熟页面。
- 开发态、内测态、正式态缺少统一表达。

改进方案：

新增统一功能开关能力：

```text
feature_flags
  strategy_experiment
  market_regime
  strategy_asset
  ai_indicator_workbench
  mcp_tools
```

前端控制：

- 菜单是否显示。
- 路由是否允许进入。
- 页面是否显示“内测中 / 设计中 / 暂不可用”。
- 操作按钮是否可点击。

后端控制：

- RPC 是否允许调用。
- 对未开放能力返回 `PermissionDenied` 或 `FailedPrecondition`。
- 记录审计日志。

验收标准：

- 未开放功能不能仅通过 URL 访问。
- 菜单、路由、按钮、RPC 权限一致。
- 每个功能有 `hidden / internal / beta / stable / deprecated` 状态。

### 5.2 策略任务流割裂

当前策略模板、回测、调度、日志、健康、市场状态、策略实验分布在多个页面和弹窗中。

用户真实任务流是：

```text
写策略 -> 校验 -> 回测 -> 调参 -> 上线调度 -> 观察 -> 复盘 -> 优化
```

改进方案：

建设策略生命周期工作台：

```text
策略详情页
  - 基本信息
  - 代码与参数
  - 校验结果
  - 回测历史
  - 实验优化
  - 市场状态适配
  - 调度实例
  - 运行日志
  - 健康分析
  - 风险提示
```

验收标准：

- 用户可围绕单个策略完成从创建到上线、复盘、优化的闭环。
- 策略详情页能追溯模板版本、参数、回测、调度和运行结果。
- 回测、市场状态、策略实验不再作为割裂概念呈现，而是嵌入策略分析链路。

### 5.3 错误提示技术化

当前前端全局错误可能直接展示 ConnectRPC 方法名、错误码或内部消息。

问题：

- 用户不知道如何恢复。
- 权限错误、网络错误、业务前置条件错误混杂。
- 高频错误 toast 干扰操作。

改进方案：

建立错误分层与统一提示：

| 错误类型 | 用户提示 | 操作建议 |
|----------|----------|----------|
| 网络错误 | 后端连接失败，请稍后重试 | 重试 / 检查网络 |
| 未登录 | 登录已过期，请重新登录 | 跳转登录 |
| 权限不足 | 当前账号无此权限 | 联系管理员 |
| 前置条件不足 | 请先完成账户连接或策略校验 | 跳转对应页面 |
| 服务异常 | 服务暂不可用，错误已记录 | 复制诊断信息 |
| AI 超时 | 模型响应超时，可重试或切换模型 | 重试 / 切换模型 |

验收标准：

- 页面不直接暴露后端堆栈、内部错误或无意义错误码。
- 所有关键失败路径都有可操作建议。
- 错误日志仍保留诊断字段，但默认隐藏在“复制诊断信息”中。

### 5.4 高级能力缺少渐进式引导

AI 辩论、代码生成、回测、策略实验、市场状态、MCP 等能力对普通用户理解成本高。

改进方案：

首页和空状态按用户意图组织：

```text
我想：
  1. 创建一个策略
  2. 用 AI 帮我写策略
  3. 回测已有策略
  4. 上线一个策略调度
  5. 查看正在运行的策略
  6. 分析为什么亏损
```

每个高级页面必须提供：

- 首次进入引导。
- 示例输入。
- 当前步骤说明。
- 下一步建议。
- 风险提示。
- 可恢复操作。

验收标准：

- 新用户无需理解系统内部模块名，也能完成首个策略创建和回测。
- 高风险操作均有明确风险提示和人工确认。

## 6. P1：工程化核心缺陷

### 6.1 前端类型债集中在策略链路

现有技术债扫描显示，前端手写 `any/as any` 热点集中在：

- `StrategySchedulePage.tsx`
- `StrategyScheduleLogsPage.tsx`
- `StrategyTemplatePage.tsx`
- `ScheduleTable.tsx`
- `LogManagement.tsx`
- `accounts`
- `pythonStrategy` client
- `trading` client

风险：

- ConnectRPC 生成类型没有贯穿页面。
- 后端 proto 变化后前端不一定编译失败。
- 表单、表格、API DTO、页面 view model 混杂。

治理顺序：

1. `frontend/src/client/trading.ts`
2. `frontend/src/client/pythonStrategy.ts`
3. `StrategySchedulePage.tsx`
4. `StrategyTemplatePage.tsx`
5. `ScheduleTable.tsx`
6. 账户相关页面

目标分层：

```text
gen protobuf type
  -> client adapter return type
  -> hook view model
  -> page component props
```

验收标准：

- 新增策略链路代码不得引入新的 `any/as any`。
- 热点页面逐步改为生成类型、显式 view model 或 `unknown` + 窄化。
- 前端 lint 或脚本能统计 `any/as any` 趋势。

### 6.2 本地校验入口与 CI 不一致

当前 CI 包含后端、前端、proto drift 和行数检查，但本地 `Makefile verify` 覆盖不完整。

改进方案：

新增统一任务：

```text
verify-backend
verify-frontend
verify-strategy
verify-proto
verify-all
```

建议覆盖：

- `backend`: `go test ./... -short`、`go build -o /dev/null ./...`
- `frontend`: `npm run lint`、`npm run build`
- `strategy-service`: `python -m compileall`、项目已有测试脚本
- `proto`: `make proto` 后检查生成物 drift
- `line-limit`: `scripts/check-file-lines.py`

验收标准：

- 开发者本地执行 `make verify-all` 能覆盖 CI 主要检查。
- CI 和本地命令文档一致。
- 策略服务至少有轻量编译检查。

### 6.3 CI 缺少策略服务校验

策略服务未来将承载 ML、回测、用户指标、AI 代码校验和策略运行，是高风险执行面。

改进要求：

- CI 增加 strategy-service job。
- 第一阶段至少运行 `python -m compileall app tests`。
- 若已有测试框架，增加 `pytest`。
- 后续增加沙箱、回测、指标 runtime 的单元与集成测试。

验收标准：

- 策略服务语法错误不能进入主分支。
- 策略服务核心运行路径有基础测试覆盖。

### 6.4 后端服务注册过于集中

当前后端服务注册集中在 server 装配层，随着 ConnectRPC 服务数量增加会继续膨胀。

改进方案：

按领域拆分装配：

```text
internal/wire/auth.go
internal/wire/trading.go
internal/wire/strategy.go
internal/wire/ai.go
internal/wire/admin.go
internal/wire/market.go
```

或由 `app.Container` 提供 service group。

验收标准：

- 新增领域服务不需要修改一个超大注册函数。
- 服务依赖关系可按领域单独测试。

## 7. P1：数据流与接口治理

### 7.1 轮询治理

当前部分页面存在 `setInterval` 定时刷新，和数据流优先原则不一致。

改进要求：

- 梳理所有 `setInterval`、周期刷新和伪实时状态读取。
- 调度状态、运行日志、Job 进度、健康变化优先走 SSE 或 Connect streaming。
- 如确实无法流式实现，必须在设计文档中记录例外原因。

验收标准：

- 新增业务能力不再默认使用轮询。
- 调度状态和 Job 进度通过事件流实时更新。

### 7.2 EventSource token 传递风险

当前部分 SSE 场景使用 query 参数传递 access token。

风险：

- token 可能进入代理日志、浏览器历史或错误日志。

改进要求：

- 优先使用短期一次性 stream token。
- stream token 只允许访问特定 job / stream，且很快过期。
- 禁止将长期 access token 暴露在 URL 中。

验收标准：

- SSE URL 不再携带长期 access token。
- stream token 具备 scope、过期时间和审计记录。

### 7.3 MCP 工具面分级

MCP 工具面应按风险分级。

| 等级 | 工具类型 | 默认策略 |
|------|----------|----------|
| L0 | 只读 list/get | 默认可开启 |
| L1 | 分析任务 submit backtest | 需 scope |
| L2 | 生成资产 clone/create draft | 需确认 |
| L3 | 调度、实盘、交易 | 默认禁止 |

验收标准：

- MCP token 必须有 scope。
- 高风险工具默认不注册或不可用。
- 所有 MCP 写操作必须有审计记录和幂等键。

## 8. P2：文档与产品治理

### 8.1 文档权威索引不足

当前 `docs/` 中存在多个专项、`专项设计/未完成项`（已收敛为仅含大块延后类专项，见该目录 **目录说明.md**）、domains、remediation 和重名相近文档。

改进方案：

新增 `docs/架构与设计文档索引.md`，记录：

```text
当前权威文档
设计中未完成项
历史归档
领域文档
实施状态
最后复核时间
```

每篇设计文档头部建议增加：

```text
状态：实施中 / 设计中 / 已废弃 / 历史参考
权威性：高 / 中 / 低
对应模块：
最后复核日期：
```

验收标准：

- 新开发者能通过一个索引找到当前准绳。
- 未完成项和已实施项状态清晰。

### 8.2 AI 能力边界需要产品化

AI 相关能力应统一展示：

- 当前模型。
- 预计耗时。
- token 或费用消耗。
- 输出风险级别。
- 是否可直接用于实盘。
- 校验状态。
- 人工确认状态。
- 审计记录。

验收标准：

- AI 输出默认不被视为实盘可执行结果。
- AI 生成策略、AI 指标、AI 建议均必须经过校验和人工确认。

### 8.3 市场状态、回测、策略实验统一产品语言

用户视角应统一为：

```text
回测：策略表现评估
市场状态：行情环境解释
策略实验：候选策略 / 参数搜索
```

改进要求：

- 不让三者作为割裂模块呈现。
- 回测报告中展示按市场状态拆分表现。
- 策略实验评分中体现市场状态适配度。
- 市场状态作为解释层，而不是替代回测。

验收标准：

- 用户能理解三者关系。
- 策略详情页和回测报告中自然串联三类能力。

## 9. P2：前端设计系统治理

### 9.1 样式与组件抽象

当前页面存在较多 inline style 和局部视觉实现。

改进要求：

建立设计 token：

- color
- spacing
- radius
- shadow
- layout width
- status color
- risk level color

抽象通用组件：

- `PageHeader`
- `PageSection`
- `DataState`
- `RiskAlert`
- `JobProgress`
- `EmptyGuide`
- `ErrorState`
- `ConfirmDangerAction`

验收标准：

- 新页面优先使用设计系统组件。
- 风险状态、运行状态、Job 状态视觉一致。

### 9.2 国际化治理

当前部分 TSX 仍存在中文硬编码。

改进要求：

- 菜单、按钮、错误、表格列、Modal 文案进入 i18n。
- CI 或脚本增加 TSX 中文硬编码扫描。

验收标准：

- 主导航无硬编码中文。
- 新增页面默认提供中文和英文 key。

### 9.3 统一数据加载抽象

当前页面普遍手写 loading、error、refresh。

改进方案：

建立 hooks：

- `useAsyncTask`
- `usePageQuery`
- `useMutationWithToast`
- `useJobStream`
- `usePermissionedAction`

验收标准：

- 新页面不重复实现基础 loading/error/toast 模板。
- Job 和 SSE 交互有统一 hook。

## 10. P3：中长期能力建设

### 10.1 策略运行全链路可观测性

建议建立统一 trace 关联：

```text
strategy_template_id
schedule_id
run_id
job_id
account_id
order_id
risk_decision_id
market_regime_id
indicator_snapshot_id
```

用户应能追溯：

- 为什么这次下单。
- 使用了哪个策略版本。
- 当时参数是什么。
- 当时市场状态是什么。
- 风控是否拦截。
- AI 是否参与过。

### 10.2 资源预算模型

未来 AI 指标、策略实验、ML 训练、批量回测、MCP Job 都需要统一资源治理。

建议建立：

```text
user_quota
job_quota
worker_quota
indicator_quota
backtest_quota
ai_quota
```

验收标准：

- 每类异步或计算密集任务都有用户级和实例级资源限制。
- 超额任务返回明确错误并可被前端解释。

### 10.3 能力成熟度等级

每个能力应具备统一状态：

```text
hidden
internal
beta
stable
deprecated
```

建议初始状态：

| 能力 | 建议状态 |
|------|----------|
| Market Regime | `internal` / `beta` |
| Strategy Experiment | `internal` |
| Strategy Asset | `internal` |
| AI Indicator | `design` |
| MCP | `internal` |

验收标准：

- 前端、后端、文档均使用同一成熟度状态。
- 进入 beta 或 stable 必须满足验收清单。

## 11. 分阶段实施计划

### Phase 1：安全与治理基线

- 移除 compose 硬编码密钥。
- 增加 CORS 白名单。
- 明确并收敛 strategy-service 特权配置。
- 建立 feature flag 设计。
- 梳理轮询清单。
- CI 增加 strategy-service 轻量校验。

### Phase 2：核心用户路径重构

- 设计策略生命周期工作台。
- 串联策略模板、校验、回测、调度、日志、健康。
- 增加首页任务式入口和空状态引导。
- 建立统一错误提示和恢复动作。

### Phase 3：类型与接口收敛

- 收紧 `trading` / `pythonStrategy` client 类型。
- 策略页逐步去除 `any/as any`。
- 统一 Job/SSE hook。
- 建立 ConnectRPC 错误映射。
- 本地 `verify-all` 与 CI 对齐。

### Phase 4：AI 与资源安全体系

- AI 操作分级。
- MCP scope 与审计。
- 用户代码 worker 沙箱。
- 用户级资源配额。
- 指标、回测、实验、AI Job 纳入资源预算。

## 12. 最关键的缺陷清单

| 优先级 | 问题 | 影响 |
|--------|------|------|
| P0 | compose 硬编码数据库密码、JWT Secret、API Key | 生产安全风险 |
| P0 | 后端 CORS 全开放 | 跨域安全边界过宽 |
| P0 | strategy-service 特权过高 | 与用户代码沙箱冲突 |
| P1 | 调度页存在周期轮询 | 违背数据流优先原则 |
| P1 | 功能成熟度靠注释和局部隐藏 | 用户可能进入半成品功能 |
| P1 | 策略用户路径割裂 | 新用户难完成完整闭环 |
| P1 | 前端策略链路类型债重 | 可维护性和变更安全差 |
| P1 | 错误提示技术化 | 用户无法恢复 |
| P1 | CI 缺少 strategy-service 校验 | 高风险服务缺少质量门禁 |
| P2 | 文档权威索引不足 | 设计、历史、实施状态易混淆 |

## 13. 明确禁止

- 禁止新增业务 REST 接口作为优化方案默认实现。
- 禁止以轮询作为新实时能力默认方案。
- 禁止前端执行业务计算、风控计算、交易决策、市场状态计算或指标计算。
- 禁止在后端 API 进程内执行用户策略代码、AI 生成代码或用户自定义指标。
- 禁止未配置权限、审计和资源配额的 MCP 写操作。
- 禁止未成熟功能仅通过注释菜单隐藏而不做路由和 RPC 权限控制。
- 禁止将 AI 输出直接绑定实盘交易动作。
