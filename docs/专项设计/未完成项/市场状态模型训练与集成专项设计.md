## 1. 背景与问题

当前市场状态识别为后端 Go 代码中的固定规则：

```text
DetectMarketRegime
  -> KlineService.GetKlines
  -> detectRegimeFeatures
  -> market_regimes 入库
  -> 返回 MarketRegime
```

现状问题：

- `rule-v1` 只是写死在 `backend/internal/connect/market_regime_service.go` 的规则版本字符串。
- 阈值不可动态调整。
- 没有模型配置表。
- 没有模型训练、评估、发布、回滚流程。
- 没有策略服务侧 ML / LLM 参与。
- 前端不能编辑规则参数或查看模型版本生命周期。
- 与回测、策略实验、调度健康尚未形成闭环，单独页面价值偏弱。

本设计目标是把 Market Regime 从“孤立检测工具”升级为“市场上下文服务”。

## 2. 目标与非目标

### 2.1 目标

- 支持规则模型的阈值动态配置。
- 支持模型配置表与版本管理。
- 支持训练接口与异步训练 Job。
- 支持策略服务侧 ML 模型参与特征计算、训练、评估与推理。
- 支持 LLM 参与规则解释、特征建议和报告生成，但不让 LLM 直接决定交易。
- 支持前端用户通过 AI 生成自定义指标草稿，经校验后在策略服务长驻运行并持续产出指标数值。
- 支持将自定义指标数值作为市场状态识别、策略编写、回测分析和策略实验的输入特征。
- 支持前端查看模型版本、编辑规则参数、提交训练、切换激活版本。
- 接入回测报告，输出按市场状态拆分的策略表现。
- 接入策略实验，提供 `regime_fit` 评分因子。
- 接入调度健康，用于解释策略表现变化。

### 2.2 非目标

- 不让前端计算市场状态、策略评分或风控结论。
- 不让 Market Regime 直接触发真实交易。
- 不用 Market Regime 替代 RiskEngine 或 ExecutionGateway。
- 不新增业务 REST API。
- 不用周期轮询模拟训练进度。
- 不在第一阶段强制引入复杂深度学习模型。
- 不接入非 MT 市场实盘交易。
- 不允许 AI 生成的指标绕过沙箱、权限、资源限制和人工确认后直接长驻运行。
- 不允许自定义指标直接发出交易指令；指标只能产出数值、标签、解释和告警建议。

## 3. 总体定位

Market Regime 应作为行情与策略分析的上下文服务：

```text
Kline / Backtest Dataset
  -> Feature Extraction
  -> Built-in Indicators / Custom AI Indicators
  -> Regime Model Inference
  -> Regime Segments
  -> Backtest / Experiment / Schedule Health Context
```

它与回测的边界：

| 能力 | Market Regime | 回测 |
|------|---------------|------|
| 关注对象 | 市场环境 | 具体策略 |
| 是否需要策略代码 | 不需要 | 需要 |
| 是否产生模拟订单 | 不产生 | 产生 |
| 主要输出 | `regime`、特征、分段、置信度 | 收益、回撤、交易、资金曲线 |
| 主要用途 | 行情标签与解释上下文 | 策略历史表现验证 |

最终产品形态不应依赖独立菜单暴露，而应优先嵌入：

- 回测报告。
- 策略实验评分。
- AI 策略生成上下文。
- 调度健康分析。

## 4. 模型类型

### 4.1 规则模型

规则模型用于第一阶段落地，支持动态阈值配置。

示例参数：

| 参数 | 说明 | 默认建议 |
|------|------|----------|
| `trend_efficiency_threshold` | 趋势效率阈值 | `0.45` |
| `trend_return_threshold` | 趋势涨跌幅阈值 | `0.01` |
| `high_volatility_threshold` | 收益波动率阈值 | `0.018` |
| `average_range_threshold` | 平均振幅阈值 | `0.025` |
| `confidence_base` | 置信度基础分 | `0.5` |
| `confidence_efficiency_weight` | 效率权重 | `0.35` |
| `confidence_volatility_weight` | 波动权重 | `8` |
| `confidence_cap` | 置信度上限 | `0.95` |

### 4.2 ML 模型

**当前策略：先离线验证，按需接入。** 不急于建设训练管道。

ML 模型由策略服务负责训练和推理，后端负责模型元数据、版本、权限、Job 和审计。

候选算法：

- 决策树。
- 随机森林。
- Gradient Boosting。
- HMM / 状态切换模型。
- 聚类模型，例如 KMeans / GMM。

**实施策略（简化后）：**

1. **离线验证先行**：导出 K 线数据，在 Jupyter 中用 sklearn 跑 2-3 个模型，对比规则版的分类效果。若 ML 分类一致性提升 > 15%，再启动管道建设。
2. **推理优先，训练延后**：如果 ML 显著优于规则，先只接入推理路径（策略服务加载预训练模型文件），训练管道仍不建设，模型通过离线 notebook 产出。
3. **规则模型大概率够用**：市场状态分类（趋势/震荡/高波动）本质是低维问题，规则 + 可调阈值 + 参数扫描已能达到很好效果。ML 训练/评估/发布全链路建设成本高，边 际收益不确定。

### 4.3 LLM 参与边界

LLM 可参与：

- 解释市场状态变化。
- 建议新增特征。
- 生成训练报告摘要。
- 为 AI 策略生成提供 prompt 上下文。

LLM 不允许：

- 直接写入激活模型。
- 绕过评估发布模型。
- 直接决定真实交易方向、手数或开关。
- 替代 RiskEngine。

### 4.4 内置指标扩展 + LLM 建议（简化方案）

**原 AI 自定义指标方案（沙箱编译 / Runtime Pool / 长驻运行）延后。** 安全攻防与运维复杂度远超当前阶段业务价值，改用以下简化方案：

#### 4.4.1 内置指标库

从现有的 3 个内置特征（趋势效率、波动率、振幅）扩展为约 15 个标准技术指标：

| 类别 | 指标 | 说明 |
|------|------|------|
| 趋势类 | `adx`、`ma_trend`、`macd_divergence` | 趋势强度与方向 |
| 波动类 | `atr`、`bollinger_bandwidth`、`keltner_width` | 波动率多维度度量 |
| 动量类 | `rsi`、`stochastics`、`cci` | 超买超卖与动量 |
| 成交量类 | `volume_ratio`、`obv_trend` | 量价关系 |
| 结构类 | `support_resistance_proximity`、`pivot_count` | 支撑阻力与转折点 |

#### 4.4.2 LLM 特征建议替代 AI 代码生成

原始设计链路：

```text
用户自然语言描述指标 → AI 生成代码 → 沙箱校验 → Runtime 运行 → 产出数值
```

**简化链路：**

```text
用户自然语言描述需求
  → LLM 从内置库选择 2-3 个最匹配指标 + 推荐参数
  → 返回："建议使用 Bollinger 带宽(20,2) + ATR(14)，周期 H1"
  → 用户一键激活 → 后端定时任务计算内置指标 → 输出入库
```

核心区别：**不生成代码，不运行用户代码。** LLM 只做"选指标 + 推参数"，所有指标都是预编译、预测试的内置代码，零安全风险。

#### 4.4.3 指标输出

| 类型 | 示例 | 用途 |
|------|------|------|
| 数值序列 | `trend_strength = 0.72` | 模型特征、图表展示、策略条件 |
| 布尔信号 | `is_breakout = true` | 策略草稿条件、告警提示 |
| 标签 | `phase = accumulation` | 市场状态解释 |
| 多字段结构 | `support/resistance/volatility_band` | 可视化与策略编写辅助 |

#### 4.4.4 运行方式

- 指标在后端定时任务中计算（随 `DetectMarketRegime` 一起触发）。
- K 线闭合触发，`M1`/`M5`/`M15`/`H1` 周期。
- 指标输出写入 Redis 缓存或数据库，通过现有 ConnectRPC 查询。
- **不引入 Runtime Pool、沙箱、进程隔离等基础设施。**

#### 4.4.5 学习闭环（简化版）

- 用户可将某个内置指标标记为 Market Regime 特征候选。
- 参数扫描可选择内置特征参与评分。
- 回测和策略实验可记录指标值与策略表现的相关性。
- LLM 可基于指标表现建议调整参数或激活/停用指标。

### 6.3 参数扫描（简化替代 Job 事件）

`ScanRegimeThresholds` 同步执行，不经过 Job 系统：

```text
ScanRegimeThresholds (同步, 5-30 秒内返回)
  -> 对 K 线数据执行网格搜索
  -> 按标签一致性 / 稳定性评分
  -> 返回 Top 10 参数组合及评分
```

## 9. 前端能力设计

前端只负责展示和交互，不做业务计算。

### 9.1 模型列表页

展示：

- 名称。
- 版本。
- 类型。
- 状态。
- 是否 active。
- 最近训练时间。
- 评估指标。
- 创建人。

操作：

- 查看详情。
- 复制为新草稿。
- 激活。
- 回滚。
- 归档。

### 9.2 规则参数编辑

仅允许编辑 `draft` 模型。

表单字段示例：

- 趋势效率阈值。
- 趋势涨跌幅阈值。
- 高波动阈值。
- 平均振幅阈值。
- 置信度参数。
- 策略族映射。

提交后调用 `UpdateMarketRegimeModelConfig`。

### 9.3 参数扫描页（简化替代训练与评估页）

用户选择：

- 数据范围（symbol、timeframe、回看天数）。
- 参数扫描空间（每个阈值的 min/max/step，或使用预设范围）。
- 评分目标（标签一致性、稳定性、与历史 regime 的交叉熵等）。

提交后调用 `ScanRegimeThresholds`，同步返回 Top 10 参数组合：

```text
ScanRegimeThresholds
  -> 后端同步网格搜索（5-30 秒）
  -> 返回排名表格：每行一组参数 + 评分
  -> 用户选择一组 → 点"采用" → 自动填入 draft 模型配置
```

不需要异步 Job，不需要事件流订阅，不需要 `market_regime_model_runs` 表。

### 9.4 检测结果展示

检测结果应展示：

- 市场状态。
- 置信度。
- 模型版本。
- 模型类型。
- 特征摘要。
- 分段结果。
- 数据源 metadata。
- 可解释说明。

独立 `strategy/market-regime` 页面可保留为管理与调试入口，但默认不建议对普通前端导航开放。

### 9.5 内置指标管理（简化替代 AI 指标工作台）

前端应提供面向用户的指标管理，但只管理**内置指标**的启用/停用和参数调整，不生成或运行用户代码。

核心流程：

```text
用户描述需求（如"我想监控突破强度"）
  -> 调用 DescribeMarketFeatures（LLM 推荐内置指标+参数）
  -> 返回推荐："建议启用 Bollinger 带宽(20,2) + ATR(14)，周期 H1"
  -> 用户查看推荐理由和指标说明
  -> 用户一键激活 → 后端定时任务开始计算该指标
  -> 前端展示指标数值和图表
```

页面能力：

- 自然语言描述需求获取 LLM 推荐。
- 查看可用内置指标列表（15 个，按类别分组）。
- 勾选/取消勾选激活的指标。
- 调整指标参数（周期、乘数等）。
- 展示指标曲线和最新值。
- 支持将指标标记为 Market Regime 特征候选。

前端限制：

- 不执行指标代码。
- 不直接访问数据库或行情流。
- 不允许指标输出直接绑定真实交易动作。

> **与原始 AI 指标工作台的核心区别**：不展示/编辑/校验/发布用户代码，只管理预置指标的开关和参数。安全风险为零。

## 10. 与回测集成

### 10.1 回测前

可选行为：

- 回测提交时选择是否执行市场状态分段。
- 默认使用 active 模型。
- 允许高级用户选择指定模型版本用于分析。

### 10.2 回测中或回测后

建议在回测结果 metadata 中加入：

```text
regime_analysis: {
  model_id,
  model_version,
  segments,
  data_range,
  feature_summary,
  indicator_features,
  performance_by_regime
}
```

`performance_by_regime` 示例：

| 字段 | 说明 |
|------|------|
| `regime` | 市场状态 |
| `trade_count` | 交易数 |
| `net_profit` | 净收益 |
| `win_rate` | 胜率 |
| `max_drawdown` | 最大回撤 |
| `sharpe` | Sharpe |
| `avg_holding_time` | 平均持仓时间 |

### 10.3 价值

回测不只回答“策略赚不赚钱”，还应回答：

```text
策略在什么市场状态下赚钱，什么市场状态下亏钱。
```

自定义指标接入后，还应回答：

```text
哪些指标数值变化与策略盈利、亏损、回撤或市场状态切换相关。
```

## 11. 与策略实验集成

策略实验评分应增加 `regime_fit` 组件。

示例评分结构：

```text
overall_score
  = return_score
  + drawdown_score
  + stability_score
  + sample_penalty
  + regime_fit_score
```

`regime_fit_score` 可考虑：

- 候选策略声明的适用市场与实际盈利市场是否一致。
- 是否只在极少数 regime 下盈利。
- 是否在高波动状态下回撤异常。
- 是否跨 regime 稳定。

策略实验候选结果应展示：

- 最适用市场状态。
- 不适用市场状态。
- 按 regime 拆分收益和风险。
- AI 优化建议。

## 12. 与调度健康集成

调度健康不应只展示运行错误和收益变化，还应解释市场环境变化。

建议新增分析：

| 能力 | 说明 |
|------|------|
| 最近市场状态 | 当前 symbol / timeframe 的 active 模型识别结果 |
| 状态切换事件 | 从 trend 到 range 或 high volatility 的切换 |
| 策略适配度 | 当前策略历史上是否适合该 regime |
| 风险提示 | 高波动或状态不匹配时提示人工复核 |

约束：

- 调度健康可提示风险，不直接停止真实交易。
- 自动停用或降级必须仍经过既有调度、风控与权限机制。
- 不得绕过 RiskEngine。

### Phase 3：智能规则调优 ✅ 已完成

**简化替代原"训练与评估 Job + ML 参与"两个 Phase。**

- 后端实现 `ScanRegimeThresholds`：同步网格搜索参数空间，按标签一致性/稳定性评分，返回 Top 10。

- 前端内置指标管理页：选择账号/品种/周期 → 获取最新指标 → 查看实时数值。
- 不引入：沙箱编译、Runtime Pool、用户代码执行、3 张指标相关表。

### Phase 5：回测报告集成 ✅ 已完成

- 回测完成后自动调用 `computeRegimeAnalysis` 对回测区间做批量检测。
- 按检测出的 regime 分段聚合策略表现（trade_count、net_profit、win_rate、max_drawdown、sharpe）。
- 自动写入 `backtest_runs.regime_analysis`。
- 前端报告抽屉展示按市场状态拆分的表现。

### Phase 6：策略实验排名展示 ✅ 已完成

- 前端策略实验提交表单接入 `MarketRegimeContext` 选择（可选）。
- 后端 `regime_fit` 评分根据市场状态进行参数偏向调整。
- 前端候选列表展示 `Regime适配` 分数。

### Phase 7：调度健康增强 ✅ 已完成

- 调度健康接入 regime 切换检测（对比最近两次检测结果）。

- 关键指标异常/漂移提示。

## 16. 明确禁止

- 禁止新增业务 REST CRUD 作为模型管理接口。
- 禁止前端直接计算 regime。
- 禁止前端直接修改 active 模型。
- 禁止训练过程使用周期轮询获取进度。
- 禁止 LLM 直接激活模型或直接给出交易执行指令。
- 禁止前端执行 AI 生成指标代码。
- 禁止后端 API 进程直接长驻运行用户自定义指标。
- 禁止未通过沙箱校验和人工确认的指标长驻运行。
- 禁止自定义指标直接发出下单、平仓、调仓或调度启停指令。
- 禁止第一阶段开放 tick 级高频用户自定义指标。
- 禁止 Market Regime 绕过 RiskEngine 影响真实交易。
- 禁止用 Market Regime 结果直接开启、停止或修改真实交易调度。
- 禁止把 K 线缓存作为真实交易实时行情权威。
