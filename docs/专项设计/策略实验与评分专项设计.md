# 策略实验与评分专项设计

本文档定义 AntTrader 借鉴 QuantDinger AI 实验系统时的参数实验、批量回测、多因子评分与 LLM 多轮优化边界。

## 1. 目标与非目标

目标：

- 先实现确定性的非 LLM 参数实验。
- 支持参数空间、候选生成、批量回测、评分排序和结果沉淀。
- 将策略评分放在后端或策略服务计算。
- 为后续 LLM 多轮优化提供安全闭环。

非目标：

- 不允许 LLM 直接启用真实交易调度。
- 不在前端计算策略评分、风控结论或调度建议。
- 不把策略实验结果直接作为交易指令。
- 不在第一阶段实现公开策略市场或付费优化。

## 2. 现有依赖模块

- CodeAssistService：AI 生成策略与草稿。
- PythonStrategyService：策略校验、执行沙箱、回测计算。
- BacktestRunService：回测任务、结果入库与查询。
- ObjectiveScoreService：多因子评分的后端或策略服务入口。
- StrategyTemplateService：模板草稿、发布、校验与调度启用前检查。
- RiskEngine：调度和交易前的最终风控入口。

## 3. QuantDinger 可借鉴点

- 参数进化、网格搜索、随机搜索。
- 市场状态识别作为策略优化上下文。
- 多因子评分与候选排名。
- 多轮优化、早停与最佳候选沉淀。

## 4. AntTrader 实施边界

策略实验只产生候选参数、候选草稿、评分和建议，不产生真实交易行为：

```text
StrategyTemplate
  -> Parameter Space
  -> Candidate Generator
  -> Batch Backtest
  -> Objective Score
  -> Ranked Candidates
  -> Draft / Report
```

启用调度仍必须经过模板校验、回测建议、RiskEngine 与现有调度流程。

## 5. 数据模型草案

策略实验建议字段：

| 字段 | 说明 |
|------|------|
| `id` | 实验 ID |
| `user_id` | 所属用户 |
| `base_template_id` | 基准模板 |
| `status` | `queued` / `running` / `succeeded` / `failed` / `cancelled` |
| `parameter_space` | 参数空间定义 |
| `search_method` | `grid` / `random` / `llm_round` |
| `max_candidates` | 最大候选数 |
| `objective` | 评分目标 |
| `market_regime_ref` | 可选市场状态引用 |
| `best_candidate_id` | 最佳候选 |
| `job_id` | 通用 Job ID |
| `created_at` | 创建时间 |
| `finished_at` | 完成时间 |

候选建议字段：

| 字段 | 说明 |
|------|------|
| `id` | 候选 ID |
| `experiment_id` | 实验 ID |
| `parameters` | 参数组合 |
| `draft_code_ref` | 可选候选草稿引用 |
| `backtest_run_id` | 回测结果 |
| `score` | 总分 |
| `grade` | 评级 |
| `score_components` | 分项评分 |
| `rank` | 排名 |
| `summary` | 摘要 |
| `recommendation` | 建议 |

## 6. 参数空间

参数空间应由后端校验，示例结构：

```text
parameter_space
  symbol: fixed or list
  timeframe: fixed or list
  numeric_param:
    min
    max
    step
  enum_param:
    values
```

约束：

- 候选总数必须有上限。
- 参数类型必须与模板参数 schema 匹配。
- 非法参数不得进入回测队列。
- 前端只提交参数空间和展示校验错误。

## 7. 评分口径

策略评分应由后端或策略服务计算。建议组件：

- 收益分。
- 年化收益分。
- Sharpe 分。
- 盈亏比分。
- 胜率分。
- 回撤分。
- 稳定性分。
- 样本量惩罚。
- regime fit 分。

输出：

```text
overall_score
grade
components
summary
recommendation
```

评分必须记录版本，避免后续评分公式变化导致历史结果无法解释。

## 8. ConnectRPC 接口草案

```text
StrategyExperimentService
  SubmitStrategyExperiment
  GetStrategyExperiment
  ListStrategyExperiments
  CancelStrategyExperiment
  ListExperimentCandidates
  GetExperimentCandidate
  PromoteCandidateToDraft
```

长任务事件通过通用 Job：

```text
JobService
  SubscribeJob
```

`PromoteCandidateToDraft` 属写能力，Agent 调用必须具备 `W` scope；若触发回测或实验，还必须具备 `B` scope。

## 9. LLM 多轮优化边界

第二阶段再引入 LLM：

```text
DetectRegime
  -> BuildPrompt
  -> GenerateCandidates
  -> BatchBacktest
  -> ScoreCandidates
  -> NextRound
  -> EarlyStop
  -> BestDraft
```

建议默认：

| 参数 | 默认 |
|------|------|
| `max_rounds` | 3 |
| `candidates_per_round` | 5 |
| `early_stop_score` | 82 |

LLM 只能生成候选参数或候选草稿，不能直接发布模板、启用调度或下单。

## 10. 前端职责

前端只负责：

- 提交参数空间。
- 展示 Job 进度。
- 展示候选排名、分项评分和报告。
- 触发候选提升为草稿的用户操作。

前端不得：

- 计算评分。
- 判断候选是否可上线。
- 跳过后端校验直接启用调度。

## 11. 分阶段实施

1. 非 LLM 参数实验：grid/random、候选上限、批量回测。
2. 多因子评分与排名。
3. 候选提升为草稿。
4. Market Regime 作为评分上下文。
5. LLM 多轮优化与早停。

## 12. 验收标准与禁止事项

验收标准：

- 同一实验的候选、回测与评分可追踪。
- 评分公式版本可查询。
- 实验失败时有稳定错误码和失败阶段。
- 候选提升为草稿后仍需走策略校验。

禁止事项：

- 禁止 LLM 直接启用真实交易调度。
- 禁止前端计算策略评分。
- 禁止实验绕过回测与模板校验。
- 禁止实验结果直接转化为交易指令。

## 13. 当前页面用法与用户理解成本

`/strategy/experiments` 页面用于基于既有策略模板发起参数实验。用户选择基准模板、搜索方式、评分目标与参数空间后，后端生成候选参数组合、执行评分并沉淀候选结果。

当前页面的核心问题是参数空间仍要求用户直接填写 JSON。该方式适合开发和高级用户验证，但对普通用户存在明显理解门槛：

- 用户不一定知道模板有哪些可调参数。
- 用户不一定知道参数名必须与模板参数 schema 或策略代码约定一致。
- 用户不一定知道参数单位，例如百分比、点数、ATR 倍数或价格偏移。
- 用户不容易估算组合数量，可能无意提交过大的实验。
- 用户不容易判断哪些参数适合实验，哪些参数属于策略结构或风控敏感项。

因此，后续产品化升级不应长期依赖手写 JSON，而应将参数空间配置表单化、模板化和后端 schema 化。

## 14. 搜索方式说明

搜索方式表示后端如何从参数空间中选择候选组合。

### 14.1 `grid`

`grid` 表示网格搜索。后端按参数空间中的取值做笛卡尔积展开，在候选上限内生成参数组合。

适用场景：

- 参数数量较少。
- 每个参数候选值较少。
- 用户希望结果稳定、可复现。

风险：

- 参数维度或候选值过多时组合数量会快速膨胀。
- 必须在后端强制执行候选总数上限。

示例：

```json
{
  "fast_ma": [5, 10, 20],
  "slow_ma": [30, 60],
  "risk_pct": [0.5, 1.0]
}
```

### 14.2 `random`

`random` 表示随机搜索。后端从参数空间中随机抽取候选组合，而不是完整穷举。

适用场景：

- 参数数量较多。
- 参数范围较宽。
- 用户希望先快速探索大致方向。

风险：

- 结果存在随机性。
- 必须记录随机种子或候选生成版本，以便历史实验可解释。

### 14.3 `auto` / `bayesian` / `llm_round`

此类搜索方式表示由系统自动或智能选择候选。当前阶段不应承诺完整贝叶斯优化或 LLM 自主上线能力。若实现，应保持如下边界：

- 智能搜索只能生成候选参数或候选草稿。
- 智能搜索不能直接发布模板。
- 智能搜索不能直接启用调度。
- 智能搜索不能直接产生交易指令。

## 15. 评分目标说明

评分目标表示实验候选的排序偏好。评分计算必须由后端或策略服务完成，前端只提交目标并展示结果。

常见目标定义如下：

| 目标 | 含义 | 适用场景 | 风险 |
|------|------|----------|------|
| `balanced` | 综合收益、回撤、稳定性、胜率、样本量等指标 | 默认推荐 | 需要明确评分公式版本 |
| `sharpe` | 优先风险调整后收益 | 稳健型策略筛选 | 低交易次数样本可能失真 |
| `return` | 优先总收益或收益率 | 激进初筛 | 可能选出高回撤候选 |
| `drawdown` | 优先控制最大回撤 | 保守策略筛选 | 可能牺牲收益能力 |
| `profit_factor` | 优先盈利因子 | 盈亏结构筛选 | 可能受极端交易影响 |
| `win_rate` | 优先胜率 | 高频或短线策略初筛 | 高胜率不代表高收益 |

建议默认目标为 `balanced`。如果用户不理解目标含义，前端应提供中文解释型选项，例如“综合平衡”“稳健优先”“收益优先”“回撤优先”。

## 16. 参数空间产品化升级方案

后续升级应将参数空间从手写 JSON 升级为“模板参数 schema + 表单配置 + 后端校验”的闭环。

### 16.1 模板参数 schema

每个策略模板应声明可实验参数 schema。建议字段：

| 字段 | 说明 |
|------|------|
| `name` | 参数名，必须与策略模板参数一致 |
| `label` | 用户可读名称 |
| `type` | `number` / `integer` / `enum` / `boolean` |
| `unit` | 单位，例如 `%`、`pips`、`bars`、`ATR` |
| `default` | 默认值 |
| `min` | 最小值 |
| `max` | 最大值 |
| `step` | 推荐步长 |
| `values` | 枚举值 |
| `risk_level` | `low` / `medium` / `high` |
| `description` | 参数说明 |

### 16.2 参数空间表单化

前端不应让普通用户直接填写 JSON。建议提供：

- 参数下拉选择。
- 数值列表输入。
- 范围加步长输入。
- 枚举多选。
- 是否参与实验开关。
- 组合数量实时预估。
- 超过上限时阻止提交并提示原因。

### 16.3 推荐参数模板

系统应根据策略类型提供预设参数空间：

- 均线类：快线周期、慢线周期、风险比例。
- RSI 类：RSI 周期、超买阈值、超卖阈值。
- 突破类：回看窗口、突破缓冲、止损止盈。
- 风控类：仓位比例、止损、止盈、最大持仓。

### 16.4 后端校验

后端必须作为参数空间权威校验方：

- 校验参数是否存在于模板 schema。
- 校验类型、范围、单位和候选数量。
- 校验高风险参数是否需要额外确认。
- 校验候选总数不超过系统上限。
- 生成稳定错误码供前端展示。

前端不得自行认定参数空间合法。

## 17. 实验结果保留策略

当前实验结果写入数据库，具备持久化能力。实验主记录和候选记录不应只存在前端状态中。

当前语义应定义为：

```text
持久化保留，但尚非完整归档资产管理。
```

后续应补充以下能力：

- 实验归档。
- 实验收藏。
- 实验删除或软删除。
- 低价值实验自动清理策略。
- 实验报告导出。
- 评分公式版本记录。
- 候选生成算法版本记录。
- 参数空间快照记录。

历史实验必须可解释。即使评分公式或候选生成逻辑升级，历史候选也应能说明当时使用的评分版本、生成版本和参数空间。

## 18. 实验到上线闭环

实验候选不得直接上线。安全闭环应保持如下流程：

```text
实验候选
  -> 提升为草稿模板
  -> 人工检查或系统校验
  -> 发布模板
  -> 创建调度草案
  -> 调度前检查
  -> RiskEngine 校验
  -> ExecutionGateway 执行
```

当前已支持的方向是“候选提升为草稿模板”。后续不建议实现“候选一键实盘上线”，而应实现“候选生成调度草案”。

### 18.1 调度草案

调度草案不是正式上线。它只预填以下信息：

- 策略模板。
- 推荐品种。
- 推荐周期。
- 推荐参数。
- 推荐风控配置。
- 推荐调度频率。
- 实验来源 ID。
- 候选 ID。

用户必须确认后才能创建正式调度。

### 18.2 上线前检查清单

调度草案转正式调度前必须完成：

- 模板已发布。
- 模板代码校验通过。
- 账户归属和权限校验通过。
- 品种可交易。
- 行情数据可用。
- 最近回测或复核回测通过。
- 风控配置完整。
- 最大回撤、仓位、频率等指标未超过限制。
- RiskEngine 返回允许。

### 18.3 候选复核回测

实验阶段的候选分数不能直接作为上线依据。建议在候选生成调度草案前执行一次标准复核回测：

- 使用统一回测口径。
- 使用最新数据集。
- 输出标准风险指标。
- 记录复核回测 ID。
- 复核失败则禁止生成正式调度。

## 19. 推荐实施阶段

后续升级建议按以下顺序推进：

1. **参数 schema 与后端校验**
   - 为策略模板补充参数 schema。
   - 实现参数空间合法性校验和候选数量预估。

2. **前端参数空间表单化**
   - 用表单替代手写 JSON。
   - 支持高级模式保留 JSON 编辑。

3. **评分目标中文化**
   - 将 `balanced`、`sharpe`、`return` 等目标映射为中文选项。
   - 增加目标说明和适用场景提示。

4. **实验归档与报告**
   - 增加收藏、归档、删除、导出。
   - 记录评分公式版本和候选生成版本。

5. **候选调度草案**
   - 候选只能生成草案。
   - 草案必须经过人工确认和后端上线前检查。

6. **复核回测与上线门禁**
   - 对候选执行标准复核回测。
   - 后端根据复核结果、风控结果和模板状态决定是否允许进入调度流程。

## 20. 升级验收标准

升级完成后应满足：

- 普通用户无需手写 JSON 即可发起实验。
- 参数空间错误由后端返回稳定、可读的错误信息。
- 页面展示候选数量预估。
- 实验结果刷新、重启后可继续查询。
- 历史实验具备评分版本和参数空间快照。
- 候选不能直接上线，只能生成草稿模板或调度草案。
- 调度草案转正式调度必须经过模板校验、复核回测和 RiskEngine。
- 前端仍只负责展示、交互和轻量 UI 状态，不承担评分、风控或上线决策。

## 21. Agent 可消费的行情上下文设计

当前 `MarketRegimeService` 只提供检测和查询市场状态的基础能力，尚未形成面向 Agent 的完整“行情分析上下文接口”。如果目标是由 AI 编写策略、分析行情并提出决策建议，则 Market Regime、K 线特征、实验结果、策略资产和历史评分都应被组织为 Agent 可消费的上下文包。

该能力的目标不是让 Agent 直接下单，而是让 Agent 在以下任务中有可靠依据：

- 编写新策略。
- 优化已有策略。
- 解释当前行情适合哪类策略。
- 选择实验参数空间。
- 为候选策略生成报告。
- 生成调度草案或风控建议。

核心原则：

- Agent 只能读取后端整理后的上下文。
- Agent 不能直接访问数据库、交易适配器或账户密钥。
- Agent 不能直接发布模板、启用调度或下单。
- Agent 生成的策略、参数和建议必须回到后端校验、回测、评分和风控流程。

## 22. 行情上下文包

建议新增统一的 `MarketAnalysisContext`，作为 Agent 编写策略和分析行情时的输入。

建议结构：

```text
MarketAnalysisContext
  request_id
  user_id
  account_id
  symbol
  timeframe
  as_of
  kline_window
  regime
  regime_confidence
  regime_features
  technical_features
  volatility_profile
  support_resistance
  strategy_family_suggestions
  recent_experiment_refs
  suitable_asset_refs
  risk_hints
  data_quality
  model_versions
```

字段说明：

| 字段 | 说明 |
|------|------|
| `request_id` | 本次上下文生成 ID，便于审计和追踪 |
| `account_id` | 行情所属账户 |
| `symbol` | 交易品种 |
| `timeframe` | 周期 |
| `as_of` | 上下文生成时间 |
| `kline_window` | 使用的 K 线范围、数量和时间边界 |
| `regime` | 市场状态，例如 `trend_up`、`trend_down`、`range`、`high_volatility` |
| `regime_confidence` | 市场状态置信度 |
| `regime_features` | 市场状态特征，例如趋势收益、波动率、效率比 |
| `technical_features` | 技术指标特征，例如 RSI、均线、ATR、MACD 摘要 |
| `volatility_profile` | 波动率分层和风险提示 |
| `support_resistance` | 支撑阻力摘要，非交易指令 |
| `strategy_family_suggestions` | 适合的策略族建议 |
| `recent_experiment_refs` | 最近相关实验引用 |
| `suitable_asset_refs` | 适合当前状态的策略资产引用 |
| `risk_hints` | 风控提示，例如降仓、暂停、扩大止损 |
| `data_quality` | 数据质量，例如 K 线数量是否足够、是否存在缺口 |
| `model_versions` | regime、指标、评分等模型版本 |

该上下文包由后端生成，前端和 Agent 只能读取与提交需求，不得自行计算权威结论。

## 23. Agent Gateway 接口设计

建议在现有 Agent Gateway 基础上新增面向策略生成和行情分析的 ConnectRPC 接口。

```text
AgentMarketContextService
  BuildMarketAnalysisContext(BuildMarketAnalysisContextRequest) returns (MarketAnalysisContext)
```

```text

  account_id
  symbol
  timeframe
  kline_count
  include_technical_features
  include_recent_experiments
```

## 24. Agent 决策依据的数据流

推荐数据流如下：

```text
KlineService
  -> MarketRegimeService
  -> TechnicalFeatureService
  -> StrategyExperimentService
  -> StrategyAssetService
  -> AgentMarketContextService
  -> AgentStrategyAuthoringService / AgentExperimentAdvisorService / AgentScheduleAdvisorService
  -> StrategyTemplateService / StrategyExperimentService / ScheduleDraft
  -> Backtest / Score / RiskEngine
```

Agent 不应直接拼接各服务数据。上下文聚合应由后端完成，原因：

- 可以统一鉴权。
- 可以统一数据质量检查。
- 可以记录上下文版本。
- 可以保证 Agent 输入可审计。
- 可以避免前端承担业务聚合逻辑。

## 25. Prompt 输入边界

Agent 调用 LLM 时，Prompt 应由后端构造。建议只包含以下内容：

- 市场状态摘要。
- 技术特征摘要。
- 适合策略族。
- 用户目标。
- 风控约束。
- 可用模板或资产摘要。
- 禁止事项。

禁止在 Prompt 中包含：

- 账户密钥。
- 数据库连接信息。
- 用户隐私信息。
- 未脱敏的敏感日志。
- 可绕过后端校验的内部指令。

Prompt 中必须明确：

```text
你只能生成策略草稿、参数建议、实验计划或调度草案。
你不能发布策略。
你不能启用调度。
你不能下单。
所有建议必须经过后端校验、回测、评分和风控。
```

## 26. 上下文审计与可解释性

为保证 Agent 决策可追踪，应新增上下文审计记录。

建议表：

```text
agent_market_contexts
  id
  user_id
  account_id
  symbol
  timeframe
  kline_from
  kline_to
  regime_id
  context_snapshot
  model_versions
  created_at
```

Agent 生成的策略草稿、实验计划和调度草案都应保存来源：

- `source_context_id`
- `source_agent_id`
- `source_prompt_version`
- `source_model`
- `source_request_id`

这样后续可以解释：

```text
这个策略为什么这样写？
它基于什么市场状态？
它用了哪些行情特征？
它参考了哪些实验结果？
它的风险提示是什么？
```

## 27. 后续升级实施建议

建议按以下阶段升级：

1. **Agent 行情上下文接口**
   - 新增 `AgentMarketContextService`。
   - 聚合 Market Regime、K 线窗口、技术特征和风险提示。

2. **上下文审计表**
   - 持久化 `MarketAnalysisContext` 快照。
   - 为后续策略草稿和实验计划提供来源引用。

3. **Agent 策略草稿生成**
   - 基于上下文生成策略模板草稿。
   - 强制进入 `StrategyTemplateService` 校验流程。

4. **Agent 实验计划建议**
   - 基于上下文和模板 schema 生成参数空间。
   - 后端校验后才能提交实验。

5. **Agent 调度草案建议**
   - 基于候选、上下文和风控约束生成调度草案。
   - 禁止直接启用调度。

6. **闭环评分与反馈**
   - 草稿回测。
   - 实验评分。
   - 将结果反馈给 Agent 进行下一轮优化。

## 28. Agent 接口验收标准

完成后应满足：

- Agent 能通过 ConnectRPC 获取统一行情分析上下文。
- Agent 生成策略时能引用 `market_context_id`。
- Agent 生成实验计划时参数空间必须经过后端 schema 校验。
- Agent 生成调度草案时不能绕过调度前检查。
- 所有 Agent 输出都能追溯到上下文快照、Prompt 版本和模型版本。
- 前端不承担行情分析、评分、风控和决策聚合。
- Market Regime 结果可以作为 Agent 分析行情和编写策略的依据，但不能单独作为交易指令。
