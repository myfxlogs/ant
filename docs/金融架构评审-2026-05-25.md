## 总判断

> **杠杆点放错了**。代码体现的是"机构级低延迟系统"的工程质感（Buffer engine、tick dedup、P99<500ms SLO、三方对账），但业务实质是 **"散户托管 MT broker + AI 生成策略"**。这俩世界的痛点完全不重叠——**真正会吃掉用户收益的金融风险，在当前架构里几乎全部缺位**。

## 一、致命级（一定亏钱 / 监管风险）

### 1. 风控是仓位计数器，不是真风控

`backend/internal/risksvc/precheck.go:31-79` 的全部 4 条规则：
- `MaxPositionsPerSymbol` / `MaxTotalPositions` / `MaxExposurePerAccount` / `MaxMarginUtilizationPct`

这是 **零售 MT EA 默认面板** 的水平，不是量化风控。缺的真东西：

| 缺失项 | 后果 |
|---|---|
| **跨标的相关性集中度** | 同时 long EURUSD + short USDJPY，仓位计数=2 看似分散，实际是 2× USD 空头——一次美元飙升双倍亏 |
| **VaR / CVaR / 历史情景压力测试** | 1987/2015 SNB/2020 covid 重放下组合的尾部损失，目前 0 计算 |
| **波动率目标 sizing**（vol-targeting） | 当前无 sizer，策略发什么 lot 就下什么 lot——VIX 从 12→40 时仓位风险暴涨 3× |
| **drawdown-based scaling** | 净值跌 5%/10%/20% 自动缩 lot；无此机制下连亏路径就是死亡螺旋 |
| **流动性风险（仓位/ADV）** | 散户单 0.5 lot 通常无碍，但 SaaS 聚合 1000 用户后整体可能撞到 broker 限额 |
| **Greeks（即使 CFD）** | delta/gamma 暴露，做差价合约组合时必须 |

更糟的代码细节：`MaxExposurePerAccount: 100000, // 1 standard lot` 注释错——standard lot 是 100k **base currency**，EURUSD 是 100k EUR ≈ 108k USD，不是 100k USD。**这种 unit 错位在金融工程里就是钱**。

**修法**：把 `risksvc` 拆成 `HardLimit`（硬阻断：KYC/jurisdiction/margin floor）+ `PositionSizer`（vol-target / Kelly fraction / risk-parity → 推荐 lot 而非二值 deny）。后者是 BlackRock Aladdin / GS SecDB 的标准做法。

### 2. OMS 状态机缺关键 broker 行为状态

`backend/internal/oms/statemachine.go:62-110` 实现的转换图缺：

- **SLIPPAGE_EXCEEDED**：broker 给的价偏离请求价超容差时的拒绝路径
- **REQUOTED**：mtapi 实际会触发 requote，当前直接当 REJECTED 处理 → 丢失合法重报机会
- **MARGIN_CALL**：处于 margin call 时 OMS 应禁发新单，目前无该状态
- **UNKNOWN/RECONCILING**：**最危险的"幽灵订单"**——SUBMITTED 后 broker 30s 无响应（连接丢失/超时），订单可能已成可能未成。当前没有 timeout transition，重连后用户能看到双倍仓位
- spec/22 文档里写了 `MODIFY_PENDING/CANCEL_PENDING`，**但 `statemachine.go` 实际没实现**

**修法**：补全 timeout / unknown 路径，并强制 `mthub.Reconciliation`（已存在 `backend/internal/mthub/reconciliation.go`）在重连后**对账完成前** OMS 不接受新订单。

### 3. 幂等性方案在 failover 时不可靠

`backend/internal/mthub/idempotency.go`:
- Redis SETNX + 24h TTL
- **TTL 不够**：周五 22:00 关市 → 周一 05:00 开市 = 55h，跨过 TTL；用户重启重发命中已过期 key → 重复下单
- **failover race**：Redis 主从异步复制，主切从期间 SETNX 不可靠
- **未利用 broker 端幂等**：MT5 `OrderSendAsync` 的 `magic` 字段可做 broker-side 去重，**完全没用**

**修法**：双层幂等——Postgres advisory lock（强一致）+ broker `magic` 字段；Redis 仅作热缓存。TTL 至少 **96h**（覆盖 G7 假期）。

## 二、严重级（架构性偏差）

### 4. "数据质量"在错误的地方做检查

`backend/internal/mdgateway/quality.go` 只挡 `bid>ask` 等显式脏数据。真问题：

- **stale quote**：tick 10s 没更新但 broker 不主动推 → 策略以为价格还是旧价 → fade-the-spike 类策略全错
- **NFP/CPI 时点差宽化**：平时 0.1 pip 的 EURUSD 数据公布瞬间到 50 pip——挂单全部触发糟糕成交
- **swap rollover window**（broker 服务器 17:00 EST）：bar 应标记 `swap_window=true`，策略可选择跳过；ant 当前 bar 没这个 flag
- **三角套利一致性**：EURUSD × USDJPY ≈ EURJPY，偏离 5 pip 以上必然其中一条线 broker 有问题
- **broker holiday calendar**：圣诞节、中国春节、伊斯兰假期，每家 broker 不同，**当前完全无概念**

**修法**：把 `mdgateway` 从"管道"升级为"市场状态机"：

```
MarketState { last_quote, quote_age_ms, is_tradeable,
              spread_zscore, gap_marker, swap_window }
```

策略消费 `MarketState`，不是 raw tick。**Quality 从"挡脏数据"升级为"声明可交易性"**。

### 5. "回测/仿真/实盘统一代码路径"是过度抽象

`docs/spec/21-backtest-replay.md` §1：「**完全相同**的 factor → signal → execution 代码路径」

这是软件工程师的洁癖；**金融上这是过拟合温床**：

| 实盘有 | 回测如果"完全相同"会怎样 |
|---|---|
| Partial fill | 回测假设全成交 → 高估容量 |
| Slippage（实盘 0.5 pip） | 回测=0 滑点 → 高估收益 |
| Commission + swap | 默认 0 → 高频策略表面盈利实际亏 |
| Look-ahead bias | bar.Close 在收盘瞬间不可用，但回测能看到 → alpha 全是假的 |
| Survivorship bias | 跑路的 broker 数据没了 → 现存 broker 数据高估存活策略 |

**修法**：回测引擎应**故意比实盘更悲观**：
- 强制 commission/swap 模型（即使用户不填）
- 强制 slippage 模型（按 spread × √(lot/ADV) 给出滑点）
- 强制 fill probability < 1.0 在 limit order 上
- bar 消费严格遵守 `now ≥ bar.close_ts + finality_delay`（已有 `LoadFinalizedBars` 但**没强制策略侧时序**）

### 6. AI 策略生成是合规 + 信誉双重风险

`docs/spec/26-ai-strategy-generation.md`：「让普通人用自然语言生成可在实盘运行的策略」

金融视角下三个红色警报：

- **监管**：中国/美国/欧洲都可能触发投顾牌照问题——「提供具体交易建议」在多数司法辖区是受监管行为。**当前 spec 0 字符提及合规边界**
- **多重检验偏差**：用户对同一份历史用 100 条 prompt 测 → **必然**找到夏普看起来很好的那条（p-hacking）。需要 **Deflated Sharpe Ratio**（López de Prado 2014）+ 按 prompt 次数衰减 SR
- **缺强制 paper-gate**：AI 出的策略应**强制** N 天仿真且 OOS 表现 ≥ X% IS 表现 才能进实盘——spec 26 没要求

**修法**：加 `docs/spec/26.md §validation_gate`：
```
NL prompt → DSL → walk-forward (5-fold OOS)
   reject: train_sharpe - test_sharpe > 1.0 (overfitting)
   reject: max_DD > 20%, trade_count < 30
→ paper trading 14d forced gate
   reject: paper_return < 0.5 × backtest_return
→ 用户显式风险声明 + 2FA 才能进实盘
```

## 三、架构层缺失（中等优先级）

### 7. 时间维度的金融正确性

`ts_unix_ms` 一路用 UnixMs，**完全无时区/会话感知**：

- FX 24×5：NY 17:00 daily rollover——tick 在 17:00:00 vs 17:00:01 的 **swap 归属和 bar 归属都不同**
- bar period `"1m"/"5m"` 字符串：**没规范** "5m bar 是 [17:00:00, 17:04:59] 还是 [17:00:00.001, 17:05:00.000]"——边界错位在回测/实盘对比时会差出几个 bp 的 alpha
- **NTP 漂移**：`ArrivedUnixMs`（本机）vs `TsUnixMs`（broker 服务器），可差 100-2000ms——`ObserveE2eLatency` 把这个时钟差也算进延迟，**测出来的 P99 不是真延迟**

**修法**：引入 `SessionClock` 抽象：
```go
type SessionClock interface {
    SessionOpen(broker, symbol, t time.Time) bool
    BarBoundary(t time.Time, period string) (open, close time.Time)
    SwapWindow(broker string, t time.Time) bool
    BrokerClockOffsetMs() int64  // 单独可观测
}
```

### 8. 多账户聚合 / 平台风控缺失

ant 是 SaaS 多租户，但：

- **跨账户净敞口**：User A 在 broker X long EURUSD 1 lot，User B 在 broker Y short 1 lot——**真平台敞口=0，但保证金占用=2**。当前 `risksvc` 是单账户的
- **block trade 分配**：一个 AI 策略推到 100 个用户，部分成交时按什么规则分（FIFO / pro-rata / random）？**SEC Rule 17j-1 和 FINRA 4512 类规定要求公平分配**，当前无规则
- **平台总量限额**：所有用户 EURUSD 合计不能超过 broker 给本平台的限额——若不在 ant 侧先拦，触发 broker 拒单会 cascade 拒所有用户

### 9. 缺事件溯源（event sourcing），对账只能"近似"

当前：PG 存订单状态 + Redis 存幂等 + CH 存行情。三方不一致时**无法精确对账**——只能用 `mthub.Reconciliation` 拉 broker 当前持仓粗对。

**问题**：当 broker reconcile 显示 "broker 有 ticket=12345 但 PG 没有" 时，是 **submit 后崩溃** 还是 **broker 多发**？两者修复路径完全不同，但 ant 没法区分。

**修法**：用 NATS JetStream 已经在的事件流做 **append-only event log = 唯一事实源**，PG/Redis/CH 都是投影。对账 = 重放 event 重建。这不是大改造，因为 NATS 已部署。

### 10. 缺 SRE 级风控控制面

- **kill switch**：一键停所有用户所有订单（监管事故/异常波动响应必备）—— 0
- **strategy-level circuit breaker**：策略 30min 亏 X% 自动熔断 —— 0（broker 抖动有 spill 熔断 `docs/adr/0005`，但策略亏损无）
- **canary / shadow trading**：新版策略先在小账户跑 1 周再全量 —— 0
- **风控规则灰度发布 + 回滚** —— 0

## 四、对架构本身的两条根本性建议

### A. 诚实声明系统的"延迟段位"

mtapi.io 是**两层代理**（见 `mt-gateway` skill），typical 50-300ms。在这种基础上：

- 追求 `P99 < 500ms` 端到端是合理的
- 追求 **tick 级 strategy 决策**是不合理的——tick 到决策的可用窗口已经被两层代理吃光

**建议**：在 `docs/spec/27-signal-execution-slo.md` 顶部明确写：

> **本系统设计的最小有效持仓周期 = 5 分钟。 < 5min 的策略不应在本系统部署。**

这一行能避免下游对错误的预期做错误的工程投入（Buffer engine 0 延迟优化、tick dedup ringbuffer 等都可以**降级为 1s 批处理**，省 30% CPU）。

### B. 重新分层：按"金融正确性"而非"代码模块"

当前分层是 mdgateway / oms / risksvc / mthub / strategy-service —— 这是按**进程模块**分。建议按**金融语义**重分：

```
┌─ Tier-0  Trade Ledger (append-only PG/CH event log)
│           ←── 唯一事实源；任何状态都能从这里重算
│
├─ Tier-1  Live State Cache (Redis + in-memory)
│           ←── OMS state, position cache, idempotency
│           ←── 一致性靠 Tier-0 重放修复
│
├─ Tier-2  Derived Quantities (5s 重算)
│           ←── PnL, Greeks, VaR, margin, exposure
│
└─ Tier-3  Decision Surface (read-only Tier-2)
            ←── Strategy / RiskEngine / Sizer
```

**关键不变量**：所有状态必须能从 Tier-0 完全重建——这是金融系统对账的基础。当前 OMS state 是"主"而非"派生"，对账只能近似。

## 总结

| 维度 | 当前水平 | 应达水平 |
|---|---|---|
| 工程质量 | 机构级 OEMS 初版骨架 ✅ | （已合格）|
| 行情管道 | tick→bar→CH 完整 ✅ | （已合格）|
| 风控模型 | 零售 MT EA 增强版 ❌ | vol-target + VaR + 跨账户净敞口 |
| OMS 状态 | 缺 timeout/unknown/margin-call ❌ | 含幽灵订单恢复路径 |
| 幂等性 | Redis SETNX 单点 ⚠️ | PG advisory lock + broker magic |
| 回测保真 | "与实盘相同"过度抽象 ❌ | 强制比实盘悲观（commission/slippage/fill prob） |
| AI 策略 | 无 OOS / paper gate ❌ | walk-forward + 14d paper + Deflated SR |
| 对账 | 拉 broker 粗对 ⚠️ | event-sourced ledger 重放 |
| SRE 控制面 | 无 kill switch ❌ | kill switch + strategy circuit breaker |

**一句话**：**ant 当前的代码工程质量配得上一个机构级 OEMS 的初版骨架，但 spec 与代码体现出的金融模型停留在"零售 MT EA 增强版"水平。最该补的不是更多 tick 处理性能，而是：(1) 真风控（vol-targeting + drawdown scaling + 跨账户净敞口）、(2) AI 策略的 OOS/paper-gate 强制门、(3) 对账可重建的 event-sourced ledger。**

如果只能投入精力修一处，**第 1 条（风控升级）** 应该优先——这是用户最快真金白银受益、监管最快看到、且不需要改外围基础设施的改造。
