## 1. 项目背景与目标

> **文档导航**：默认单入口 **`docs/新成员一站式导读.md`**；规范新鲜度与易混副本见 **`docs/文档索引与维护状态.md`**；深度按日路径（选修）见 **`docs/新成员三日上手指南.md`**。

### 1.1 项目定位

AntTrader 是一个面向**外汇（Forex）量化交易**的 SaaS 平台，核心定位是：

- **多账户管理**：支持用户绑定多个 MT4/MT5 交易账户，统一管理资金、持仓与订单
- **AI 驱动策略生成**：通过多模型辩论（Debate）生成 Python 量化策略代码
- **策略调度与执行**：支持策略模板管理、定时调度、自动交易与风险控制
- **实时数据流**：提供行情报价、K 线数据、账户状态、交易事件的实时推送
- **回测与实验**：支持历史数据回测、策略参数进化实验、市场状态识别

### 1.2 目标用户

- **个人交易者**：通过 AI 辅助生成策略，降低量化交易门槛
- **策略开发者**：使用 Python DSL 编写策略，在沙箱中安全执行
- **管理员**：通过 Admin 后台管理用户、账户、交易监控与系统配置

### 1.3 核心目标

| 目标 | 说明 |
|------|------|
| **低门槛策略生成** | 通过自然语言描述，AI 自动生成可执行的 Python 策略代码 |
| **安全可靠执行** | 策略在隔离沙箱中运行，通过风控引擎限制单笔/单日风险敞口 |
| **实时性** | 行情、订单、账户状态的变化在秒级内推送至前端 |
| **可扩展性** | 支持多交易所适配（当前 MT4/MT5，未来可扩展币圈/美股） |

### 2.2 服务划分

| 服务 | 技术栈 | 职责 | 端口 |
|------|--------|------|------|
| **frontend** | React 19 + Vite + Tailwind CSS | 用户界面、Admin 后台 | 8012 (Nginx 8080) |
| **backend** | Go 1.25 + ConnectRPC | 核心业务逻辑、API 网关、流式推送 | 8080 |
| **strategy-service** | Python 3 + FastAPI | 策略执行、回测引擎、Python 沙箱 | 8081 |
| **postgres** | PostgreSQL 15 | 主数据库，存储用户、账户、订单、策略等 | 5432 |
| **redis** | Redis 7 | 缓存、分布式锁、会话、流式状态 | 6379 |

### 2.3 部署架构

- **容器化**: 全部服务通过 Docker Compose 编排
- **镜像仓库**: 支持 `hub.rat.dev` 与 `docker.io` 双源
- **环境变量**: 通过 `.env` 文件配置，支持 `VITE_API_URL`、`VITE_STREAM_URL` 等前端构建参数
- **健康检查**: PostgreSQL 与 Redis 通过 `healthcheck` 确保依赖就绪后启动后端

## 3. 核心功能模块实现逻辑

### 3.1 用户与账户管理

**文件位置**:
- 后端: `backend/internal/connect/account_service.go`, `backend/internal/repository/account_repo.go`
- 前端: `frontend/src/pages/accounts/`, `frontend/src/client/account.ts`

**核心逻辑**:
1. **账户绑定**: 用户提交 MT4/MT5 账户信息（Broker、Login、Password），后端加密存储并通过 MT Gateway 验证连接
2. **账户状态**: 支持 `isDisabled` 禁用状态，禁用后不可交易但可查看历史
3. **权限控制**: 通过 `interceptor` 的 Scope 机制控制 API 访问权限（`ScopeMarketRead`, `ScopeTrading` 等）

### 3.2 AI 辩论与策略生成

**文件位置**:
- 后端: `backend/internal/service/debate_v2_service.go`, `backend/internal/service/debate_v2_advance_job.go`
- 前端: `frontend/src/pages/ai/debate/`

**核心逻辑**:
1. **多模型辩论**: 用户输入策略需求，系统调用多个 LLM（OpenAI、DeepSeek、Claude 等）生成候选策略
2. **代码生成**: 通过 `debate_v2_code_gen_budget.go` 控制生成预算，避免过度消耗 Token
3. **异步 Job**: 辩论过程通过 `Job + SSE` 异步执行，前端订阅 `EventSource` 接收进度与结果
4. **策略保存**: 生成的策略代码保存为模板，支持版本管理与回滚

### 3.3 策略调度与执行

**文件位置**:
- 后端: `backend/internal/service/strategy_schedule_runner.go`, `backend/internal/service/strategy_schedule_service.go`
- 前端: `frontend/src/pages/strategy/StrategySchedulePage.tsx`

**核心逻辑**:
1. **调度配置**: 用户选择策略模板、交易账户、交易品种、时间周期，配置风控参数
2. **定时执行**: 通过 `StrategyScheduleRunner` 的 `loop` 机制，按 Cron 表达式或事件触发策略执行
3. **运行时状态**: 调度状态存储于 PostgreSQL，支持 `pending`/`running`/`paused`/`stopped` 等状态
4. **日志记录**: 每次调度执行生成运行日志，记录信号、订单、盈亏详情

### 3.4 交易执行与风控

**文件位置**:
- 后端: `backend/internal/service/trading_service.go`, `backend/internal/service/trading_risk.go`, `backend/internal/service/execution_gateway.go`
- 前端: `frontend/src/hooks/useTrading.ts`

**核心逻辑**:
1. **订单路由**: 通过 `ExecutionGateway` 统一路由至 MT4/MT5 适配器
2. **风控检查**: 每笔交易前检查：
   - 账户是否禁用
   - 单笔风险敞口是否超限
   - 单日亏损是否触及止损线
   - 保证金是否充足
3. **订单类型**: 支持市价单、限价单、止损单、追踪止损等
4. **成交回报**: 通过 MT Gateway 异步接收成交回报，更新持仓与资金状态

### 3.5 实时数据流

**文件位置**:
- 后端: `backend/internal/connect/stream_subscribe_events.go`, `backend/internal/stream/manager.go`
- 前端: `frontend/src/providers/ConnectProvider.tsx`, `frontend/src/hooks/useRealtimeUpdates.ts`

**核心逻辑**:
1. **SSE 推送**: 服务端通过 `EventSource` 推送订单更新、盈亏变化、账户状态
2. **Connect 流式**: 部分场景使用 ConnectRPC 的 Server-Stream（如行情订阅）
3. **前端状态同步**: 通过 Zustand Store 管理全局状态，实时更新 UI

### 3.6 回测引擎

**文件位置**:
- 后端: `backend/internal/service/tick_backtest_engine.go`, `backend/internal/service/backtest_run_service.go`
- 策略服务: `strategy-service/app/routes/backtest.py`

**核心逻辑**:
1. **Tick 级回测**: 基于历史 Tick 数据模拟成交，支持滑点、手续费、点差配置
2. **成本模型**: 通过 `backtest_cost_model.go` 计算交易成本，影响最终盈亏
3. **结果分析**: 生成夏普比率、最大回撤、胜率等指标，支持可视化展示

## 5. 数据流程设计

### 5.1 用户注册与登录

```
用户 → 前端 Login/Register 页面
  → POST /antrader.AuthService/Login (Connect Unary)
  → Backend: 验证密码 → 生成 JWT → 返回 token
  → 前端: 存储 token 至 localStorage → 跳转 Dashboard
```

### 5.2 账户绑定

```
用户 → 前端 BindAccount 页面
  → POST /antrader.AccountService/CreateAccount
  → Backend: 加密存储账户信息 → 通过 MT Gateway 验证连接
  → 返回 account_id
  → 前端: 刷新账户列表
```

### 5.3 AI 辩论生成策略

```
用户 → 前端 DebatePage 输入策略需求
  → POST /antrader.DebateV2Service/Advance (Connect Unary)
  → Backend: 创建 Job → 返回 job_id
  → 前端: 订阅 SSE /antrader.DebateV2StreamService/SubscribeEvents?job_id=xxx
  → Backend: 多模型辩论 → 生成代码 → 推送 progress/complete/error 事件
  → 前端: 实时显示进度 → 完成后保存模板
```

### 5.4 策略调度执行

```
用户 → 前端 StrategySchedulePage 配置调度
  → POST /antrader.StrategyService/CreateSchedule
  → Backend: 存储调度配置 → 启动 ScheduleRunner
  → ScheduleRunner: 按 Cron 触发 → 调用 Strategy-Service 执行策略
  → Strategy-Service: 在沙箱中运行 Python 策略 → 返回信号
  → Backend: 风控检查 → 通过 ExecutionGateway 下单至 MT Gateway
  → MT Gateway: 执行订单 → 返回成交回报
  → Backend: 更新持仓、记录日志 → 推送 SSE 事件至前端
```

### 5.5 实时行情订阅

```
用户 → 前端 Dashboard/Trading 页面
  → GET /antrader.StreamService/SubscribeEvents (Connect Server-Stream)
  → Backend: 订阅 MT Gateway 行情 → 推送 Quote/OrderUpdate/ProfitUpdate
  → 前端: 通过 Zustand 更新全局状态 → UI 实时刷新
```

## 6. 接口规范

### 6.1 接口形态约定

根据 `docs/接口与数据流架构约定.md`，项目采用严格的接口规范：

| 能力 | 首选技术 | 说明 |
|------|----------|------|
| 浏览器/网关可调用的 RPC | **Connect RPC** | 支持 Unary 与流式，Unary 可走 HTTP/1.1 |
| 服务间高性能调用 | **gRPC** | 与 Connect 共用 Protobuf 契约 |
| 浏览器单向实时推送 | **SSE** | 事件驱动，禁止用 WebSocket |

### 6.2 明确禁止的形态

| 形态 | 结论 |
|------|------|
| **WebSocket** | 禁止新引入，双向需求由 Connect 流式或 Unary + SSE 组合满足 |
| **REST 作为新 API 面** | 禁止，存量 REST 仅允许窄幅缺陷修复 |
| **轮询作为默认模式** | 禁止，不得以定时 GET 拉状态作为实时性主路径 |
| **定时任务作为默认编排** | 禁止，不得以 crontab 式周期性扫库替代事件订阅 |

### 6.3 核心服务列表

后端注册的服务（`backend/internal/transport/connect/register.go`）:

- `AuthService` — 认证与授权
- `AccountService` — 账户管理
- `TradingService` — 交易执行
- `MarketService` — 行情与品种
- `StreamService` — 实时数据流
- `StrategyService` — 策略与调度
- `AIService` — AI 对话
- `DebateV2Service` / `DebateV2StreamService` — 辩论与代码生成
- `AdminUserService` / `AdminAccountService` / `AdminTradingService` — 管理后台
- `AnalyticsService` — 数据分析
- `PythonStrategyService` — Python 策略交互
- `BacktestDatasetService` / `BacktestTradesService` — 回测数据
- `AutoTradingService` — 自动交易
- `SystemAIService` — 系统 AI 配置
- `AgentService` / `JobService` — Agent 与 Job 管理
- `StrategyExperimentService` / `StrategyAssetService` / `MarketRegimeService` — 实验与资产

### 6.4 Protobuf 规范

- **版本**: Protobuf-ES v2（`@bufbuild/protobuf` v2.x）
- **生成工具**: `buf generate` + `protoc-gen-es` v2 + `protoc-gen-connect-go`
- **生成路径**: `frontend/src/gen/**/*.ts`, `backend/gen/proto/**/*.go`
- **禁止**: v1 残留（`extends Message`, `proto3.util` 等）

## 8. 开发规范

### 8.1 代码规范

1. **文件长度**: 任何脚本超过 800 行应考虑拆分（proto 生成文件除外）
2. **类型安全**: 前端优先使用生成类型，减少 `any` / `as any`
3. **错误处理**: Go 使用显式 error 返回，禁止 panic 传播至外部
4. **日志**: 使用 Zap 结构化日志，包含 `trace_id`、`user_id` 等上下文

### 8.2 接口开发流程

1. **先更新文档**: 修改 `docs/接口与数据流架构约定.md` 或专项设计文档
2. **再改 .proto**: 更新 Protobuf 定义
3. **生成代码**: 执行 `make proto` 生成前后端代码
4. **实现逻辑**: 编写 Handler 与 Service
5. **验证**: 执行 `go test ./... -short` 与 `npm run build`

### 8.3 数据库规范

- **迁移脚本**: 使用顺序编号（`001_xxx.up.sql` / `001_xxx.down.sql`）
- **字段命名**: 蛇形命名（`user_id`, `created_at`）
- **时间戳**: 统一使用 `timestamptz`
- **软删除**: 关键表使用 `deleted_at` 而非物理删除

### 8.4 提交规范

- **proto 一致性**: CI 包含 `proto-drift` job，确保 `.proto` 与生成物一致
- **构建验证**: 提交前执行 `go build` 与 `npm run build`
- **容器更新**: 修改后执行 `docker compose up -d --build`

## 9. 潜在技术风险与解决方案

### 9.1 前端类型债务

**风险**: 前端存在约 518 处 `any` / `as any`，集中在策略页面与 client 层，可能导致运行时错误。

**解决方案**:
- 按模块逐步替换为生成 DTO 类型或 `unknown` + 窄化
- 优先处理 `catch (e: any)`、事件回调参数等低风险点
- 每次按单页或单 client 控制 diff，避免大 PR

### 9.2 后端数据库驱动并存

**风险**: `lib/pq` 与 `pgx` 并存，可能导致行为差异（数组类型、事务、COPY）。

**解决方案**:
- 评估将 pq 专用逻辑迁移至 pgx
- 建立测试基线，验证行为等价性后移除 `lib/pq`

### 9.3 MT4/MT5 Protobuf 大文件

**风险**: 生成物超过 800 行且含大量 deprecated 注释，违反代码规范且难以维护。

**解决方案**:
- 从 `.proto` 拆包，减小单个生成文件体积
- 增加适配层隔离，避免业务代码直接依赖生成物

### 9.4 策略服务依赖偏旧

**风险**: FastAPI、Uvicorn、Pydantic、pandas 等版本固定，可能存在安全漏洞或性能问题。

**解决方案**:
- 建立测试基线后分批升级
- 优先升级安全相关依赖（如 `urllib3`, `certifi`）

### 9.5 边缘网关超时

**风险**: Cloudflare 等边缘网关对长连接有超时限制（524 错误），可能导致 SSE 或流式 RPC 中断。

**解决方案**:
- 长耗时工作流采用异步 Job + SSE 推送，首包快速返回
- 参考 `docs/边缘网关与长连接问题处理参考.md` 配置超时与重连策略

### 9.6 单点故障

**风险**: 当前架构无高可用设计，PostgreSQL、Redis、Backend 均为单实例。

**解决方案**:
- PostgreSQL: 主从复制 + 读写分离
- Redis: Sentinel 或 Cluster 模式
- Backend: 多实例部署 + 负载均衡
