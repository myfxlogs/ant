## 一、后端架构缺陷

### 🔴 严重: 分层违规——处理程序绕过服务层直连数据库

`internal/connect/` 中 30+ 个处理程序存在四种互不一致的数据访问模式：

| 模式 | 示例文件 | 问题 |
|------|----------|------|
| **A (正确)** | `account_handler.go` → `service.PlatformService` | 符合分层架构 |
| **B (跨层)** | `analytics_handler.go`, 6 个 `admin_*_handler.go` | 直接调 `repository.AdminRepository`，跳过了 service 层 |
| **C (裸 SQL)** | `auth_handler.go`, `job_handler.go`, `stream_handler.go`, `ai_handler.go` | 处理程序内联 SQL/直接拿 `pgxpool.Pool` |
| **D (混合)** | `market_handler.go` | 同时用 service + 原始 clickhouse.Conn |

**后果**: 无接口抽象→无法单元测试处理程序；业务逻辑散落三层；修改数据库查询要改处理程序代码。

`auth_handler.go:141` 更有效 `panic("crypto/rand failed: " + err.Error())`——会在生产环境直接崩溃服务器。

### 🔴 严重: 双重数据库连接

`cmd/server/main.go:51-62` 对同一个 Postgres 同时维护 `pgxpool.Pool` 和 `sqlx.DB`：
- `pgxpool` → 新代码用
- `sqlx` → "遗留"仓库用，迁移半途而废

两个连接池各自占用内存和连接数。

### 🟡 神对象 + 死代码

| 位置 | 规模 | 问题 |
|------|------|------|
| `internal/repository/admin_repo*.go` | 7 文件 1151 行 | 一个 AdminRepository 管用户/账户/交易/配置/日志/管辖权 6 个领域 |
| `internal/model/models.go` | 456 行单文件 | DB model + API DTO + 查询参数全混在一起 |
| `internal/controlplane/controlplane.go` | 267 行 | KillSwitch/StrategyBreaker/Canary——完全未被任何代码引用 |
| `internal/quantengine/subscriber.go` | 108 行 | 信号订阅者——完全未被引用 |
| `internal/connect/stub_handlers.go` | 80 行 | DebateV2 全方法返回 `CodeUnimplemented` |
| `cmd/mdtest/`, `cmd/mt4trade/`, `cmd/live-test/` | 3 个 cmd | 开发期脚手架残留 |
| `gorm.io/gorm` + `gorm.io/driver/postgres` | go.mod 依赖 | 无任何生产代码使用 |
| `configs/config.yaml` | 完整的 YAML | main.go 只用 os.Getenv，完全忽略它 |

### 🟡 安全: 无速率限制

- 登录/注册端点无速率限制（`pkg/errors/errors.go` 定义了 `RateLimited` 错误码但从未使用）

## 二、前端 UX 缺陷

### 🔴 严重: 路由断裂——用户可点击但 404

| 问题 | 影响 |
|------|------|
| `Marketplace.tsx` 存在但 **App.tsx 中无路由** | 市场页面完全无法导航 |
| Dashboard 链接到 `/trading` 和 `/market` | 两个路由均未定义 |
| Dashboard 链接到 `/accounts` | `/accounts` 路由不存在 |
| 登录页"忘记密码"链接到 `/forgot-password` | 路由不存在 |

### 🔴 严重: 状态管理三分天下

同一应用内存在三种互不兼容的状态方案：

1. **Zustand** — 6 个 store (auth, account, trading, notification, ai, adminAccount)
2. **手动 useState** — Dashboard, AccountDetail, BindAccount, UserManagement 等各自管理 loading/error
3. **React Query** (`useRpcQuery`) — 仅 JurisdictionGate, TradingMonitor, StrategyAssetPage 使用

账户数据同时在 `accountStore.accounts[]` / `tradingStore.accountInfoMap` / `tradingStore.userSummary` 三处存储。

### 🟡 加载/错误/空状态缺失

| 页面 | 缺失的状态 |
|------|------------|
| `MarketRegimePage` | 无全页 loading，无初始 empty state |
| `StrategyExperimentPage` | 无 StatusResult 包裹 |
| `StrategySchedulePage` | loading/empty 未传给 StatusResult |
| `LoginPage.tsx` | 整个文件是死代码（旧 localStorage 版），应删除 |

### 🟡 依赖冗余

- 三个图标库: `@ant-design/icons` + `@tabler/icons-react` + `lucide-react`
- 两个图表库: `recharts` + `lightweight-charts`
- 前端测试目录 `src/test/` 存在但为空

## 三、工程化缺陷

### 🔴 严重: 配置系统——双系统且都不完整

1. `configs/config.yaml` — 完整 YAML 模板，但 **main.go 完全忽略**
2. `main.go` 的 `env()` — 40+ 处 `os.Getenv` 硬编码默认值

无单一配置来源，无启动时必需项验证（缺失 `JWT_SECRET` 不会启动报错，会在运行时异常）。

### 🔴 严重: Prometheus 被引入但未正确使用

- `go.mod` 有 `prometheus/client_golang v1.23.2`，但 `mdgateway/metrics.go` **手写直方图和 text/plain 格式化**
- 未用 `promhttp.Handler()`，自己在 bytes.Buffer 里拼 HTTP 响应
- 缺少: Go 运行时指标、HTTP 请求统计、DB 连接池指标、NATS 连接指标
- `docs/spec/15-observability.md` 列出的 `/readyz` 端点未实现

### 🟡 CI: 无 CD、无覆盖率、无安全扫描

- **无 CD**: 合并 main 后不触发任何部署/镜像构建/迁移
- **无覆盖率**: `go test` 不带 `-coverprofile`，无阈值
- **无安全扫描**: 没有 gosec/trivy/snyk/dependabot
- **集成测试从不运行**: `//go:build integration` 标签在 CI 中被跳过

### 🟡 错误链断裂

`internal/pkg/errors/errors.go` 的 `AppError` 包装 error 但不用 `%w`，导致 `errors.Is`/`errors.As` 无法穿透。

### 🟡 测试债务

- 966 次 `t.Fatal` vs 215 次 `t.Error`（应多报错而非遇错即停）
- 无 `t.Parallel()`——全部串行
- 无 fuzz test、无 benchmark
- `factor/dsl/alignment_test.go:116` 残留 `t.Skip("将在卡片 M10.5-4 中实施")`，卡片已完成但 skip 未除
