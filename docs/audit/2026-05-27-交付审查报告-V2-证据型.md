### S-4 `platform_service.UpdateAccount` SQL 参数错位

`@/opt/ant/backend/internal/service/platform_service.go:187-196`：
```go
_, err := s.pg.Exec(ctx, `
    UPDATE mt_accounts SET
        broker_company = COALESCE(NULLIF($2, ''), broker_company),
        broker_server  = COALESCE(NULLIF($3, ''), broker_server),
        broker_host    = COALESCE(NULLIF($4, ''), broker_host),
        is_disabled    = COALESCE($5, is_disabled),
        updated_at     = CURRENT_TIMESTAMP
    WHERE id = $1::uuid AND user_id = $2
`, id, userID, brokerCompany, brokerServer, brokerHost, isDisabled)
```
- 函数签名 6 参，SQL 只引用 `$1..$5`
- `$2` 同时出现在 `broker_company = COALESCE(NULLIF($2, ''), ...)` 和 `WHERE user_id = $2`
- 位置 `$2` 的值是 `userID`（UUID 字符串）
- → 调用一次，`broker_company` 字段被**静默覆写为 user_id 的 UUID 字符串**；`isDisabled` 第 6 参完全未使用

任何调过这个 RPC 的用户的 `mt_accounts.broker_company` 都已损毁。

### S-5 三层幂等 Layer 1 实现错误（事实上不起作用）

`@/opt/ant/backend/internal/mthub/idempotency.go:102-117`：
```go
err = g.pg.QueryRow(ctx, "SELECT pg_try_advisory_lock($1, $2)", k1, k2).Scan(&ok)
// ...
unlock = func() {
    _, _ = g.pg.Exec(context.Background(), "SELECT pg_advisory_unlock($1, $2)", k1, k2)
}
```
PG `pg_try_advisory_lock` 是 **session-level 锁**，绑定到当前后端连接。`pgxpool` 在 `QueryRow` 完成后把连接归还连接池。`unlock` 是从池里**另取一个连接**执行 `pg_advisory_unlock` → 返回 false（当前连接没持锁）→ **锁实际泄漏直到原连接被复用或销毁**。

**正确做法**：
- 用 `pg_try_advisory_xact_lock`（事务级，COMMIT 自动释放）；或
- 用 `pool.Acquire()` 拿单条 `*pgx.Conn`，加锁/解锁在同一 conn 完成

附加：`@/opt/ant/backend/cmd/server/main.go:145` 实际 wire 的是老的单层 `IdempotencyGuard`：
```go
idemGuard := mthub.NewIdempotencyGuard(rdb.Client())
```
**新的 `ThreeLayerGuard` 0 处实例化**。所以这个 PG 锁 bug 当前还没触发到生产 —— 但代码已埋雷，下次切换时会立刻造成线上事故。

## 6. vs 同类 OSS（每条带源码证据）

### 6.1 vs NautilusTrader（Rust core + Python）
- ✅ OMS 15-state enum 命名对齐 Nautilus `OrderStatus`（设计层对齐）
- ❌ Nautilus 整个 `ExecutionEngine` 是订单**必经路径**；ant 的 OMS 状态机**0 个 import** = 整条 NEW→VALIDATED→RISK_APPROVED→SUBMITTED 转换链运行时完全跳过
- ❌ Nautilus 回测 `BacktestEngine` 强制注入 `FillModel`+`LatencyModel`；ant 的 `internal/backtest` 包除 fill_model 外**0 处导入**

### 6.2 vs QuantConnect Lean（C#，Apache 2.0）
- 设计层：ant `risksvc/pipeline.go` 6-stage 流水线对标 Lean Alpha→Portfolio→Risk→Execution
- ❌ Lean `Algorithm.Framework` 是订单必经；ant `NewSignalPipeline()` **0 处调用** (`@/opt/ant/backend/internal/risksvc/pipeline.go:81`)
- ❌ Lean 有 `IPortfolioConstructionModel`（Markowitz/Black-Litterman）；ant 没有此抽象层，单策略 → 直接订单，无组合优化

### 6.3 vs freqtrade（Python，GPLv3）
- ✅ Walk-forward + Hyperopt + 多 gate 算法 ant 都写了 `internal/ai/`
- ❌ freqtrade FreqAI dashboard 给散户可见 walk-forward 进度；ant 前端 0 入口（`internal/ai` 死代码 + 前端无对应页面）
- ❌ freqtrade Telegram bot 内置；ant **0 个推送通道**
- ❌ freqtrade 安装到 paper 30 分钟；ant Trading 页占位符 = **不可用**

### 6.4 vs Hummingbot（Python，Apache 2.0）
- ❌ Hummingbot 60+ exchange connector 抽象；ant 仅 mt4/mt5 二选一
- ✅ Funding rate 实现 `@/opt/ant/backend/internal/costsvc/funding_rate.go` 存在（D1 ☑）；但 costsvc 在订单路径中由于 `SetCostEstimator` 0 调用 → 仍不生效

### 6.5 vs vnpy（Python，MIT）
- vnpy `event/engine.py` MessageBus 简化版可借鉴；ant `mthub/trade_event_store.go` 写 NATS Tier-0 但**没有对应的 cache rebuild 路径生产验证**

## 7. 系统性结论

### 7.1 ROADMAP 完成度真相

| ROADMAP 声明 | 源码事实 |
|---|---|
| M10-BASE 38/38 ☑ | **B1-B6 + C1-C6 + D1-D7 + E0-E6 共 26 张卡片所属的 oms/ai/strategy/backtest/risksvc(pipeline) 5 个包，运行时 0 处调用**。代码、测试都存在但不在生产路径——等价于"实验室 ☑、车间 ⚰️" |
| M11 5/5 ☑ | M11-13 execalgo 包 0 import；M11-16 jurisGate `_ = jurisGate` 丢弃；M11-14/15 在 oms 包内部，oms 0 import |

ROADMAP §0.2 卡片协议要求 ☑ 必有 handover log。**M10-BASE 38 张 + M11 5 张 = 43 张 ☑ 卡 0 份 handover**（实测 `ls docs/handover/ | grep -E "M10-BASE|M11-" | wc -l = 0`）。

### 7.2 安全攻击链（5 分钟可演练）

1. 任意账号注册并登录 → 拿 JWT
2. curl `AdminUserService.ListUsers` → 拿所有用户邮箱
3. curl `AdminUserService.UpdateUser({id: <any>, role: "admin"})` → 把自己改成 admin
4. curl `AdminUserService.ResetUserPassword({id: <target>})` → 拿到 target 新密码
5. curl `StrategyService.PublishTemplateDraft({id: <target_template>})` → 把 target 私有策略改公开
6. curl `StrategyService.UpdateSchedule({id: <target_schedule>, accountId: <self_acc>})` → 把别人调度的 account_id 指到自己账户，**自动收别人的信号执行到自己 MT 账户**（如果上层运行调度的话）

**这套攻击链 100% 是源码可推导的，不是猜测**。

### 7.3 金融大脑路径真相

```
用户在 UI 点 "下单"
  ↓
ConnectRPC MtHubServer.PlaceOrder
  ↓
✅ JWT validated (authInterceptor)
✅ user owns account (platformSvc.UserOwnsAccount)
❌ 跳过 capability tier
❌ 跳过 HardLimit (KYC/jurisdiction/margin/Killswitch)
❌ 跳过 platform exposure limits
❌ 跳过 Risk Engine pre-check
❌ 跳过 PositionSizer (Kelly/VolTarget/risk-parity)
❌ 跳过 BlockAllocator
❌ 跳过 CostEstimator
❌ 跳过 OMS state machine (NEW → VALIDATED → RISK_APPROVED → SUBMITTED)
✅ Redis SetNX idempotency (single-layer; Layer 1 PG 设计有 bug 但当前未启用)
✅ ReconcileGate.CanAccept
✅ exec.PlaceOrder (MT4/MT5 adapter)
✅ NATS event publish (Tier-0)
```

签名都对，但**金融语义全部走表层**，与 `freqtrade backtest --short --some-bs-strategy` 出错以后的"无安全网"模式区别仅在于 MT 协议封装。
