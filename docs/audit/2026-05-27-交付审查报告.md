## 二、阻塞性问题（交付前必须修）

### B1 · JWT 令牌存储于 localStorage（高危）

**现状**：
- `authStore.ts:27-28` — `access_token` 和 `refresh_token` 直接写 `localStorage`
- `transport.ts:99` — 每次请求从 `localStorage.getItem('access_token')` 读取
- `debateV2.ts:233` — SSE 的 `access_token` 作为 URL query param 传递
- `auth_handler.go:60` — `access_token` 和 `refresh_token` 是**同一个 24h JWT**
- 无 token rotation，无 httpOnly cookie

**风险**：任何 XSS 均可静默窃取令牌，攻击者获得 24h 完整 API 访问权限。

**修复方案**（~1.5 工日）：
1. 后端拆分两个 JWT：access_token（15min，仅存内存）和 refresh_token（7d，httpOnly Secure SameSite=Strict cookie）
2. 新增 `POST /token/refresh` HTTP endpoint（非 ConnectRPC，需读 cookie）
3. Refresh token 使用 rotation：每次使用后旧 token 加入 Redis 黑名单（TTL=7d）
4. 前端 `authStore` 不再 persist token 到 localStorage，仅保留 user 信息
5. SSE 认证改为：后台先发一个 short-lived SSE token（60s），前端用该 token 连接 SSE

### B2 · AI 策略生成后端缺失（阻塞核心功能）

**现状**：
- 前端 `debateV2.ts`（348 行）、`useDebateFlow.ts`（480 行）、`DebatePageV2.tsx` — **完整实现**
- `connect.ts:80` — `debateV2Client` 和 `debateV2StreamClient` 已创建
- Proto 定义完整：16 个 RPC（StartDebateV2 / ChatDebateV2 / AdvanceDebateV2 / StartAdvanceJob ...）
- Go connect 代码已生成：`debate_v2_service.connect.go`（527 行）、`debate_v2_stream.connect.go`（109 行）
- **但是**：`main.go:317` 注释 `// Stub: DebateV2, DebateV2Stream` — **无 handler，无 service，无 PG 表**
- 前端调用所有 DebateV2 RPC 都会收到 `CodeUnimplemented`

**影响**："自然语言通过 AI 生成量化策略"是产品的核心差异化功能，当前完全不可用。

**修复方案**（~4 工日）：

2. 创建 `backend/internal/service/debate_v2_service.go` — 会话 CRUD + 步骤状态机 + LLM 调用
3. 创建 `backend/internal/connect/ai/debate_v2_handler.go` — 16 个 RPC handler
4. 创建 SSE endpoint：`/sse/debate-v2/{advance-jobs,chat-jobs}/{jobId}/stream`
5. Wiring 进 `main.go`

## 五、MT 密码存储评估（已纠正）

**原评估**："MT 密码明文存储 → 致命"

**纠正**：MT 交易密码提交给 mtapi gRPC 服务器时**必须是明文**（MetaTrader 协议要求）。因此：
- 这不是"不应存储密码"的问题，而是**必须可逆加密**的问题
- 攻击者获取 DB 后能否解密取决于主密钥是否也被窃取
- **正确做法**：用 `secrets.Client`（AES-256-GCM）加密后落库，主密钥通过环境变量注入，不存 DB
- **当前状态**：未接入 secrets.Client，密码以明文形式存储在 `mt_accounts.password` 列

**定级**：从"致命"下调为"中优先级加固项"。在 `ANT_MASTER_KEY` 已正确部署的前提下，风险可控。
