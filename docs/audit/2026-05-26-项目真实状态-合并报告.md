## 1. 致命问题（直接影响产品能否工作）

### 1.1 🔴 MT5 核心交易 RPC 全部未实现

```@/opt/ant/backend/internal/mdgateway/adapter/mt5/gateway.go:557-565
return 0, fmt.Errorf("mt5: PlaceOrder not yet implemented")
return fmt.Errorf("mt5: CloseOrder not yet implemented")
return fmt.Errorf("mt5: ModifyOrder not yet implemented")
return nil, fmt.Errorf("mt5: FetchSymbolParams not yet implemented")
```

MT4 `GetOrderHistory` 同样 stub：`@/opt/ant/backend/internal/mdgateway/adapter/mt4/gateway.go:568`

**实际效果**：上层 15-state OMS、PositionSizer、capability 检查、cost estimator、Kelly/VolTarget sizer、Block allocator、Kill Switch 全部跑不到 broker。MT5 用户只能看 K 线，不能下单。

**根因**：`make detect-stubs` 正则盲区：

```bash
# 现行（漏 Errorf 模式）：
grep -E '(Printf|Println|Print)\("[^"]*\b(stub|TODO|placeholder|not (wired|connected|implemented))\b'
# 应改为：
grep -E '(Printf|Println|Print|Errorf|errors\.New)\("[^"]*\b(not (yet )?implemented|TODO|stub|placeholder)\b'
```

→ 整套"卡片关闭判据"在这里出现盲区，所有 ☑ 都需要重新审计。

| 包 / 文件 | 行数 | 被 import | 备注 |

| `internal/tenant/` | ~150 | **0** | 上轮废弃 tenant 模型时漏删，已被 `usermgr` 替代 |
| `internal/dataquality/` | 6 文件 | **0** | M10-BASE-F1~F6 在 mdgateway 内重做了 |
| `internal/marketstate/` | 6 文件 | **0** | 同上 |

### 1.4 🔴 生产路径含 `panic`

```@/opt/ant/backend/internal/connect/auth_handler.go:141
panic("crypto/rand failed: " + err.Error())
```

生产环境 `crypto/rand` 罕见但可能短暂失败（容器熵不足）；直接 panic 会让整个 RPC server 退出。应改为返回 `AppError(Internal)` 让 RPC 层正常返回 500。

## 2. 严重违约（spec 写了，代码不遵守）

### 2.1 🟠 mdgateway 总行数超 spec 限制 4-6 倍

`docs/spec/11-mdgateway.md:4` 写 "非测试文件 ≤ 800 行"，实际：

| 文件 | spec 限 | 实际 |
|---|---|---|
| `mdgateway/` 全包（非测试） | 800 | **5266** |
| `mdgateway/` 全包（含测试） | 1500 | **7288** |
| `runner.go` | 200 | 585 |
| `adapter/mt5/gateway.go` | — | 632 |
| `adapter/mt4/gateway.go` | — | 578 |
| `clickhouse_writer.go` | 180 | 275 |

`spec/11 §16` 的 LOC 验收命令从未跑过；跑一次会立即 fail。要么改 spec 放宽，要么拆 adapter 为子包。

### 2.2 🟠 service / connect 分层失控

```
backend/internal/service/   7 文件 / 1494 行
backend/internal/connect/  30 文件 / 5267 行
```

`connect/` 中存在 4 种互不一致的数据访问模式：

| 模式 | 例 | 问题 |
|---|---|---|
| A（正确） | `account_handler.go → service.PlatformService` | 符合分层 |
| B（跨层） | `analytics_handler.go`、6 个 `admin_*_handler.go` | 直调 `repository.AdminRepository`，跳过 service |
| C（裸 SQL） | `auth_handler.go`、`job_handler.go`、`stream_handler.go`、`ai_handler.go` | handler 内联 SQL/拿 `pgxpool.Pool` |
| D（混合） | `market_handler.go` | 同时用 service + 原始 `clickhouse.Conn` |

**后果**：无接口抽象 → 无法单元测试 handler；业务逻辑散落三层；改 SQL 要改 handler。

### 2.3 🟠 双数据库连接池未完成迁移

```@/opt/ant/backend/cmd/server/main.go:51-62
// pgxpool（新代码）
pool, err := pgxpool.New(ctx, dsn)
// sqlx connection (for repos that predate pgxpool)
sqlxDB, err := sqlx.Open("pgx", dsn)
```

同一 PG 实例双连接池、双连接数、双内存。`gorm.io/gorm` 还在 go.mod 但只剩 1 个 shell 脚本用 = 三套 ORM 并存。

### 2.4 🟠 Prometheus 被引入但手写文本格式

```@/opt/ant/backend/internal/mdgateway/metrics.go
// 0 处 promhttp / prometheus.Register / MustRegister
// 24 处 fmt.Fprintf / bytes.Buffer 自拼 text/plain 响应
```

`go.mod` 含 `prometheus/client_golang v1.23.2` 但全程未用 collector。缺失：
- Go 运行时指标（GC、goroutine、heap）
- HTTP 请求 latency/error 统计
- pgxpool / clickhouse / NATS 连接池指标
- spec/15 列出的 `/readyz` 端点

### 2.5 🟠 配置系统二元割裂

```
configs/config.yaml     完整 YAML 模板 (~150 行)
cmd/server/main.go      40+ 处 os.Getenv 硬编码默认值
```

- 无单一配置来源
- 缺失 `JWT_SECRET` **启动不报错**，运行时才异常
- 团队改配置时不知道改哪个文件

### 2.6 🟠 错误链断裂（errors.Is/As 失效）

```@/opt/ant/backend/internal/pkg/errors/errors.go:151
type AppError struct { ... }
// 全文件 0 处 fmt.Errorf("...%w...", err)
```

`AppError` 包装 error 但不用 `%w`，导致 `errors.Is(err, ErrXxx)` 和 `errors.As(err, &target)` 全部失效。下游 retry / circuit breaker / 错误分类逻辑全部受影响。

## 4. UX / 产品体验问题

### 4.1 🔴 路由断裂（见 §1.3）

### 4.2 🟠 前端状态管理三分天下

同一应用并存三种互不兼容方案：

1. **Zustand** — 6 个 store（auth, account, trading, notification, ai, adminAccount）

**账户数据同时存于 3 处**：`accountStore.accounts[]` / `tradingStore.accountInfoMap` / `tradingStore.userSummary`。


但 spec/31 §1.3 定的是 `user_risk_profiles.capability_tier` 0/1/2/3 体系。**两套权限模型不联动**，且 single-role 字段不支持现实多角色（一个客服可能同时是 audit）。

- 2 个图表库：`recharts` + `lightweight-charts`

### 4.8 🟡 PageWrapper 套娃式包装

54 个页面统一 `<PageWrapper>` = 同一种 Suspense fallback。但 AI debate / 行情看板 / 风控页加载特征完全不同，应允许传 `fallback`/`errorFallback`。
