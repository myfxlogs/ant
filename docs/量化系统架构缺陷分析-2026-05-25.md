# 量化系统架构缺陷分析

> 审计日期：2026-05-25
> 审计范围：`backend/internal/` 全部 Go 源码 + `docs/architecture/` 架构文档
> 方法论：以量化金融专业视角，对比文档设计与代码实现，识别 gap 与缺失

---

## 一、风险引擎不够深

### 现状

`risksvc/` 仅 6 条同步规则（`precheck.go` + `rules.go`）：

| 规则 | 类型 | 说明 |
|------|------|------|
| `MaxPosition` | 阈值 | 单品种持仓数限制 |
| `DailyLoss` | 阈值 | 当日亏损上限 |
| `Drawdown` | 阈值 | 回撤百分比上限 |
| `Session` | 时间 | 周末禁止交易 |
| `Margin` | 阈值 | 最低保证金水平 |
| `CanonicalAuth` | 白名单 | 品种交易权限 |

全部是简单阈值判断，`DefaultRiskLimits()` 对所有用户返回相同默认值。

### 缺失

**1.1 无相关性/净敞口风控**

外汇交易的核心风险是**方向性净敞口**，不是简单的手数计数。用户同时持有：
- 1 手 EURUSD 多头
- 1 手 GBPUSD 多头
- 0.5 手 AUDUSD 多头

三笔全是美元空头敞口。当前 `MaxPosition` 规则只看单一品种手数，对组合的方向暴露完全不可见。需要一个 **NetExposure** 规则，以账户币种为基准，聚合所有持仓的 delta 敞口。

**1.2 无 VaR / CVaR**

没有在险价值计算。无法回答"下一个交易日最坏亏多少"这个风控最基本的问题。建议引入历史模拟法 VaR（基于过去 90 天日频 bar），异步计算、同步检查。

**1.3 无 Kill Switch**

没有一键清仓能力。以下场景需要：
- AI 生成的策略跑飞，连续发出错误信号
- broker 返回异常价格，订单在错误价位执行
- 检测到市场操纵行为的自动防御

Kill Switch 应该是：取消所有挂单 + 市价平掉所有持仓（或至少停止所有自动交易）。

**1.4 无波动率自适应仓位**

`DefaultRiskLimits` 对 BTCUSD 和 EURUSD 使用同一个 `MaxExposurePerAccount=100,000`。但 BTCUSD 日均波动率是 EURUSD 的 5-10 倍。同样名义金额的实际风险完全不对等。应该用 ATR 或历史波动率归一化仓位。

## 二、交易成本模型缺失

### 现状

`model/models.go` 有 `Commission`、`Swap`、`Fee` 字段，`trade_record_repository.go` 做 `UPSERT` 写入这些字段。但**没有任何代码自动计算这些值**。`log_service.go:UpdateOrderHistoryClose` 只是被动接收调用方传入的 swap/commission 参数。

### 为什么是 P0

这直接违反架构文档 **准则 6（回测即实盘）**：

> 回测和实盘必须走完全相同的 factor → signal → execution 代码路径。

没有成本模型 → 回测 P&L 虚高 → 用户用回测结果决定上实盘 → 必然亏损。

MT4/MT5 broker 的成本结构：

| 成本类型 | 描述 | 计算方式 |
|----------|------|----------|
| **Commission** | 每手固定费用 | `volume_lots × commission_per_lot × 2（开+平）` |
| **Swap** | 隔夜利息 | `volume_lots × swap_rate × 持仓天数`（周三三倍） |
| **Spread cost** | bid-ask 价差 | `half_spread × volume`（入场+出场各一次） |

### 建议

建独立的 `costsvc/`（交易成本服务）：

```
costsvc/
  ├── spread_model.go      # 从 tick 流实时计算点差
  ├── commission.go        # 按 broker × symbol 查费率表
  └── swap_calculator.go   # 周三 triple swap
```

所有下单路径（实盘 + 回测）必须经 `costsvc.Estimate(order)` → 写入订单成本 → 累积到 P&L。回测结果必须区分 **Gross P&L** 和 **Net P&L**。

## 三、回测执行引擎精度不足

### 现状

回测是 bar 级别 walk-forward（K 线逐根回放）。`paper_repo.go` 的 `PaperOrder` 只有 `SlippageBps` 一个质量字段。`PaperAccount` 初始化后 `CurrentBalance`/`Equity` 没有看到随订单成交更新的逻辑。

### 缺失

**3.1 滑点模型过于简单**

滑点 ≠ 单一数值。应拆分为三类：

| 类型 | 来源 | 可优化方向 |
|------|------|-----------|
| **Spread cost** | bid-ask spread 半宽 × 交易量 | 选择流动性更好的时段/品种 |
| **Slippage** | 信号到执行间的价格移动 | 降低延迟、优化订单路由 |
| **Market impact** | 大单推高/压低价格 | 拆分订单、TWAP 算法 |

三类来源不同，归因不同，可优化方向不同。当前一个 `SlippageBps` 掩盖了所有细节。

**3.2 无 tick 级回放**

对于基于 tick 的策略（高频/剥头皮），bar 级回测与实际执行偏差极大。bar 内无法确定信号触发和订单执行的确切时序。应引入 `TickReplaySource`（除了现有的 bar 级回放）。

**3.3 加密货币永续合约缺资金费率**

永续合约的 funding rate 是持有成本的核心组成部分（每 8 小时结算一次）。不建模 funding rate 对 BTC/ETH 永续的策略回测完全不可靠。

**3.4 `cost_model_snapshot` 未实际使用**

`backtest_run_repository.go` 定义了 `CostModelSnapshot []byte` 字段，表结构也已就绪，但未发现实际填充的数据流。回测运行没有记录使用了什么成本参数，审计回测结果时无法追溯。

## 四、市场数据质量问题

### 现状

`quality.go` 做：
- bid > ask 丢弃 ✓
- 非正价格丢弃 ✓
- 5σ MAD 异常检测（只标记不丢弃）✓
- gap 检测（metric only）✓
- 时钟偏移检测（metric only）✓

### 缺失

**4.1 无突发/Quote Stuffing 检测**

broker 突然发出 1000 tick/秒（正常 50/秒），可能是：
- Quote stuffing（故意灌入大量报价干扰策略）
- broker 端故障（重复发送）

应检测 tick rate 异常突增并触发告警或暂停该 symbol 的 order 生成。

**4.2 无点差扩大告警**

EURUSD 正常点差 0.2 pip → 突然拉到 5 pips。可能是：
- 重大新闻事件（非农发布）→ 策略应暂停或扩大止损
- broker 操作异常 → 暂停交易

当前没有任何点差监控。策略在非正常点差下执行会产生隐性大额成本。

**4.3 Bar 聚合时间戳选择有已知偏差**

文档 `03-data-flow.md §1.2` 明确写了：

> BarAggregator 用 ArrivedUnixMs 而非 broker TsUnixMs

这是有意识的设计选择，但场景如下：
- broker 发送的 tick `TsUnixMs=09:00:00.500`
- 网络延迟导致 `ArrivedUnixMs=09:00:01.200`
- bar 周期边界是 `09:00:01.000`

用 ArrivedUnixMs → tick 归入 09:00:01 的 bar，OHLCV 中包含了本属于上一秒的 tick → bar 失真。应用 broker 时间戳 + 可配置的最大延迟缓冲窗口。

**4.4 无 session 时段校验**

broker 可能在闭市期间发送过时 tick。`Quality.Check` 缺少 session 校验——应该拒绝 `TsUnixMs` 落在已知非交易时段的 tick。

## 五、AI 策略代码生成的特殊风险

这里是 ant 的核心差异化能力（自然语言 → 策略代码），也引入了传统量化平台没有的风险类别。

### 现状

`ai/` 模块有 `code_compliance.go`（13 条合规规则扫描），但主要关注代码合法性和安全性。对于量化策略特有的风险，防护不足。

### 缺失

**5.1 无前视偏差检测**

AI 在生成 DSL 代码时可能无意识地使用未来数据：
- `close[t+1]` — 显式未来引用
- `ref(close, -1)` — 通过负偏移访问未来
- `high[t] > high[t-1] AND high[t] > high[t+1]` — 用未来 bar 判断当前是否为高点

DSL validator 应扫描 AST，检测任何对 `t+δ (δ>0)` 的引用。

**5.2 无过拟合检测**

AI 可能生成参数过多的策略。规则：参数数量 ≤ 回测数据点数 / 10。超过阈值应标记警告——这是最基本的防过拟合规则（AIC/BIC 更精确但需要更复杂的似然计算）。

**5.3 无策略相关性矩阵**

用户运行 20 个 AI 策略。如果其中 15 个高度相关（信号方向 > 70% 重叠），从组合角度看这 ≈ 1 个策略 × 15 倍杠杆，风险完全重叠但分散化效果为零。应计算策略间信号方向一致率，高相关策略组发出警告。

**5.4 最小质量门槛太低**

当前 `PromoteToLive` 条件（架构文档所述）：
- Sharpe > 0 — 门槛太低，Sharpe 0.1 的正期望值在实际成本下面很容易变负
- max_drawdown < risk_limit — 没有绝对值要求

建议：
- Sharpe > 0.5（年化）
- Max Drawdown < 30%
- 仿真交易 ≥ 7 天且 Net P&L > 0
- 与现有实盘策略相关性 < 0.7
- 满足以上全部条件 → 才允许升级到实盘

## 六、持仓与损益管理

### 缺失

**6.1 Swap 自动计算**

MT4/MT5 broker 的 swap 费率可以从 `Symbols` 接口获取（swap_long, swap_short 字段）。当前代码完全没有做这件事，swap/commission 都是靠外部传入。应该：
1. 从 broker symbol 数据读取 swap 费率
2. 每日结算时自动计算并写入
3. 周三自动 triple（外汇行业标准）

**6.2 无成本基础追踪（FIFO/LIFO/HIFO）**

对于税务报告和精确的 P&L 归因，需要知道平掉的仓位对应哪一笔开仓。当前没有成本基础追踪。这对零售用户不是最高优先级，但对专业用户是硬需求。

**6.3 P&L 归因不可分解**

你无法回答：
- 策略这个月赚钱是信号好还是执行好？
- 亏钱是策略信号反转了还是交易成本上升了？

P&L 应可分解为：
```
Net P&L = Gross P&L（信号质量） - Slippage（执行质量） - Commission - Swap（持有成本）
```

每个维度有独立的 metric 和时间序列。

## 七、架构层面优化建议

### 7.1 风险引擎分层架构

```
Layer 1 (同步, <1ms):  PreCheck
  ├─ 当前 4 项（仓位/日亏/回撤/保证金）
  ├─ Kill Switch 检查
  └─ Session 校验

Layer 2 (同步, <5ms):  NetExposure
  ├─ 跨账户净敞口聚合
  ├─ 按币种拆分 delta 敞口
  └─ 相关性检查（同一方向的高度相关品种）

Layer 3 (异步, 30s):   Portfolio Risk
  ├─ 历史模拟法 VaR (90 天窗口)
  ├─ 压力测试（预设场景：2008 式暴跌、FOMC 秒杀）
  └─ CVaR（尾部风险）

Layer 4 (异步, 事件驱动):  Anomaly Detection
  ├─ 点差异常 → 暂停该品种
  ├─ Quote stuffing → 告警
  └─ 策略漂移（behavior 偏离历史模式）
```

### 7.2 成本服务独立

```
costsvc/
  ├── spread_model.go       # 从 tick 流实时计算 spread
  ├── commission.go         # broker × symbol 费率表
  ├── swap_calculator.go    # 隔夜利息自动计算
  └── funding_rate.go       # 永续合约资金费率

所有下单路径：因子 → 信号 → 成本估算 → 风控 → 执行
```

### 7.3 回测 FillModel 拆分

```
FillResult {
    GrossPrice       float64  # 信号触发时的市场价格
    SpreadCost       float64  # half_spread × volume × 2
    SlippageBps      float64  # f(price_change_during_latency_ms)
    MarketImpactBps  float64  # f(volume / avg_daily_volume)
    NetFillPrice     float64  # GrossPrice + sum(costs)
}
```

### 7.4 AI 策略多级门控

```
Compliance Gate（代码合法）
  → Overfit Gate（参数/数据点 < 1/10）
    → LookAhead Gate（无未来数据引用）
      → Performance Gate（Sharpe>0.5, DD<30%）
        → Paper Gate（仿真 7 天, Net P&L>0）
          → Correlation Gate（与已有实盘策略相关性 < 0.7）
            → LIVE
```

## 八、优先级矩阵

| 优先级 | 项 | 影响范围 | 原因 |
|--------|-----|----------|------|
| **P0** | 交易成本自动计算 + Gross/Net P&L 区分 | 回测可信度 | 直接违反准则6，回测结果不可信 |
| **P0** | AI 策略质量门控 | 用户资金安全 | AI 生成是核心差异化，不加门控则输出不可控 |
| **P1** | 相关性 / 净敞口风控 | 组合风险 | 多品种持仓不做 netting，风控形同虚设 |
| **P1** | Bar 聚合改用 broker 时间戳 + 延迟缓冲 | 数据质量 | 文档已知偏差，bar OHLCV 失真 |
| **P1** | 点差异常监控 + 突发检测 | 执行质量 | 异常行情下策略误执行 |
| **P2** | Tick 级回放 | 策略精度 | 低频为主时优先级低于 bar 级成本模型 |
| **P2** | VaR / 压力测试 | 机构级能力 | 当前用户群以零售为主 |
| **P2** | Swap 自动结算 | 持仓管理 | 手工传入可工作，自动化降低运维成本 |
| **P3** | FIFO/LIFO 成本基础 | 税务 | 零售用户当前不会驱动需求 |
| **P3** | P&L 归因分解 | 策略优化 | 高级功能，先做完成本模型再说 |

## 九、架构设计值得肯定的地方

作为平衡视角，以下设计在量化系统中是成熟的、正确的：

- **7 层架构** — 职责划分清晰，无环依赖，与 alpaca/alfq 一脉相承
- **单一数据路径** — 所有业务读 CH，禁止直调 broker client，杜绝数据一致性问题
- **canonical 在最早位置** — Normalizer 在 adapter 出口、gateway 入口完成，下游零负担
- **CircuitBreaker + SpillWriter + Dedup** — 三层防护覆盖 broker/CH/重复三类故障
- **回测/实盘同一代码路径** — Source interface 抽象，消除代码路径分叉（当前最大违反是成本模型缺失）
- **订单状态机 + 对账恢复** — 10-state 状态机 + 30s 轮询对账 + 启动全量对账，生产级可靠性
- **Bar 修订级联** — version 递增、INSERT 非 UPDATE、信号级联重算，这是生产系统中罕见但正确的设计
- **OTel 全链路追踪** — 信号→执行延迟分阶段归因，235ms SLO 有具体预算分配
- **数据二分法** — 平台共享 vs 用户私有，为 M11 marketplace 预留了正确的基础
- **`cost_model_snapshot` 字段预留** — 证明设计阶段已意识到成本模型的重要性
