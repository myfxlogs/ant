## 二、致命级（P0：一定亏钱 / 监管风险）

### A. 风控是仓位计数器，不是真风控

**现状**（`backend/internal/risksvc/precheck.go:31-79` + `rules.go`）：6 条规则全是简单阈值
- `MaxPosition` / `DailyLoss` / `Drawdown` / `Session` / `Margin` / `CanonicalAuth`
- `DefaultRiskLimits()` 对所有用户返回相同默认值
- `MaxExposurePerAccount: 100000, // 1 standard lot` 注释错——standard lot 是 100k **base currency**，EURUSD 是 100k EUR ≈ 108k USD

**缺失**：

| 维度 | 后果 |
|---|---|
| **跨标的相关性集中度** | long EURUSD + long GBPUSD + long AUDUSD = 3× USD 空头敞口，仓位计数=3 看似分散，实际是单一方向集中 |
| **NetExposure（账户币种基准聚合 delta）** | 多空对冲后真敞口接近 0 但保证金仍占用，资金效率差 |
| **VaR / CVaR / 历史情景压力测试** | 1987 / 2015 SNB / 2020 covid 重放下尾部损失，目前 0 计算 |
| **波动率自适应仓位** | BTCUSD 和 EURUSD 用同一 `MaxExposurePerAccount`，但 BTC 日波动率是 EUR 的 5-10 倍——同名义金额风险完全不对等 |
| **drawdown-based scaling** | 净值跌 5%/10%/20% 自动缩 lot，无此机制下连亏路径就是死亡螺旋 |
| **流动性风险（仓位/ADV）** | 散户单笔无碍，SaaS 聚合 1000 用户后整体可能撞 broker 限额 |
| **Greeks**（即使 CFD/FX option） | delta/gamma/vega 暴露 |

**修法（范式转换）**：

```
当前：Signal → PreCheck(allow/deny) → broker submit
              └─ deny 直接丢 signal = 浪费 alpha

改为：Signal → HardLimit(KYC/jurisdiction/margin floor) [硬阻断]
            → PositionSizer(vol-target / Kelly-fraction / risk-parity) [软调节]
            → broker submit

软限通过 sizer 渐进逼近 lot，不直接 deny。
HardLimit 只挡 4-5 条不可妥协的硬规则。
```

这是 BlackRock Aladdin / Goldman SecDB 的标准做法。**Sizer 把"是否下单"问题转成"下多少手"问题**——这是架构层的范式转换，不是补丁。

**卡片**：M11-1 (P0, 4d)、M11-4 (P1, 3d，跨账户聚合)、M11-9 (P2, 3d，VaR)

### B. OMS 状态机缺幽灵订单恢复路径

**现状**（`backend/internal/oms/statemachine.go:62-110`）：

转换图缺：
- **SLIPPAGE_EXCEEDED**：broker 给的价偏离请求价超容差
- **REQUOTED**：mtapi 实际会触发 requote，当前直接当 REJECTED → 丢失合法重报机会
- **MARGIN_CALL**：margin call 时 OMS 应禁发新单
- **UNKNOWN/RECONCILING**：**最危险的"幽灵订单"**——SUBMITTED 后 broker 30s 无响应（连接丢失/超时），订单可能已成可能未成。当前没有 timeout transition，**重连后用户能看到双倍仓位**
- spec/22 文档里写了 `MODIFY_PENDING/CANCEL_PENDING`，**但代码实际没实现**

**修法**：

```go
type OrderState int
const (
    StateNew OrderState = iota
    StateValidated
    StateRiskApproved
    StateSubmitted
    StateWorking
    StatePartiallyFilled
    StateFilled
    StateCancelled
    StateRejected
    StateExpired
    StateFailed
    // 新增 ↓
    StateRequoted        // broker 报新价，等待重确认（10s 超时则 Rejected）
    StateSlippageRejected// 滑点超容差被拒（与 Rejected 区分用于审计）
    StateUnknown         // SUBMITTED 后 30s 无响应（最危险）
    StateReconciling     // 启动/重连后正在对账
    StateMarginCall      // 账户处于 margin call，禁发新单
)

// reconcile-before-accept gate:
// OMS 重启或检测到连接断开时，所有账户进入 StateReconciling
// 直到 mthub.Reconciliation 完成（broker 持仓全量拉回 + 对账）
// 期间 PlaceOrder 直接拒绝
```

**卡片**：M11-2 (P0, 2d)

### C. 幂等性方案在 failover 时不可靠

**现状**（`backend/internal/mthub/idempotency.go`）：
- Redis SETNX + 24h TTL
- 三个真问题：
  1. **TTL 不够**：周五 22:00 关市 → 周一 05:00 开市 = 55h，跨过 TTL；用户重启重发命中已过期 key → 重复下单
  2. **failover race**：Redis 主从异步复制，主切从期间 SETNX 不可靠
  3. **未利用 broker 端幂等**：MT5 `OrderSendAsync` 的 `magic` 字段可做 broker-side 去重，**完全没用**

**修法**：双层幂等

```
Layer 1 (强一致)：Postgres advisory lock
  - pg_try_advisory_lock(hash(account_id, client_id))
  - 跨 broker 端幂等 + 故障转移安全

Layer 2 (热缓存)：Redis SETNX + 96h TTL（覆盖 G7 假期）
  - 仅作快速短路，避免每次打 PG

Layer 3 (broker 端)：OrderSendAsync magic = hash(client_id) 截 32 位
  - broker 自身去重，最终保险
```

**卡片**：M11-3 (P0, 1d)

### D. 交易成本模型完全缺失（采纳 D2 §二）

**现状**：
- `model/models.go` 有 `Commission` / `Swap` / `Fee` 字段
- `trade_record_repository.go` 做 `UPSERT` 写入
- **但没有任何代码自动计算这些值**——`UpdateOrderHistoryClose` 只被动接收调用方传入的参数

**为什么是 P0**：直接违反架构准则 6（回测即实盘）：
- 回测 commission/swap = 0 → P&L 虚高
- 用户用回测结果决定上实盘 → 必然亏损（高频策略尤其严重）

MT4/MT5 broker 的成本结构：

| 成本类型 | 计算方式 |
|---|---|
| **Commission** | `volume_lots × commission_per_lot × 2`（开+平） |
| **Swap** | `volume_lots × swap_rate × 持仓天数`（**周三三倍**——FX 行业标准） |
| **Spread cost** | `half_spread × volume`（入场+出场各一次） |
| **Funding rate** | 永续合约每 8 小时结算（不建模 → BTC/ETH 永续策略完全不可靠） |

**修法**：建独立 `costsvc/`

```
backend/internal/costsvc/
  ├── spread_model.go      # 从 tick 流实时计算 spread
  ├── commission.go        # broker × symbol 费率表（从 SymbolParams 读取）
  ├── swap_calculator.go   # 隔夜利息自动计算（周三 triple）
  ├── funding_rate.go      # 永续合约资金费率（每 8h）
  └── estimator.go         # Estimate(order) → CostBreakdown
```

所有下单路径强制经过：

```
Signal → CostEstimate → RiskCheck → Sizer → Submit
                ↓
        写入 order.cost_breakdown (PG)
                ↓
        累积到 P&L 时区分 Gross / Net
```

回测引擎必须区分：
```
Gross P&L  = signal-driven returns
Net P&L    = Gross - SpreadCost - Commission - Swap - Slippage
```

**卡片**：M11-5 (P0, 3d)

## 三、严重级（P1：架构性偏差）

### F. "数据质量"在错误的地方做检查

**现状**（`mdgateway/quality.go`）：
- bid > ask 丢弃 ✓
- 非正价格丢弃 ✓
- 5σ MAD 异常检测（只标记不丢弃）✓
- gap 检测（metric only）✓

**缺失**（D1+D2 合并）：
- **stale quote**：tick 10s 没更新但 broker 不主动推 → 策略以为价格还是旧价
- **NFP/CPI 时点差宽化**：平时 0.1 pip 的 EURUSD 数据公布瞬间到 50 pip
- **swap rollover window**（broker 服务器 17:00 EST）
- **三角套利一致性**：EURUSD × USDJPY ≈ EURJPY，偏离 5 pip 必有问题
- **broker holiday calendar**：圣诞节/中国春节/伊斯兰假期，每家 broker 不同
- **Quote stuffing 检测**：tick rate 突然 1000/秒（正常 50/秒）→ 暂停该 symbol
- **session 时段校验**：拒绝 `TsUnixMs` 落在已知非交易时段的 tick

**修法（架构升级，不是补丁）**：把 `mdgateway` 从"管道"升级为"市场状态机"：

```go
type MarketState struct {
    LastQuote       Quote
    QuoteAgeMs      int64
    IsTradeable     bool      // 综合所有 quality 检查的最终判断
    SpreadZscore    float64   // > 3σ 自动暂停
    GapMarker       bool      // 周一开盘 gap
    SwapWindow      bool      // broker 17:00 EST 翻 3 倍前后
    SessionPhase    string    // pre-market / regular / after / closed
    HolidayMarker   bool
    TickRateZscore  float64   // > 4σ = quote stuffing
    TriangulationDelta float64 // 三角套利偏离 pip
}
```

**策略消费 `MarketState`，不再消费 raw tick**。Quality 从"挡脏数据"升级为"声明可交易性"。

**卡片**：M11-6 (P1, 4d)

### G. 回测 / 仿真 / 实盘"完全相同代码路径"是危险的过度抽象

**现状**：`docs/spec/21-backtest-replay.md §1`：「**完全相同**的 factor → signal → execution 代码路径」

这是软件工程师的洁癖；**金融上是过拟合温床**：

| 实盘有 | 回测如果"完全相同"会怎样 |
|---|---|
| Partial fill | 假设全成交 → 高估容量 |
| Slippage（实盘 0.5 pip） | =0 滑点 → 高估收益 |
| Commission + swap | 默认 0 → 高频策略表面盈利实际亏 |
| Look-ahead bias | bar.Close 在收盘瞬间不可用，但回测能看到 → alpha 全是假的 |
| Survivorship bias | 跑路的 broker 数据没了 → 现存 broker 数据高估存活策略 |

**纠正 D2 偏差**：D2 §3.1 把滑点拆 spread / slippage / market impact 三类是对的，但仍只是 **"添加更多模型字段"**，没解决"回测和实盘共用代码"的根本问题。

**修法**：回测引擎应**故意比实盘更悲观**：

```go
// FillModel 拆分（采纳 D2 §7.3）
type FillResult struct {
    GrossPrice       decimal // 信号触发时市场价
    SpreadCost       decimal // half_spread × volume × 2
    SlippageBps      decimal // f(price_change_during_latency_ms)
    MarketImpactBps  decimal // f(volume / avg_daily_volume) (Almgren-Chriss)
    Commission       decimal // 强制 > 0，即使用户没填
    Swap             decimal // 周三 triple
    NetFillPrice     decimal // GrossPrice + sum(costs)
}

// 回测路径强制注入：
//   1. commission/swap/spread 强制非零（即使 broker 配置缺失也用保守默认）
//   2. fill probability < 1.0 在 limit order 上
//   3. bar 消费严格 now ≥ bar.close_ts + finality_delay (已有 LoadFinalizedBars 但没强制策略侧时序)
//   4. 回测 P&L 输出强制双轨：Gross P&L + Net P&L
```

**卡片**：M11-7 (P1, 3d)

### H. AI 策略生成是合规 + 信誉双重风险

**现状**：`docs/spec/26-ai-strategy-generation.md`「让普通人用自然语言生成可在实盘运行的策略」+ `ai/code_compliance.go` 13 条合规规则（侧重代码安全，非金融正确性）。

**纠正 D2 偏差 2 + 4**：D2 §5.2「参数数 ≤ 数据点/10」是 1990 年代教科书规则，对真实策略意义不大。D2 §5.4「Sharpe > 0.5」**没做 multiple testing correction**——用户测 100 个 prompt，即使真 alpha=0，也有 30-50% 概率找到 Sharpe>0.5 的（p-hacking）。

**修法（多级 gate + 现代统计）**：

```
NL prompt
  ↓
Compliance Gate（代码合法 + 沙箱）
  ↓
LookAhead Gate（AST 扫描 t+δ where δ>0 的引用）
  ↓
Walk-Forward (5-fold OOS) + CPCV
  reject: train_sharpe - test_sharpe > 1.0  (overfitting)
  reject: 所有 fold 中 max_DD > 30%
  reject: trade_count < 30 (统计不显著)
  ↓
Multiple Testing Correction
  Deflated Sharpe Ratio (López de Prado 2014)
  按用户对该策略族的尝试次数 N 衰减:
    DSR = SR × √[(1 - γ·log(N)) / (1 - skew·SR + (kurt-1)/4 · SR²)]
  reject: DSR < 0.95 (95% confidence)
  ↓
Paper Gate (14 天强制仿真)
  reject: paper_return < 0.5 × backtest_net_return  (regime fail)
  ↓
Correlation Gate (与该用户已实盘策略相关性 < 0.7)
  ↓
显式风险声明 + 2FA
  ↓
LIVE
```

**监管/合规边界**（D1+D2 均缺位）：

- 在中国/美国/欧洲，AI 生成投资建议 = 触发投顾牌照监管
- 必须加 **jurisdictional gate**：KYC + IP 检测 + 不向受限地区提供
- 必须加**显式免责声明** + 用户风险问卷
- **trade reporting 准备**：EMIR / MiFID II 类报告（现在不需要，但 schema 应预留）

**卡片**：M11-8 (P1, 5d)

### I. 跨账户净敞口（SaaS 平台层风控，D1+D2 均不充分）

**ant 是 SaaS 多租户**，但 `risksvc` 只看单账户：

- **跨账户净敞口**：User A 在 broker X long EURUSD 1 lot，User B 在 broker Y short 1 lot——**真平台敞口=0，但保证金占用=2**
- **block trade / pro-rata allocation**：一个 AI 策略推到 100 个用户，部分成交时按什么规则分（FIFO / pro-rata / random）？**SEC Rule 17j-1 和 FINRA 4512 类规定要求公平分配**，当前无规则
- **平台总量限额**：所有用户 EURUSD 合计不能超过 broker 给本平台的限额——若不在 ant 侧先拦，触发 broker 拒单会 cascade 拒所有用户

**修法**：

```go
// backend/internal/risksvc/platform_aggregator.go
type PlatformAggregator struct {
    // 5s 刷新的全平台聚合视图
    NetExposureBySymbol  map[string]decimal  // canonical → net lot (long+, short-)
    TotalMarginUsed      decimal
    BrokerLimitUsage     map[string]float64  // broker → used / limit
}

func (p *PlatformAggregator) PreCheckPlatform(req *Order) error {
    // 平台限额优先于单账户限额
    if p.BrokerLimitUsage[req.Broker] > 0.8 {
        return ErrPlatformBrokerLimitNear  // 进入"软冻结"，仅允许平仓
    }
    // ...
}

// 公平分配规则（block trade）
type AllocationRule string
const (
    AllocPro  AllocationRule = "pro_rata"  // 默认
    AllocFIFO AllocationRule = "fifo"
    AllocVWAP AllocationRule = "vwap"      // 各用户按申请时间加权
)
```

**卡片**：M11-4 (P1, 3d)

## 四、架构层缺位（P2：根因性问题）

### J. 时间维度的金融正确性 + 双时间戳设计

**现状**：`ts_unix_ms` 全用 UnixMs，**完全无时区/会话感知**。

**纠正 D2 偏差 1**：D2 §4.3 建议「Bar 聚合改用 broker 时间戳」是**误读了 ADR-0009 的设计意图**——ADR-0009 故意选 `ArrivedUnixMs` 是**反作弊设计**：散户托管场景下 broker 是潜在不可信方，broker 服务器时间戳可被对手方操纵（先拉爆挂单触发止损、再倒回老 ts 让 tick 看起来正常）。

**正确修法**：双时间戳并保留 + NTP 偏离阈值丢弃

```go
type Tick struct {
    BrokerTsUnixMs   int64  // 用于 bar 归属（仅当 ClockSkewMs < 阈值）
    ArrivedTsUnixMs  int64  // 用于审计 / 延迟测量 / 反作弊回退
    ClockSkewMs      int64  // = Arrived - Broker
}

// Quality.Check 增加：
//   if abs(t.ClockSkewMs) > 5000 { drop + metric.broker_clock_anomaly++ }

// BarAggregator 增加策略：
//   prefer t.BrokerTsUnixMs if |ClockSkewMs| < 500ms
//   else t.ArrivedTsUnixMs (anti-manipulation fallback)
```

并引入 `SessionClock` 抽象：

```go
type SessionClock interface {
    SessionOpen(broker, symbol string, t time.Time) bool
    BarBoundary(t time.Time, period string) (open, close time.Time)
    SwapWindow(broker string, t time.Time) bool       // 17:00 EST ±5min
    HolidayMarker(broker string, date time.Time) bool
    BrokerClockOffsetMs() int64                        // 单独可观测
}
```

**FX 24×5 关键场景**：
- NY 17:00 daily rollover → tick 在 17:00:00 vs 17:00:01 的 swap 归属和 bar 归属都不同
- bar period `"5m"` 必须明确 `[17:00:00, 17:04:59.999]` 半开区间
- 周一开盘 gap 应专门标注

**卡片**：M11-10 (P2, 2d)

### K. event-sourced ledger（D2 §6.3 是症状，根因在此）

**现状**：PG 存订单状态 + Redis 存幂等 + CH 存行情。三方不一致时**无法精确对账**——只能用 `mthub.Reconciliation` 拉 broker 当前持仓粗对。

**问题**：当 broker reconcile 显示 "broker 有 ticket=12345 但 PG 没有" 时，是 **submit 后崩溃** 还是 **broker 多发**？两者修复路径完全不同，但 ant 没法区分。

**D2 §6.3 讲"P&L 归因不可分解"是症状**，根因是无 append-only event log，无法重放推算各时点状态。

**修法（按金融语义分层）**：

```
┌─ Tier-0  Trade Ledger（append-only NATS JetStream + CH 投影）
│           ←── 唯一事实源
│           ←── 任何状态都能从这里重算
│           ←── 对账 = 重放 events 重建 state
│
├─ Tier-1  Live State Cache（Redis + in-memory）
│           ←── OMS state, position cache, idempotency
│           ←── 一致性由 Tier-0 重放修复（重启时全量重放最近 N 天）
│
├─ Tier-2  Derived Quantities（5s 重算）
│           ←── PnL（Gross/Net 双轨）
│           ←── Greeks / VaR / margin / exposure
│           ←── 全部从 Tier-0 + 行情快照算出，不存
│
└─ Tier-3  Decision Surface（read-only Tier-2）
            ←── Strategy / RiskEngine / Sizer
            ←── 严禁直接修改 Tier-1
```

**关键不变量**：所有状态必须能从 Tier-0 完整重建——这是金融对账的基础。

NATS JetStream 已部署，**这不是大改造**，而是把 NATS 从"广播总线"升格为"唯一事实源"。

**卡片**：M11-11 (P2, 5d)

### L. SRE 级风控控制面（D1 §3.10）

完全缺位：

| 控制 | 状态 | 严重度 |
|---|---|---|
| **Kill Switch**（一键停所有用户所有订单） | ❌ | 监管事故响应必备 |
| **Strategy circuit breaker**（策略 30min 亏 X% 自动熔断） | ❌ | broker 抖动有 spill 熔断 ADR-0005，**策略亏损无** |
| **Canary / shadow trading**（新版策略小账户跑 1 周再全量） | ❌ | 灰度发布 |

```
controlplane/

  ├── strategy_breaker.go  # 策略级亏损熔断
  ├── canary.go            # 新版本灰度发布器
  └── rules_change_log.go  # 风控规则变更审计 + 回滚
```

## 五、架构本质性建议（贯穿全部卡片）

### 建议 1：诚实声明系统的"延迟段位"

**纠正 D2 §7.1 偏差**：D2 把 PreCheck 标 `<1ms`、NetExposure 标 `<5ms`——**和 ant 实际延迟段位不匹配**。

mtapi.io 是**两层代理**（见 `mt-gateway` skill），typical 50-300ms。在 200ms 总链路里优化 PreCheck 从 1ms 到 0.5ms = **0.25% 边际收益**。

**应在 `docs/spec/27-signal-execution-slo.md` 顶部加**：

> **本系统设计的最小有效持仓周期 = 5 分钟。 < 5min 的策略不应在本系统部署。**
>
> 端到端延迟段位：50-300ms（mtapi 两层代理决定下限）。
> 风控同步段预算：≤ 10ms（合并 PreCheck + HardLimit + Sizer，无需进一步拆分）。
> 风控异步段（VaR / 压力测试）：30s 周期。

这一行避免下游做错误的工程投入（Buffer engine 0 延迟优化、tick dedup ringbuffer 等可降级为 1s 批处理，省 30% CPU）。

### 建议 2：execution algo 层（D1+D2 均漏）

当前所有订单都是裸的市价单/限价单。SaaS 聚合后真问题：

- **TWAP / VWAP / POV / Implementation Shortfall** algo 缺位
- **adverse selection**（限价单被反向选择吃单的风险）
- **market impact**（Almgren-Chriss / Kissell-Glantz）
- **information leakage / front-running 防御**

**散户单笔不重要，但 SaaS 把 1000 个用户的同一信号聚合后就是机构级订单**。

**修法**：在 OMS 和 mthub 之间加 `execalgo/` 层：

```
Strategy → Signal → Sizer → ExecAlgo (TWAP/VWAP/POV)
                              ↓
                         拆成多笔小单
                              ↓
                            mthub.Place
```

**卡片**：M11-13 (P3, 5d)，建议 M11 暂缓到 M12

| 缺口 | 现在做的成本 | 关闭后再补的成本 | 倍数 |
|---|---|---|---|
| Per-user limiter 主管线 | 1 个新包 `internal/usermgr` + 各服务接入 | 上线后遇跑飞用户才装 = 事故重复 × 发生次数 | 不可估量 |
| Clock 抽象 | 一次性迁移 + lint rule | 后期改要重写所有测试 + 风险面巨大 | 8-10× |
| Strategy Lifecycle | 在 E1 之前先做 E0 | E1-E6 全部用裸接口写完后重构 | 5× |
| Backpressure | 每段流水线建立时一起明确 | 上线后流量峰值砸下来才补 | 不可估量（事故级） |

**结论**：地基阶段不做，后期补 = 拆掉重建。这就是 M11 路线图整体前置到 M10-BASE 的同一逻辑。
